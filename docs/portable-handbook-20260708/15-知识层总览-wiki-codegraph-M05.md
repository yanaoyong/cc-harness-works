# 知识层总览：wiki、codegraph、M0.5

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

Harness 不能只靠当前对话上下文工作。Agent 需要知道历史规则、架构决策、失败案例、代码影响面和外部调研结论。知识层就是为这些问题提供证据的能力集合。

本篇解释 wiki、codegraph、M0.5 和 L 旁路之间的关系。即使你没有原始 wiki，也能理解：哪些问题适合文档知识库，哪些问题适合代码图谱，哪些问题需要调研，以及为什么这些证据不能直接替代 Owner 和用户的裁决。

## 核心模型

知识层回答的不是“是否通过”，而是“判断时需要哪些证据”。它提供背景、线索和影响面，但不夺取流程门禁。

| 能力 | 适合回答 | 输出是什么 | 不适合回答 |
|---|---|---|---|
| wiki | 规则、历史决策、失败案例、跨篇综合 | 摘要、来源、两跳溯源 | 代码调用关系、全部字面出现 |
| codegraph | 符号、调用方、被调方、影响面、索引新鲜度 | 代码结构线索 | 文档规则、历史决策 |
| M0.5 | 重大未知、外部方案、模型/框架选择 | 调研 dossier、开放问题、三源证据 | 直接替代用户拍板 |
| `rg` | 字符串、错误消息、配置键、全部出处 | 原文命中行 | 跨篇概念综合 |
| Owner/HITL | 综合证据并推进流程 | 阶段结论、用户授权 | 被旁路工具替代 |

这张表是知识层的核心：工具给证据，Owner 做组织，用户在关键点裁决。

可以用同一个场景理解四类能力的区别。假设一张卡要修改“测试通过后是否可以合并”的判断规则。wiki 适合回答：为什么项目曾经强调测试评审，历史上有哪些 false-green 失败，规则和 ADR 如何解释这个边界。codegraph 适合回答：测试守护逻辑在代码里由哪些模块调用，改动会影响哪些命令或 hook。M0.5 适合回答：如果要引入新的测试质量度量，外部有哪些方案和取舍。`rg` 适合回答：`false green`、`total=0`、某个错误消息或配置键在仓内哪里出现过。

这四类证据最终都不能直接说“可以合并”。wiki 能提醒历史风险，但不能证明当前 diff 正确；codegraph 能提示影响面，但不能证明行为正确；M0.5 能降低方案不确定性，但不能替用户选择取舍；`rg` 能找全字面出现，但不能理解语义。Owner 要把这些证据放回阶段、评审、测试、CI 和 HITL 中，形成可复查判断。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 想知道某条规则为什么存在 | 需要历史和跨篇综合 | 用 wiki 或 portable 正文理解规则背景 | 只看当前文件可能误删失败驱动的护栏。 |
| 想知道某函数影响谁 | 需要代码结构 | 用 codegraph 或源码搜索定位 callers/callees，再读源码核对 | 只靠文字搜索可能漏动态调用和影响面。 |
| 想找某个错误消息出现在哪里 | 需要字面匹配 | 直接用 `rg` 或等价全文搜索 | 用 wiki 会因提炼改写而漏字面出现。 |
| 要比较外部方案 | 当前项目知识不够 | 进入 M0.5，收集多源证据和开放问题 | 只凭模型记忆会把过时或单源信息当事实。 |
| 判断是否能合并 | 需要门禁证据 | 回到质量门禁和 HITL | 工具命中不能替代测试、CI 或用户授权。 |

“证据不夺权”的判断标准很简单：证据只能回答它擅长的问题，不能越过问题边界。wiki 命中某条规则，只说明有相关规则背景；codegraph 找不到调用方，只说明在当前索引和查询条件下未发现；M0.5 推荐某方案，只说明外部证据支持该方案；`rg` 没命中，只说明字面未出现。任何证据一旦被写成“因此阶段通过”“因此可以合并”“因此用户已接受”，就越权了。

在实际工作中，Owner 应把证据写成“判断输入”。例如：“wiki 显示该规则来自 false-green 失败记录；codegraph 显示改动影响 test guard 和 CI hook；源码核对确认命令路径一致；测试评审仍要求补 total=0 场景。”这段话没有让工具替代裁决，而是把工具输出组织成下一步行动。

## 最小使用步骤

1. 先判断问题类型：文档背景、代码结构、字面搜索、外部调研还是门禁裁决。
2. 选择最小合适工具：wiki/codegraph/M0.5/`rg`/人工阅读。
3. 记录证据来源和置信边界，例如索引是否新鲜、调研是否多源、源码是否核对。
4. 将证据交给 Owner 整合，而不是让工具直接决定流程通过。
5. 如果证据影响范围、架构、测试或合并，回到对应阶段和 HITL。

## 例子

Reviewer 看到一次重构修改了 `ChangeStore`，不确定影响哪些命令。正确做法不是只读当前 diff，也不是让 wiki 回答调用关系。应先用代码结构工具或源码搜索找到调用方和被调用方，再回到源码核对关键路径，然后把影响面写进 review 或测试建议。

另一个例子是用户问“为什么不能跳过测试评审”。这不是代码结构问题，而是规则和失败案例问题。此时知识层应提供历史背景：测试可能 false green，测试本身也需要被审视。最终是否放行仍回到质量门禁和用户授权。

再看短链服务的过期链接问题。用户问“为什么过期链接要返回明确错误？”wiki 或 portable 正文能解释契约和质量规则；codegraph 能找出跳转解析和错误响应的调用路径；`rg` 能找出已有 `expired` 错误码文本；如果团队考虑 404、410 或自定义错误页的外部规范，M0.5 可以做调研。最终选择哪种错误语义，要回到 M3 接口契约和 HITL，而不是由某个工具单独决定。

反例是：“codegraph 没发现外部调用方，所以可以直接改 public API。”这个结论跳过了源码核对、契约判断、测试和用户授权。正确写法是：“当前 codegraph 未发现调用方，但 public API 改动仍需检查文档、测试和调用方契约；建议回到阶段 4/5 补证据。”

## 维护者注意事项

维护知识层文档时，要避免把工具写成神谕。任何工具说明都应包含适用问题、不适用问题、置信边界和回退路径。没有边界的工具说明，会鼓励 Agent 把命中结果当裁决。

如果新增知识工具，应先把它放入这张对比表：它回答什么问题，输出是什么，不适合回答什么，结果如何被 Owner 使用。只有通过这四个问题，工具才不会和 wiki、codegraph、M0.5 或 `rg` 混用。

行为验收要覆盖“是否在正确场景使用工具”，而不是只验证命令能跑。一个 Agent 在文档规则问题上用 codegraph，或者在代码影响面问题上只查 wiki，即使工具调用成功，也是不合格使用。

## 边界与反模式

- 不要用 wiki 回答代码调用关系；否则会把文档摘要误当源码事实。
- 不要用 codegraph 回答流程规范；代码图谱不理解历史决策。
- 不要把 M0.5 调研结论当成用户决策；调研只降低不确定性。
- 不要把工具命中当成完成证据；完成仍需要评审、测试、CI、部署和 HITL。
- 不要因工具未命中就断言不存在；应说明回退路径和置信边界。

## 读完后你应该能做什么

- 能把一个问题路由到 wiki、codegraph、M0.5、`rg` 或 Owner/HITL。
- 能解释“证据不夺权”的含义。
- 能说明为什么知识层需要行为验收：工具存在不等于 Agent 会在正确场景使用。
- 能识别知识层证据的置信边界。
