# ClaudeCode 桥接、hooks 与运行时集成

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

Harness 的核心设计尽量厂商无关，但实际运行需要接到某个 Agent 运行时。Claude Code 桥接层负责把常驻契约、hooks、commands、prompt_state 和恢复机制落到具体运行环境。

本篇解释桥接层如何帮助 Agent 在正确时刻看到正确上下文，同时避免把厂商机制误当通用核心。

## 核心模型

| 组件 | 作用 | 风险 |
|---|---|---|
| `CLAUDE.md` | 会话入口和常驻契约 | 过胖挤占上下文，过薄失去约束 |
| Resident Contract | 可重建的精简常驻约束 | 与模板漂移 |
| SessionStart hooks | 初始化、脚手架、状态提示 | hook 缺失时体验降级 |
| prompt_state | 提高当前流程状态可见性 | 不应替代事实源 |
| commands | 暴露人工入口 | 路径和脚本漂移 |
| setup skill | hook 失效兜底 | 需要 Owner 记得触发 |

## 通用运行时映射

portable 读者不需要使用 Claude Code，但需要保留同一组运行时语义。迁移到其他 Agent、IDE、CLI 或编排器时，可以按下表翻译：

| Claude Code 机制 | 通用运行时概念 | 必须保留的语义 | 可替换实现 |
|---|---|---|---|
| `CLAUDE.md` | 会话入口契约 | Agent 启动后先看到身份、流程、约束和第一动作 | system prompt、项目说明、启动模板 |
| Resident Contract | 常驻最小规则 | 精简、稳定、可重建，不依赖隐式会话记忆 | pinned prompt、policy bundle、project instruction |
| SessionStart hook | 启动生命周期钩子 | 初始化状态，提示当前阶段和可用入口 | startup script、pre-run hook、IDE session initializer |
| UserPrompt/Tool hooks | 交互和工具护栏 | 在关键动作前提醒边界、权限和证据要求 | middleware、command wrapper、preflight check |
| commands | 人工操作入口 | 用户能查询状态、恢复脚手架、触发诊断 | CLI command、IDE command、dashboard action |
| prompt_state | 状态提示层 | 提醒当前流程位置，但不充当事实源 | status file、dashboard、context note |
| setup skill | 手动恢复入口 | hook 失效或安装不完整时能重建运行面 | bootstrap script、repair command、installer check |

这张表的核心是：文件名可以替换，语义不能丢。一个非 Claude Code 运行时只要能提供会话入口契约、生命周期初始化、人工恢复入口、项目本地状态和行为验收，就能承载 Harness；反过来，如果只复制了 `.claude/` 文件名，却没有这些语义，也不算完成桥接。

hook 失效恢复要先分层。第一层是文件是否存在，例如运行时入口、`CLAUDE.md`、commands、状态目录和脚手架；第二层是权限和路径是否可执行；第三层是 hook 是否在正确生命周期触发；第四层是 hook 是否把结果写回了正确项目目录；第五层是 Agent 是否真的采纳了提示。前四层是运行时问题，第五层是行为验收问题，不能混在一起排查。

恢复策略也要区分 fail-open 和 fail-closed。维护提醒、状态展示、文档索引刷新这类 hook 应尽量 fail-open：失败时记录诊断并允许协作继续。涉及危险写入、跨项目状态、权限提升或分发源修改的 hook 应更接近 fail-closed：失败时阻止继续或要求 HITL。这个区分能避免小故障阻断全部工作，也能避免高风险动作静默越界。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 插件装完没提示 | 初始化可能缺失 | 检查 SessionStart 和 setup 兜底 | Agent 无常驻约束。 |
| `CLAUDE.md` 丢失 | 常驻契约不可见 | 使用 restore 机制重建 | 会话入口失去流程纪律。 |
| 状态串项目 | state dir 边界错误 | 使用项目本地状态目录 | 一个项目的状态污染另一个。 |
| hook 报错 | 判断是否应 fail-open | 维护类 hook 不应阻断全部工作 | 小故障导致无法协作。 |
| 做 portable 文档 | 不应依赖 Claude Code | 把运行语义写成普通说明 | 外部读者没有该运行时。 |

当用户报告“插件装了但 Agent 像普通聊天一样工作”，排查顺序应是：确认安装态资产是否落盘；确认会话入口是否加载常驻契约；确认 SessionStart 是否执行并显示状态；确认项目本地状态目录是否存在；确认 commands 是否能查询状态；最后用一个真实任务观察 Agent 是否按 Owner、阶段和证据纪律行动。只检查文件存在会漏掉“文件有但未被采纳”的问题。

当 hook 报错时，先把错误归类，再决定恢复动作：

1. 路径错误：通常通过重新生成脚手架或修正项目本地路径恢复。
2. 权限错误：显式记录阻塞，交由用户确认权限或执行环境。
3. 依赖缺失：降级到手动入口，并记录缺少的工具或版本。
4. 状态损坏：隔离当前项目状态，不能复用其他项目状态。
5. 生命周期不匹配：修改桥接配置，确认 hook 是否在正确时机触发。
6. 逻辑异常：回到测试和 ADR，判断是否影响分发边界或恢复语义。

## 最小流程

1. 安装或首次运行时落盘必要脚手架。
2. 通过常驻契约给 Agent 注入身份、流程和第一动作。
3. SessionStart 提示当前状态和可用入口。
4. UserPrompt/Tool hooks 提供护栏或提醒。
5. commands 允许人类查询状态。
6. hook 失效时使用 setup skill 或手动兜底。

## 例子

一个新项目安装 Harness 后，如果首次会话没有自动生成基础目录和常驻契约，Agent 会像普通聊天模型一样工作。first-run bootstrap 和 CLAUDE restore 的意义，就是让安装态也能自恢复，而不是依赖维护者手工记住所有步骤。

具体恢复例子：某项目打开后没有阶段提示，Owner 先检查 `CLAUDE.md` 是否存在并包含 import；再检查运行时 hook 配置是否指向项目本地 `.harness`；然后执行 setup 兜底生成基础目录；接着用 command 查询当前状态；最后给 Agent 一个小任务，观察它是否主动确认范围、读取状态并给出证据链。如果最后一步失败，问题已经不是 hook 文件，而是常驻契约或行为验收需要修复。

反例是把 `.claude/` 整个目录复制到另一个项目。短期看似恢复了 hook，实际可能把旧项目路径、状态和命令一起带过去，造成跨项目污染。正确做法是从可分发资产或 setup skill 重新生成安装态文件，并确认状态目录项目本地化。

## 维护者注意事项

维护桥接层时，要保持“核心语义”和“运行时接线”分离。Owner、阶段、HITL、证据链属于核心语义；SessionStart、commands、prompt_state、`.claude/` 属于具体运行时接线。portable 文档只能依赖核心语义，运行时文档才描述 Claude Code 的具体恢复动作。

每次修改 hook，都要同时检查三件事：失败时是否有可读诊断；用户是否有手动恢复入口；恢复后是否能通过真实行为验收。没有行为验收的 hook 修复很容易停在“命令不报错”，但 Agent 仍然没有采纳约束。

## 边界与反模式

- 不要把 `.claude/` 当成厂商无关核心；它是具体桥接层。
- 不要让 hook 失败阻断所有工作；维护提醒应尽量 fail-open。
- 不要把 prompt_state 当事实源；事实源仍应是落盘状态。
- 不要让 `CLAUDE.md` 无限膨胀；常驻契约应精简。
- 不要在 portable 文档里假设 Claude Code 必然存在。

## 读完后你应该能做什么

- 能解释 core 设计和 Claude Code 桥接层的区别。
- 能判断 first-run、state dir、CLAUDE restore 各解决什么问题。
- 能识别 hook、commands、prompt_state 的边界。
- 能把运行时专有机制转写成 portable 说明。
