# M3：架构、接口、ADR 与定制规则

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

M3 把 M2 的范围转成可落地的系统设计。它回答：系统怎么分层、接口如何定义、数据和横切关注点如何处理、哪些决策必须写成 ADR、哪些项目约束需要变成定制 Rules。

没有 M3，M4 工程基线会变成“先搭一个壳”，编码阶段再临时补架构债。

## 核心模型

M3 分三段：

| 子阶段 | 解决的问题 | 典型产物 | 下游如何消费 |
|---|---|---|---|
| M3.1 架构 | 系统结构、组件边界、领域模型、部署拓扑 | C4、domain、ER | M4 建目录、模块和服务边界 |
| M3.2 接口与质量 | API/CLI、错误、并发、认证、可观测性、数据生命周期 | interface、QAS、cross-cutting | 测试、CI、运行信号 |
| M3.3 ADR 与 Rules | 关键选择、替代方案、项目约束 | ADR、定制 Rules | M4 物理化，长期维护查证 |

M3 的交付不是文档完成，而是 M4 能据此落地工程基线。

承接短链服务例子，M3 要把 M2 的范围转成工程可执行的设计。因为本轮只做创建、跳转、过期和基础审计，架构不应引入管理后台、复杂统计管道或多租户权限。M3 的任务是把“本轮范围”变成组件边界、接口契约、数据模型、错误语义和需要长期保留的决策。

一个短链服务的 M3 可以这样落地：组件上分为 API 层、短码生成服务、链接存储、跳转解析和审计日志；接口上定义 `POST /links` 创建短链、`GET /{code}` 跳转、错误码覆盖不存在、过期、非法 URL；数据上定义 links 表，字段包括 code、target_url、expires_at、created_at；ADR 记录短码生成策略和存储选择；定制 Rules 规定接口错误格式、数据迁移和测试覆盖要求。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 要新增模块 | 需要组件边界和领域模型 | 做 M3.1，再进入接口设计 | 直接编码会让模块职责混乱。 |
| 要引入新存储 | 属于数据架构决策 | 写数据生命周期和 ADR | 未来迁移、备份、恢复无依据。 |
| 接口会被前端/外部消费 | 需要契约 | 定义请求、响应、错误、兼容策略 | 测试和联调会反复 mismatch。 |
| 决策有多个方案 | 需要 ADR | 记录选项、决策、后果 | 维护者会反向修改已权衡的选择。 |
| 项目有特殊约束 | 需要定制 Rules | 从 M3.3 进入 M4 物理化 | 写进通用模板会污染其他项目。 |

判断 M3 是否足够进入 M4，可以看工程师是否还需要猜关键设计。如果还不知道短码冲突如何处理、过期链接返回什么、审计日志是否本轮必须落地、存储迁移如何初始化，M3 就还没有完成。如果只剩下目录命名或框架配置选择，可以交给 M4。

ADR 的判断也要具体。不是所有设计都需要 ADR；但如果一个选择未来维护者可能想推翻，或者有多个合理方案，就应该记录。例如短链服务可以为“短码生成使用随机码还是递增 ID 哈希”写 ADR，因为它影响碰撞、可预测性和迁移；不需要为“控制器文件叫 links.ts”写 ADR，因为那只是实现细节。

## 最小流程

1. 从 M2 范围中提取系统能力和边界。
2. 画出上下文、容器、组件和领域模型。
3. 定义 API/CLI/event 契约、错误模型和兼容要求。
4. 写清 QAS：性能、可靠性、安全、可观测性、并发和恢复。
5. 对重要决策写 ADR。
6. 对项目特有长期约束写定制 Rules。
7. 交给 M4 时，逐项说明每个设计产物如何落地。

## 例子

短链服务的 M3 不只是“做一个 API”。它需要定义：创建短链和跳转解析的组件边界；短码生成和冲突处理；数据库表和索引；错误码；过期策略；访问统计是否在本轮范围内；如果选择 SQLite/Redis/PostgreSQL，要写清代价和后果。否则 M4 建工程骨架时只能凭实现者临时判断。

完整一点的 M3 例子可以拆成三组产物：

1. 架构视图：系统对外只有创建 API 和跳转入口；内部包含 short-code service、link repository、audit logger；访问统计只记录原始事件，不做聚合报表。
2. 接口契约：创建请求包含 `target_url` 和可选 `expires_at`；响应包含 `code` 和 `short_url`；跳转成功返回 302；过期返回明确错误；非法 URL 返回 400。
3. ADR 与定制 Rules：记录为什么本轮先用单库表结构而不是引入缓存；记录短码冲突重试策略；项目规则要求新增接口必须有契约测试和错误码测试。

反例是 M3 只写“使用 REST API + 数据库”。这不能指导 M4 建表、写测试、配置 CI，也不能指导 M5 拆卡。M3 的价值在于把设计约束落到后续能执行的粒度。

## 维护者注意事项

维护 M3 文档时，要警惕“架构图很漂亮但工程不可落地”。每个架构元素都应能映射到 M4 的目录、模块、配置、测试或规则。无法映射的内容要么是背景说明，要么需要进一步拆解。

接口契约要写到测试能消费的程度。只说“提供创建接口”不够；至少要说明请求、响应、错误、兼容和调用方期望。否则 M5 拆出的接口卡会在 10 阶段重新发明契约。

定制 Rules 只应来自项目长期约束。短链服务的错误格式、数据迁移、审计字段可以是项目规则；“这次先不做后台”是范围结论，不应写成长期规则。

## 边界与反模式

- 不要把 ADR 写成流水账；ADR 应记录选择、替代项和后果。
- 不要让接口契约只存在于口头描述；否则测试和联调无法对齐。
- 不要把项目定制 Rules 写进 core 通用层。
- 不要在 M3 未收敛时进入 M4；否则工程基线会承载未决设计。
- 不要为普通实现细节写 ADR；ADR 只记录未来维护者需要知道的决策。

## 读完后你应该能做什么

- 能列出 M3 到 M4 的交付清单。
- 能判断一个决策是否需要 ADR。
- 能区分接口契约、QAS、数据架构和定制 Rules。
- 能识别“设计讨论稿”还不能进入工程基线的情况。
