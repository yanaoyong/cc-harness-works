# 阅读路线与术语地图

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

> 口径：手册基线 2026-07-08 · 增补至 2026-07-15。目录名保留基线日期，正文已把治理硬门、上下文分段、升级传导、成本工程等新增机制内嵌进对应主题篇，供离线读者理解当前系统状态。

## 本篇解决什么问题

这套 portable 手册面向没有仓内 `wiki/`、没有历史变更卡、也不了解 Harness 术语的新读者。它的作用不是列出 25 个文件名，而是给出一张可离线使用的阅读地图：先建立什么概念，再理解哪条流程，最后如何把规则、知识层、质量证据和维护决策连起来。

读者不需要拥有原始 wiki 文件。正文会直接解释核心模型；末尾的“离线来源依据”只告诉维护者这些结论来自哪些原始规范或记录，便于将来追溯和更新。

本篇是 25 篇的入口。读完后，你应该能判断自己属于哪类读者、该从哪篇开始、遇到一个陌生术语时应该如何理解它，以及 portable 版和 linked 版的根本差别。

这里的 linked 版指原仓内带 wiki 跳转和来源路径的版本；portable 版则把必要背景、判断规则和例子直接写进正文，适合脱离原仓使用。

## 核心模型

Harness 可以理解成一套“让 AI 编码代理受控工作”的工程治理系统。它把一次用户请求拆成可审计的流程、角色、规则、证据和知识沉淀，避免 Agent 只凭上下文窗口和自述推进工作。

这套手册按五条主线组织：

| 主线 | 覆盖文件 | 解决的问题 |
|---|---|---|
| 入口心智 | `00-04` | 说明 Harness 是什么、一张变更卡如何运行、10 阶段如何形成证据链。 |
| 编排机制 | `05-09` | 说明 Owner、子 Agent、Rules、Skills、变更卡如何配合。 |
| 项目级元流程 | `10-14` | 说明从模糊愿景到 Roadmap 的 M0-M5 流程，以及事后修订如何级联评估。 |
| 知识与旁路 | `15-19` | 说明 wiki、codegraph、M0.5、L 旁路如何提供证据但不夺取裁决权。 |
| 质量与迁移 | `20-24` | 说明完成证据、测试可信度、运行时集成、插件化分发和长期维护。 |

portable 版的关键要求是“正文自足”。如果 linked 版中的一句话原本依赖 wiki 链接才能理解，portable 版必须把必要背景写进正文，而不是只留下来源名。

这张地图还承担一个更实际的功能：防止读者用错入口。Harness 的主题之间有明显先后关系。如果先读 Rules 和 Skills，读者会知道很多约束，却不知道这些约束服务于哪条工作流；如果先读 wiki 和 codegraph，读者会看到很多工具，却不知道工具证据为什么不能直接决定合并。入口篇要先把系统目标、工作流和术语边界放稳，再让读者进入细节。

判断一篇文章该放在哪条主线时，可以问三个问题：它解决的是“为什么要这样工作”，还是“具体怎么编排”，还是“如何证明完成”。入口心智解决理解问题；编排机制解决执行问题；项目级元流程解决从 0 到 1 的收敛问题；知识与旁路解决证据来源问题；质量与迁移解决可信交付和长期维护问题。这个分类不是目录装饰，而是读者选择路线时的路标。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 第一次接触 Harness | 先建立系统目标，再看最小工作流 | 读 `01 -> 03 -> 04 -> 20` | 直接读机制篇会看到很多术语，但不知道它们为什么存在。 |
| 要用 Harness 做日常开发 | 先理解变更卡和证据链 | 读 `03 -> 04 -> 05 -> 09 -> 20` | 只学命令和目录，容易跳过 HITL、评审或测试证据。 |
| 要维护规则或技能 | 先区分规则、技能、运行时和分发层 | 读 `02 -> 07 -> 08 -> 22 -> 24` | 把一次卡内做法升级成通用规则，会污染模板和后续项目。 |
| 要做项目从 0 到 1 | 先走 M0-M5，而不是直接建 10 阶段卡 | 读 `10 -> 11 -> 12 -> 13 -> 14` | 没有愿景、范围和架构边界时，Roadmap 卡会不断返工。 |
| 要把文档带到外部项目 | 使用 portable 版，不依赖仓内 wiki 路径 | 保留正文和离线来源依据，删除本仓专属假设 | 外部读者如果必须打开本仓 wiki 才能读懂，portable 就失败了。 |

判断路线时不要只看“我想读哪篇”，而要看“我现在要做什么”。如果你要马上处理一个用户请求，最短路线是 `03/04/20`，因为你需要知道如何建卡、如何跑阶段、如何证明完成。如果你要改 Harness 本身，最短路线是 `02/07/08/22/24`，因为你需要先分清资产层级、规则层、技能层、运行时层和历史决策。如果你要把文档拿到外部团队使用，最短路线是 `00/01/02/23/24`，因为迁移最大的风险不是漏掉一篇文章，而是把本仓历史、运行时副本或路径假设当成通用资产。

遇到陌生术语时也要先判断它属于哪一类。`Owner`、`Reviewer`、`Generator` 是角色；`10 阶段`、`M0-M5` 是流程；`Rules`、`Skills` 是行为约束和阶段方法；`wiki`、`codegraph`、`M0.5` 是证据层；`SSOT`、`变更卡`、`Roadmap 卡` 是状态和产物。把这些类别分清，后续读到任何单篇时都不容易迷路。

## 最小使用步骤

1. 先读本篇，确认你要解决的是学习、使用、维护、元流程还是分发问题。
2. 按读者画像选择最短路线，不要从 25 篇逐字顺读。
3. 遇到术语时先看本篇“术语速查”，再进入对应专题文章。
4. 在专题文章里优先读正文、表格和例子；末尾来源只作维护线索。
5. 如果要把手册迁移到另一个项目，保留 portable 目录内互链，删除或改写所有本仓独有路径假设。

## 术语速查

| 术语 | 自足解释 | 边界 |
|---|---|---|
| Harness | 让 AI 编码代理按工程流程工作的一套编排、规则、技能、知识和运行时机制。 | 不是单个 prompt，也不是普通脚手架。 |
| Owner | 编排中枢，负责判断任务类型、维护状态、委派子 Agent、整合证据并组织用户拍板。 | 不等同 Generator；Owner 不应把裁决外包给执行者。 |
| 10 阶段 | 单张增量变更卡从需求分析到用户确认的主流程。 | 不负责从 0 到 1 的项目规划。 |
| M0-M5 | 项目级元流程，从愿景、需求、范围、架构、工程基线到 Roadmap。 | 产物最终喂给 10 阶段，而不是替代 10 阶段。 |
| 变更卡 | 一次具体工作的审计容器，记录阶段、产物、证据和状态。 | 不是临时待办，也不是纯聊天记录。 |
| Roadmap 卡 | M5 拆出的可执行任务单元，未来可以进入 10 阶段。 | 不是没有依赖和验收边界的愿望清单。 |
| HITL | Human-in-the-loop，用户在关键点确认范围、风险或合并。 | 不是形式确认；它会改变流程状态。 |
| Rules | 长期行为约束，规定 Agent 在工程、流程、测试、架构等方面不能越界。 | 不是一次性任务说明。 |
| Skills | 阶段性 SOP，告诉 Agent 在具体阶段如何完成分析、实现、评审、测试或规划。 | 不应全部常驻加载。 |
| wiki | 项目知识库能力，用来沉淀规则、历史决策、失败案例和跨篇综合；在 portable 手册中泛指这种能力，不要求使用原仓 wiki 实现。 | 适合综合，不适合找全部字面出现。 |
| codegraph | 代码知识图谱，用来查询符号、调用关系、影响面和索引新鲜度。 | 不替代源码核对。 |
| M0.5 | 重大未知前的调研层，用多源证据减少盲猜。 | 给证据，不替代 HITL。 |
| L 旁路 | 轻量请求和旁路证据的处理通道。 | 不是绕开流程做项目产物。 |
| SSOT | Single source of truth，某类事实的当前权威位置。 | 手册通常是解释层，不是所有事实的权威源。 |
| DF-015（治理硬门） | 授权真实性的机械兜底门：授权台账、合并与推送门、委派守护、终态翻牌门四道；授权语要逐字、同一道门只消费一次、引用编号说不清就拒绝放行。 | 是机器兜底，不替代人在关键点的真实拍板。 |
| DF-017（上下文分段） | 上下文水位到阈值时在阶段边界（第 2 到 3、第 6 到 7 两处）触发清理并写一条交接记录，实测单卡节省约 20–27% token。 | 只在阶段天然断点切分，不打断进行中的推理。 |
| 交接哨兵 | 分段清理时落下的一条跨会话交接记录，只被下一段消费一次、带时效，用来把工作接力到新会话。 | 不是普通日志；消费一次即失效。 |
| mirror_sync（升级传导） | 把权威源的升级自动传导到消费方镜像的机制，解决"升级了却没同步到安装态"的断链。 | 是自动传导纪律，不是手工逐文件拷贝。 |
| lockstep（版本锁步） | 一次版本号提升必须同步改 4 个文件（核心包、分发清单、两个 profile 声明含 pin），漏改会让守护测试出现 5 处红。 | 只改内容不提版本号，存量用户装不上升级。 |
| ADR-018（常驻契约注入） | 取消 @import 常驻、改由会话启动钩子注入完整调度契约，使子 Agent 底座更轻。 | 关键决策契约仍常驻入口文件，注入丢失也不失效。 |
| 上下文分段 | 见 DF-017：按上下文水位在阶段边界自动切分会话、写交接记录以控成本。 | 与"压缩摘要"不同，分段是主动清理 + 交接接力。 |

## 例子

一个新维护者收到任务：“把 Harness 文档整理成对外可读资料。”

错误做法是直接复制 linked 版，然后删除所有 wiki 链接。这样读者会看到很多 `stage-02-overview`、`test-guard-spec` 之类的名称，却不知道它们是什么意思。

正确做法是先读本篇和 `01`，确认 portable 版必须自足；再读 `03/04/20`，理解流程和证据链；最后重写每篇正文，把原本由 wiki 链接承载的定义、背景、判断规则和例子内嵌进去。末尾只保留“来源列入原因和离线依据说明”，供维护者将来追溯。

再看一个日常使用例子。用户说：“帮我修一下 CI 里偶发失败的测试。”如果只看术语表，你可能会立刻跳到测试守护；如果按阅读路线判断，这其实先要进入 `03` 的变更卡路由，再用 `04` 的阶段模型定位当前缺什么证据，然后读 `21` 判断是不是假绿、环境差异或契约问题。也就是说，路线不是线性读书顺序，而是帮助你把当前问题接到正确机制上。

反例是把所有问题都导向“知识层”。比如看到一个不熟悉的规则名，就让读者去查 wiki；看到一个代码符号，就让读者去跑 codegraph。这样会让工具承担裁决责任。正确做法是先在正文里解释规则和符号所在的流程位置，再把 wiki 或 codegraph 作为证据补充，而不是阅读前提。

## 维护者注意事项

维护这套目录时，优先保持路线稳定，谨慎调整文件编号。编号不仅是排序，也是跨篇互链和读者记忆的入口。新增文章时，应先判断它属于五条主线中的哪一条；如果一个主题横跨多条主线，优先放到它最常被使用的工作场景中，再在相关篇章做互链。

更新术语时，要同时检查三个位置：本篇术语速查、对应专题文章、以及来源依据中的离线吸收位置。术语解释可以简洁，但不能只留内部简称。外部读者第一次看到 `HITL-1`、`M0.5`、`L 旁路` 时，应该能从本手册中直接理解它们的用途和边界。

如果未来要把 portable 手册发布到另一个文档站，保留本目录内的相对链接即可；不要恢复仓内 wiki 路径。发布前应重新跑两类检查：一类是链接检查，确保目录内互链不悬空；另一类是依赖表达检查，确保正文没有把关键判断交给外部来源。

## 边界与反模式

- 不要把 portable 版写成链接目录；否则离线读者会失去上下文。
- 不要让来源说明承担正文解释责任；否则正文没有独立阅读价值。
- 不要在外部文档中假设读者拥有本仓路径；否则迁移后会出现大量悬空指针。
- 不要把手册当成所有规则的权威源；真正修改规则时仍要回到目标项目的规则文件和决策记录。
- 不要删除来源依据；否则未来维护者无法判断某个结论从哪里来。

## 读完后你应该能做什么

- 能为不同读者选择 25 篇中的最短路线。
- 能解释 Harness、Owner、10 阶段、M0-M5、Rules、Skills、wiki、codegraph、L 旁路之间的关系。
- 能判断一篇 portable 文章是否仍依赖外部 wiki。
- 能说明为什么来源依据必须写清列入原因、具体补足和离线使用方式，而不能只保留文件名。
