# M0-M2：愿景、需求、范围

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

M0-M2 是项目级元流程的前半段，负责把模糊想法收敛成可设计的范围。它们防止项目一开始就带着含混目标进入架构和编码。

读完本篇后，你应该能判断一个想法是否已经足够进入 M3 架构设计，或者还需要继续澄清愿景、需求和范围。

## 核心模型

| 阶段 | 目标 | 产物 | 通过标准 |
|---|---|---|---|
| M0 愿景 | 回答为什么做、服务谁、成功是什么 | vision | 问题、目标、不做声明清楚 |
| M1 需求 | 展开用户故事、功能、NFR、约束和开放问题 | requirements | 需求池可排序、可讨论 |
| M2 范围 | 明确本轮做什么、不做什么、默认决策和风险 | scope | 进出范围能指导架构 |

M0 偏方向，M1 偏发散，M2 偏裁剪。三者不能互相替代。

继续短链服务的贯穿例子。用户最初说“给内部团队做一个短链服务”，M0 不能马上列功能，而要先确认方向：内部团队是谁，是研发、运营还是客服；他们为什么需要短链，是为了分享长 URL、统一追踪、还是做营销；什么算成功，是能稳定创建和跳转，还是要有统计分析；本轮明确不做什么，例如不做公网开放注册、不做复杂报表、不做多租户计费。

进入 M1 后，愿景会展开成需求池：创建短链、跳转解析、自定义短码、过期时间、错误码、访问统计、权限、审计日志、性能目标、数据备份、开放问题。此时不急着裁剪，因为 M1 的价值是把需求摆出来并分类。进入 M2 后才做取舍：本轮做创建、跳转和过期；访问统计只记录基础日志；管理后台、自定义域名和复杂报表延期；多租户不做。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 只有一句愿景 | 还没有成功标准 | 从 M0 开始 | 直接写需求会失去方向。 |
| 用户列出很多功能 | 已进入需求池阶段 | 做 M1，分类用户故事/NFR/开放问题 | 把方案当需求会过早锁死设计。 |
| 需求越写越多 | 需要裁剪 | 进入 M2，写清出范围 | 范围膨胀会拖垮 M3/M5。 |
| 关键未知无法判断 | 需要调研证据 | 插入 M0.5，但不替代 M 阶段 | 把调研结论当决策会绕过 HITL。 |
| M2 仍说不清边界 | 不能进入 M3 | 回到 M0/M1 或补调研 | 架构会为未确定需求背锅。 |

判断 M0 是否完成，可以看它能否约束后续讨论。如果任何人仍能把短链服务解释成营销追踪平台、内部分享工具或多租户 SaaS，说明 M0 还没收敛。判断 M1 是否完成，可以看需求池是否足够支撑取舍；如果只有“短链功能”四个字，M2 无从裁剪。判断 M2 是否完成，可以看架构师能否据此设计系统；如果范围仍说“统计以后再看”，却没说明本轮是否记录事件，M3 会被迫猜。

开放问题要显式管理，而不是混在需求里。例如“短码是否允许用户自定义”可以标成延期；“链接过期后返回 404 还是专用错误页”可以交给 HITL；“短码碰撞概率如何控制”可以进入 M3；“是否必须支持高并发”可能需要 M0.5 调研。每个开放问题都有去向，M2 才能真正通过。

## 最小流程

1. M0 写清问题、目标用户、成功标准和不做声明。
2. M1 把愿景展开为用户故事、功能需求、非功能要求、约束和开放问题。
3. M2 把需求裁成当前范围、延期范围和明确不做范围。
4. 对每个开放问题标注处理方式：立即决策、进入 M0.5、延期或交给 HITL。
5. 只有范围能指导架构时，才进入 M3。

## 例子

“做一个短链服务”只是愿景，不是范围。M0 要确认为什么做、给谁用、成功指标是什么；M1 要列出创建短链、跳转、统计、权限、过期策略等需求；M2 要决定本轮只做创建和跳转，统计和后台延期。这样 M3 才能设计接口和数据，而不会同时背上所有可能功能。

一个更完整的 M0-M2 产物可以这样理解：

M0 结论：面向内部研发与运营同学，解决长链接分享不便和过期控制问题；成功标准是能稳定创建短链、跳转原链接、处理过期和不存在；本轮不做公网注册、不做营销报表、不做复杂权限。

M1 需求池：创建短链、跳转解析、过期时间、错误码、基础审计、CLI 或 API 调用、最低可用性、数据备份、未来访问统计、后台管理、用户权限。开放问题包括短码长度、冲突策略、存储选择和错误响应格式。

M2 范围：本轮 in scope 是创建 API、跳转解析、过期策略、基础审计日志和测试/CI；out of scope 是后台、统计图表、自定义域名、多租户和公网开放；延期项保留为未来 Roadmap 候选。

反例是 M2 写成“实现短链服务 MVP”。这句话看似收敛，其实没有说明 MVP 包含哪些功能、不包含哪些功能、哪些问题延期。它会把裁剪责任推给 M3 和后续编码阶段。

## 维护者注意事项

维护 M0-M2 文档时，要让每个阶段的产物能被下一阶段消费。愿景应该能约束需求；需求应该能支撑裁剪；范围应该能指导架构。如果某段文字只是“看起来完整”，但不能帮助下一阶段做判断，就需要重写。

不要把示例项目写得过于特殊。短链例子服务的是阶段推理，不是为了规定所有项目都要有 API、数据库或过期策略。外部项目可以替换领域，但仍应保留 M0 明确方向、M1 展开需求、M2 裁剪范围的节奏。

如果 M2 里出现大量“待定”，应明确这些待定项的处理方式。没有去向的待定项会在 M3 变成架构猜测，在 M5 变成 Roadmap 膨胀。

## 边界与反模式

- 不要跳过“不做声明”；否则范围会自然膨胀。
- 不要把技术方案写成需求；需求描述问题，方案留给 M3。
- 不要把开放问题伪装成已决策项；否则后续会返工。
- 不要让 M2 只是复制 M1；M2 必须裁剪。
- 不要在 M0-M2 写过多实现细节；那会提前绑死架构。

## 读完后你应该能做什么

- 能区分愿景、需求和范围。
- 能判断一个项目是否能进入 M3。
- 能写出有效的不做声明。
- 能识别需求池膨胀和开放问题伪装。
