# 决策史、失败案例、近期 ADR 与维护手册

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

Harness 是失败驱动演进出来的系统。很多看似繁琐的规则、hook、状态目录和验收纪律，都来自真实失败后的修复。长期维护者如果只看当前文件，不看决策史，就容易删掉“看起来多余、实际上防复发”的护栏。

本篇把阶段演进、失败案例、proposal、ADR、acceptance/backfill 组织成一张维护地图。读完后，你应该知道改流程、改规则、改插件、改知识层或发布 portable 文档时，哪些历史决策不能随手推翻。

## 核心模型

Harness 的维护闭环是：

1. 发生失败或发现改进机会。
2. 记录 failure-record 或 proposal，说明现象、根因和候选方案。
3. 对长期架构选择写 ADR，对长期行为约束写 Rules。
4. 增加测试、验收或行为验证，证明修复有效。
5. 更新知识层和手册，让后续维护者知道为什么这样设计。

这条链路的重点是“修复必须沉淀”。只修代码不回写规则、测试、ADR 或手册，同类问题会在下一次会话中复发。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 要删除一条规则 | 可能是失败驱动护栏 | 先找对应失败案例、ADR 或验收背景 | 删除后同类失败复发。 |
| 要改插件加载方式 | 涉及近期 ADR | 核对 loader 边界、first-run、state dir、restore 决策 | 安装态不可恢复。 |
| 要新增规则 | 判断是否通用、项目定制或一次性 | 通用才进 core，项目特有进定制层 | 规则污染其他项目。 |
| 要新增知识 | 判断是否有长期价值 | 规范、决策、失败、验收优先 | 知识库被流水日志污染。 |
| 要发布 portable 文档 | 必须正文自足 | 内嵌关键背景，来源只作列入原因和离线依据说明 | 外部读者需要原仓才能读懂。 |

## 近期 ADR 维护地图

| ADR | 解决的问题 | 改相关能力时必须保留什么 |
|---|---|---|
| ADR-013 plugin loader boundary | 插件 loader 边界不清 | loader 只负责原生组件，其他能力需有清晰边界。 |
| ADR-015 first-run default auto bootstrap | 安装后首次会话缺脚手架 | 首次运行应默认落盘基础结构，并保留逃生阀。 |
| ADR-016 state dir project-local | 跨项目状态泄漏 | 状态哨兵必须项目本地化，不能跨项目共享。 |
| ADR-017 CLAUDE.md import restore | 常驻契约缺失或过薄 | 常驻契约必须可恢复，hook 只能做桥接和兜底。 |

这些 ADR 共同说明：Harness 已经从“本仓能跑”进入“安装态可恢复、可迁移、可分发”的阶段。

ADR 修改最容易低估影响面。一个 ADR 往往不只是一段说明文字，它可能支撑规则、hook、模板、测试、验收、portable 文档和用户操作习惯。修改 ADR 时，应把它当成改变系统契约，而不是改一页历史记录。若新结论与旧 ADR 相反，还要说明旧失败模式为什么不再成立，或者新的护栏如何覆盖旧风险。

评估 ADR 修改时，可以先问四个问题：它改变了谁的责任边界；它改变了哪些默认行为；它改变了哪些恢复路径；它改变了哪些证据要求。例如把 state dir 从项目本地改为全局，不只是路径变化，还会影响隐私、跨项目污染、恢复定位、插件迁移和 portable 文档里的运行时边界。

## 最小维护流程

1. 改动前先判断触碰的是流程、规则、Skill、知识层、运行时还是分发层。
2. 查找是否已有 ADR、failure-record、proposal 或验收记录解释当前设计。
3. 如果要改变长期行为，先写清原因、替代方案和后果。
4. 更新权威源，再更新 linked/portable 手册。
5. 增加测试、行为验收或回填证据，证明改动不是口头承诺。
6. 收尾时记录维护结论，避免下一次重复争论。

做影响面评估时，可以沿着六条线查。第一是规则线：是否有 Rules、常驻契约或操作纪律引用该决策。第二是运行线：hooks、commands、setup、restore 是否受影响。第三是质量线：测试、CI、行为验收、acceptance/backfill 是否要更新。第四是分发线：core/profile、manifest、模板和版本策略是否要同步。第五是知识线：linked/portable 手册、索引、来源依据是否要改。第六是用户线：HITL 点、人工操作手册、迁移说明是否需要重新告知。

如果某个 ADR 修改没有任何下游影响，通常要怀疑两种情况：要么它其实不是 ADR，只是普通备注；要么影响面没有查全。真正的 ADR 应该能解释长期取舍，也应该能在后续维护中约束具体动作。

## 例子

维护者觉得 first-run bootstrap 太复杂，想删掉自动落盘逻辑。只看当前代码，它可能像额外负担；但从决策史看，它解决的是安装后首次会话没有脚手架、Agent 无法进入 Harness 约束的问题。正确做法不是直接删除，而是评估是否还有等价恢复路径。如果没有，就必须保留或设计新的兜底。

再看 ADR-016 的修改例子。若维护者想把项目本地状态目录改成全局缓存，需要先列出影响面：

1. 规则线：状态隔离规则要改，跨项目污染边界要重新定义。
2. 运行线：hook 读写路径、setup 和 restore 都要同步调整。
3. 质量线：需要增加跨项目污染测试和恢复错位验收。
4. 分发线：插件迁移文档要说明全局缓存如何清理和回滚。
5. 知识线：portable 文档要把“项目本地化”的判断改成新的边界。
6. 用户线：HITL 风险要提示用户，让用户知道隐私和隔离策略变化。

若这些动作没有同步，ADR 文本即使更新了，系统实际仍会处于半旧半新的状态。

反例是只改 ADR 文本，说“以后支持全局状态”，但 hook 仍写项目本地目录，模板仍生成旧结构，质量门禁仍按旧规则验收。这样的修改会制造双事实源：读 ADR 的人以为系统已迁移，运行系统的人仍看到旧行为。正确做法是把 ADR 修改作为一张维护卡，列 tasks、证据链和回滚点。

## 维护者注意事项

维护决策史时，要保留“为什么不是另一个方案”。ADR 的价值不只在结论，也在被放弃方案和失败背景。后续维护者经常会重新提出旧方案，如果文档没有解释当初为什么放弃，就会反复消耗讨论成本。

portable 手册改写 ADR 相关内容时，不应成为新的权威源。它可以解释 ADR 的操作含义、影响面和离线使用方式，但改变决策必须回到 ADR、规则、代码和验收链。手册是读者入口，不是裁决机关。

## 边界与反模式

- 不要把历史流水账塞进正文；要提炼可复用结论。
- 不要只修代码不回写规则、测试、ADR 或手册。
- 不要把 proposal 当成已决策 ADR；proposal 是候选方向，ADR 是已裁决选择。
- 不要为了“简化”删除由真实失败驱动的护栏。
- 不要让 portable 版成为第二权威源；它是解释层，权威修改仍回到规则、代码和 ADR。

## 读完后你应该能做什么

- 能按失败 -> 规则/ADR -> 验收 -> 手册的链路维护知识。
- 能解释近期 ADR 为什么服务安装态可恢复和可分发。
- 能判断一条规则或 hook 是否可能来自真实失败。
- 能维护 portable 手册而不制造第二事实源。
