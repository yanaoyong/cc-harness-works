# Harness 是什么与适用场景

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

本篇回答一个基础问题：Harness 到底是什么，为什么它需要流程、角色、规则、技能、知识层和运行时集成，而不是只靠一个强 prompt。

Harness 的目标不是让 AI 更自由地写代码，而是让 AI 的工作变得可约束、可审计、可恢复、可验证。它把需求、实现、评审、测试、CI、部署、用户确认和知识沉淀组织成一条工程链路，降低 AI 开发中常见的跳步、失忆、角色漂移和假绿风险。

读完本篇后，即使你没有原始项目文档，也应该能判断：哪些任务适合用 Harness，哪些任务太轻不必进入完整流程，以及 Harness 的五个层次分别解决什么问题。

## 核心模型

Harness 可以理解为五层协作系统：

| 层 | 作用 | 它防住的典型风险 |
|---|---|---|
| 编排层 | Owner 判断任务类型、维护状态、委派角色、整合证据。 | Agent 自己既执行又裁决，导致角色混乱。 |
| 流程层 | 10 阶段处理单张变更卡，M0-M5 处理项目级规划。 | 直接编码导致需求边界漂移、状态不可恢复。 |
| 规则层 | Rules 和常驻契约规定默认行为和不可越界的工程纪律。 | 每次会话重新解释规则，导致口径漂移。 |
| 技能层 | Skills 提供阶段性 SOP，例如需求分析、评审、测试、Roadmap 拆解。 | Agent 知道目标但不知道阶段内怎么做。 |
| 知识层 | wiki、codegraph、M0.5 提供历史、代码结构和外部调研证据。 | 凭记忆猜测项目背景或影响面。 |

这五层的关系很重要：知识层提供证据但不直接裁决，Skills 提供做法但不替代 Owner，Rules 约束行为但不替代用户 HITL，流程产生证据但不能脱离真实需求。

Harness 之所以需要五层，而不是只写一份“AI 编码规范”，是因为 AI 开发的风险不是单点风险。需求会漂移，角色会混淆，测试会假绿，会话会中断，工具输出会过期，用户授权会被默认跳过。单个 prompt 最多提醒 Agent 注意这些问题，但不能提供状态容器、阶段回退、独立评审和可复查证据。

五层之间也有优先级。用户目标和 HITL 是最终裁决入口；流程负责把裁决变成可执行阶段；Rules 和 Skills 负责让执行过程稳定；知识层负责降低盲猜；运行时集成负责把这些约束带进实际工具环境。任何一层越权，都会出现典型失败：知识层越权会把摘要当事实，Skills 越权会把 SOP 当裁决，Rules 越权会把一次性偏好常驻化，运行时越权会把某个工具厂商的接线当成通用架构。

## 操作判断

| 场景 | 是否适合 Harness | 推荐动作 | 风险 |
|---|---|---|---|
| 已有项目新增功能 | 适合 | 建立 10 阶段变更卡，明确 spec、tasks、评审、测试和合并证据。 | 直接写代码会让需求边界和验收标准不可追踪。 |
| 从模糊想法启动新项目 | 适合，但应先走 M0-M5 | 先澄清愿景、需求、范围、架构和 Roadmap，再拆成变更卡。 | 没有上游规划时，单卡会不断返工。 |
| 临时问答或解释概念 | 通常不需要完整流程 | 使用轻量对话或 L 旁路，避免强行建卡。 | 把所有小问题都流程化会降低协作效率。 |
| 修复影响面不清的 bug | 适合 | 先用知识层和代码结构证据判断影响面，再进入变更卡。 | 只改眼前错误，可能漏掉调用方和回归路径。 |
| 迁移到另一个项目 | 适合，但要区分 core/profile/模板 | 只迁移通用流程、规则、技能和模板，剥离本项目历史沉淀。 | 把历史实例当模板会污染新项目。 |

判断是否使用 Harness，可以先看三个信号。第一，任务是否会留下项目产物；只要会改代码、文档、规则、模板或状态，就需要某种审计容器。第二，任务是否存在不可逆或难复盘风险；例如合并、发布、规则修改、架构取舍，都需要用户确认和证据链。第三，任务是否跨越多人或多阶段；只要需要评审、测试、CI、部署或后续维护，就不应该只依赖聊天上下文。

反过来，也要识别不该过度使用 Harness 的场景。用户只是问“这个术语是什么意思”，不需要建卡；用户让你读一段错误日志并解释可能原因，也可以先走轻量旁路；用户只是要求列出可选方案，未要求修改项目产物，通常先给分析即可。Harness 的入口判断本身就是治理的一部分：它让重流程服务高风险任务，而不是把所有协作都变重。

## 最小使用步骤

1. 判断任务类型：轻量咨询、10 阶段变更卡、M0-M5 元流程，或维护/分发任务。
2. 如果任务会产生项目产物，建立可审计容器，例如变更卡或元流程实例。
3. 让 Owner 保持编排职责：路由、状态、委派、证据整合和 HITL。
4. 在具体阶段加载最小必要 Skills，不把所有知识都塞进常驻上下文。
5. 用 Reviewer、测试、CI、部署或验收来证明完成，而不是相信 Agent 自述。
6. 收尾时更新状态和必要知识，避免下一次会话重新摸索。

## 例子

用户说：“给一个已有服务加批量价格查询接口。”

不使用 Harness 时，Agent 可能直接写 endpoint、补几个测试，然后宣布完成。但这个过程容易漏掉接口契约、错误码、权限、金额单位、前端消费方式和用户确认。

使用 Harness 时，Owner 先判断这是一张增量变更卡。阶段 1 形成 spec/tasks，HITL-1 冻结范围；阶段 3 才开始实现；阶段 4-6 分别做代码评审、测试和测试评审；阶段 8/9 验证 CI 和运行面；阶段 10 由用户授权合并并沉淀状态。流程看起来更长，但每一步都在防一种可复发的工程风险。

如果同一个用户说：“我想做一个短链系统，但还没想清楚功能边界。”这就不是 10 阶段增量卡的最佳入口，而是 M0-M5。Harness 会先让用户澄清愿景和成功标准，再整理需求和范围，之后才进入架构、工程基线和 Roadmap。等 Roadmap 里的某张任务卡足够具体，才进入 10 阶段。这样做的好处是：单卡不会背负“替项目规划兜底”的职责。

再看一个不适用完整流程的例子。用户问：“HITL 是什么意思？”这只是解释概念，不产生项目产物。合适的回答是直接解释人类确认点、适用阶段和风险边界，而不是创建变更卡。轻量请求被强行流程化，会让用户感觉系统迟钝，也会稀释真正高风险工作需要的注意力。

## 维护者注意事项

维护本篇时，不要把 Harness 描述成某个具体工具的功能集合。工具会变，厂商接线也会变，但五层模型和证据链的思想应保持稳定。新增运行时、插件或模板时，应先判断它属于哪一层，再决定是否需要更新本篇。

如果新增一种任务类型，应同步检查三处：本篇的适用场景表、`03` 的建卡路由、`19` 的 L 旁路边界。否则读者会知道 Harness 是什么，却不知道新场景应该走轻量咨询、变更卡还是元流程。

本篇还要避免过度营销。Harness 的价值不是“让 AI 一定正确”，而是让错误更容易被发现、回退和复盘。描述时应保留这种边界感：流程能降低风险，但不能替代用户判断、真实测试和工程责任。

## 边界与反模式

- 不要把 Harness 理解成 prompt 模板；否则会忽略状态、证据和用户裁决。
- 不要把流程当成官僚动作；每个阶段都对应需求漂移、角色混淆、假绿或不可复盘等风险。
- 不要把知识层命中当成完成证据；知识层只能帮助判断，不能替代测试和 HITL。
- 不要把所有轻量请求都建成变更卡；流程也需要入口判断。
- 不要把插件化理解成复制整个旧项目；可迁移的是通用引擎和模板，不是历史实例。

## 读完后你应该能做什么

- 能用五层模型解释 Harness 为什么不只是 prompt。
- 能判断一个任务是否应该进入 10 阶段、M0-M5 或 L 旁路。
- 能说明 Owner、Rules、Skills、知识层和 HITL 分别负责什么。
- 能识别“AI 已经说完成”为什么不等于工程上真的完成。
