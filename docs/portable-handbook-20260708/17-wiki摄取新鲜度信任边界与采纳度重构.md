# wiki 摄取、新鲜度、信任边界与采纳度重构

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

知识库只有在新鲜、可信、可被使用时才有价值。本篇解释 wiki 摄取为什么要有白名单、新鲜度、lint、信任边界和采纳度验收。

即使没有 wiki，portable 读者也需要理解这个思想：知识沉淀不是“把文件丢进去”，而是选择什么值得沉淀、如何验证、如何避免自动化夺权。

本文中的 wiki 可替换为任何文档知识库或知识索引能力。重点不是组件名，而是查询、摄取、新鲜度和信任边界。

## 核心模型

| 机制 | 作用 | 风险 |
|---|---|---|
| 摄取白名单 | 只吸收规范、决策、失败、验收等高价值材料 | 流水日志污染知识库 |
| 新鲜度 | 确认知识是否跟上项目变化 | 过期知识误导 Agent |
| lint | 检查链接、frontmatter、coverage、reviews | 绿不等于结论正确 |
| 信任边界 | 自动摄取只提供候选，不自动裁决 | 自动化夺权 |
| 采纳度验收 | 验 Agent 是否真的在受益场景使用工具 | 工具存在但没人用 |

摄取和查询是两件事。摄取解决“哪些材料进入知识库”，查询解决“Agent 如何消费知识”。二者之间必须有信任边界：自动摄取可以发现候选，可以更新索引，可以提示维护者，但不能把候选直接变成权威规则；查询可以给出摘要，可以给出来源，可以提示冲突，但不能替 Owner 决定哪个来源胜出。

新鲜度也不是单一时间戳。对于规则和 ADR，新鲜度意味着是否跟最新权威文件一致；对于失败案例，新鲜度意味着修复和预防措施是否已经回填；对于验收材料，新鲜度意味着行为是否仍能复现；对于 portable 文档，新鲜度意味着正文是否吸收了必要背景，而不是只保留旧来源名。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 新增一份流水日志 | 低价值 | 不摄取或只作为原始附件 | 知识库噪声上升。 |
| 新增 ADR | 高价值 | 摄取并保留来源 | 未来维护者找不到决策背景。 |
| 自动摄取发现候选 | 需要信任边界 | 人或规则确认后采用 | 自动化把错误内容变知识。 |
| Agent 没用 wiki | 工具采纳问题 | 加 nudge、简化命令、做行为验收 | 工具存在但实际无效。 |
| portable 改写 | 需要内嵌关键背景 | 把来源结论写入正文 | 离线读者无法追溯。 |

判断一份材料是否值得摄取，可以看它是否能在未来回答一个真实问题。ADR 能回答“为什么这样选”；失败案例能回答“为什么有这条护栏”；验收报告能回答“怎样证明行为真的遵守”；规范能回答“当前规则是什么”。临时日志、一次性命令输出、未解释的草稿通常只适合作为卡内证据，不适合作为知识库入口。

判断自动摄取是否越权，可以看它有没有改变权威事实。如果 hook 自动把新 ADR 候选加入索引，并标记为待确认，这是候选更新；如果 hook 自动改写规则、删除冲突来源或宣布某条旧规则失效，就是越权。自动化应降低维护成本，但不能替代人和规则的确认。

## 最小流程

1. 判断材料类型：规范、决策、失败、验收、样例优先。
2. 摄取前确认来源和价值，避免把临时过程噪声变成知识。
3. 摄取后做结构检查和基本链接检查。
4. 对自动摄取保持信任边界：可提示、可候选、不可自动裁决。
5. 用行为验收检查 Agent 是否真的在正确场景使用知识库。

## 例子

一次变更完成后产生了 summary、review、test、CI 和临时日志。不是所有文件都应进入知识库。ADR、失败案例、验收结论和可复用规则值得沉淀；一次性命令输出如果没有长期价值，最多作为证据附件。否则以后查询“测试守护”时会被大量低价值日志淹没。

再看采纳度例子。项目提供了 wiki-query，但 Agent 处理“为什么不能跳过测试评审”时仍然只凭记忆回答。这说明工具存在不等于工具有效。合格验收应设计真实任务：给 Agent 一个需要历史背景的问题，观察它是否使用 wiki 或 portable 正文、是否带回来源边界、是否把结果交给 Owner 判断，而不是直接放行。

反例是只验命令能跑。`wiki-query` 能返回结果，只证明工具可用；不能证明 Agent 会在正确场景使用，也不能证明输出会被正确解释。采纳度重构要解决的是行为，而不只是 CLI 成功。

## 维护者注意事项

维护摄取规则时，要定期清理低价值条目。知识库噪声越多，Agent 越可能忽略它。高价值材料通常有长期解释力：规则、ADR、失败、验收、阶段规范、可复用样例。

自动摄取失败时，应软失败并可见。静默失败会让读者以为知识新鲜；硬失败阻塞所有流程又会让工具变成负担。合适的状态是：提示未摄取、说明影响范围、给出手动补救路径。

portable 文档维护时，不要把“已列来源”当成新鲜。正文要吸收来源里对读者必要的结论；来源依据只负责说明为什么这条依据保留、补足什么、如何追溯。

## auto-ingest 现状：opt-in、白名单与报数口径

前面讲的信任边界，在自动摄取上有一套具体的落地形态，portable 读者需要理解四点，才能把"自动化只能候选"这条原则落到操作。

第一，自动摄取是 **opt-in**：它由"会话结束"这类收尾钩子挂接、在后台异步运行，但缺省关闭，只有在项目本地配置里显式打开后才会触发（本项目已于 2026-07-07 打开本地开关）。缺省关的意义是：摄取是否运行，应由维护者主动决定，而不是因为环境恰好具备摄取能力就自动 live-fire。把开关默认打开，等于让环境能力替维护者做决定。

第二，即便打开，摄取源也必须先过**白名单（denylist）**过滤。摄取驱动器读到候选源后，要按白名单规则剔除流水日志、模板目录、评审中间稿等噪声源，只放行规范、决策、失败案例、验收等值转文档。跳过这层过滤直接吸收，就越过了信任边界——把不该沉淀的过程噪声灌进了知识层，稀释了后续查询的信噪比。白名单是必守纪律，不是可选优化。

第三，读新鲜度提示的数字时要注意**口径**。新鲜度钩子报的是**过滤前的原始候选数**，不是最终会入库的条目数；因为白名单过滤发生在报数之后，实际摄取条目通常少于提示数字。把它直接读成"将新增 N 条知识"会高估摄取量。正确读法是："检测到 N 个候选源，其中一部分会被白名单剔除。"

第四，积压排空有一条运行纪律：**排空要重跑驱动器**。驱动器单次运行遇到第一个失败源即停（fail-fast），不会跳过失败项继续跑后续候选。所以当积压较多、或某个源摄取失败后，要先修复该源，再**重新运行**驱动器把剩余积压推完，而不能指望单次运行一次排干。

## 边界与反模式

- 不要把所有文件都摄取；知识库不是垃圾桶。
- 不要把自动摄取结果当权威；自动化只能候选。
- 不要用 lint 绿证明内容正确；lint 只检查结构。
- 不要只看工具命令能跑；要验 Agent 是否真的使用。
- 不要在 portable 版保留“去查来源”作为理解前提。

## 读完后你应该能做什么

- 能判断哪些材料适合沉淀为知识。
- 能解释新鲜度、lint 和信任边界的区别。
- 能设计一个 wiki 采纳度验收场景。
- 能把知识来源改写成 portable 正文。
