# 变更卡与持久化记忆

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

AI 会话会中断，上下文会变短，人也会忘记上一轮做到哪里。变更卡和持久化记忆解决的就是这个问题：把一次工作从聊天中拿出来，落到可审计、可恢复的文件结构里。

变更卡不是任务标题，而是一条过程审计链。它记录当前阶段、产物、证据、未决问题、用户确认和收尾状态。

## 核心模型

一张变更卡至少承担四种职责：

| 职责 | 说明 | 典型内容 |
|---|---|---|
| 状态恢复 | 下一次会话能知道做到哪 | `summary.md` |
| 阶段产物 | 每阶段输出能被审查 | spec/tasks/review/test/ci |
| 证据链 | 完成不是自述，而是报告和命令结果 | review、test、CI、deployment |
| 历史审计 | 以后能知道为什么这样做 | 用户确认、决策、风险记录 |

其中 `summary.md` 是恢复入口。它不应是流水账，而应告诉 Owner 当前状态、最近证据、下一步和阻塞点。

持久化记忆的意义，是让下一位执行者不必相信上一轮模型的口头记忆。AI 会话可能被压缩、重启或换人；如果状态只存在聊天里，后续很容易重复工作、跳过未决风险、或者把旧结论当成当前事实。变更卡把“过程”变成可读文件，让恢复从证据开始，而不是从猜测开始。

变更卡还要同时服务两类读者：当前执行者和未来维护者。当前执行者需要知道下一步是什么、阻塞在哪里、哪些文件已经改过；未来维护者需要知道为什么这么做、用户何时确认、哪些风险被接受、哪些验证支撑了合并。`summary.md` 如果只写流水账，就无法服务这两类读者。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 工作会修改项目产物 | 需要变更卡 | 创建卡并维护 `summary.md` | 聊天上下文丢失后无法恢复。 |
| 只是一次问答 | 通常不需要卡 | 走轻量响应 | 过度建卡会增加成本。 |
| 会话中断后恢复 | 以落盘状态为准 | 先读 `summary.md`，再读阶段产物 | 凭记忆恢复会错过未决风险。 |
| 多张卡并行 | 状态必须隔离 | 每卡独立 summary 和证据链 | 证据串卡会污染判断。 |
| 上游项目级产物更新 | 可能影响卡 | 记录引用关系或触发级联评估 | 已完成卡可能基于旧前提。 |

恢复变更卡时，可以按四步做。第一，读 `summary.md`，确认当前阶段、目标、下一步和阻塞。第二，读最近阶段产物，例如 review report、test report 或 CI report。第三，核对工作区状态，确认文件变化是否与 summary 一致。第四，判断是否需要回退阶段，尤其是存在 MUST FIX、测试假绿、用户未授权或上游范围变更时。

多卡并行时，要特别防止证据串卡。一个子 Agent 在 A 卡里发现的风险，不应自动成为 B 卡的完成条件；一个 CI 结果如果针对旧 diff，也不能证明新卡通过。每张卡都要能独立回答：本卡改了什么，证据是什么，用户确认在哪里。

## 最小流程

1. 为产生项目产物的任务创建或定位变更卡。
2. 在 `summary.md` 写清当前阶段、目标、状态和下一步。
3. 每阶段把产物写入卡内对应位置，而不是只留在聊天里。
4. 阶段推进时同步更新状态和证据摘要。
5. 收尾时记录用户授权、合并状态和必要知识维护。
6. 恢复时只信落盘状态，不把模型记忆当事实源。

## 跨会话分段与交接哨兵

超长的卡有一个隐性成本：一次会话越长，主循环每轮要重读的上下文越多，缓存读随卡长单调恶化。应对办法是在安全切点开一个新会话（`/clear` 重开）继续推进。切点具体落在哪、水位到多少才切，属于主流程的水位切点问题，见 [04 篇《十阶段主流程》](04-十阶段主流程.md)；本篇只讲**分段之后如何跨会话交接**，因为这正是持久化记忆要解决的问题。

分段一旦发生，旧会话的上下文会被清掉，新会话不能靠模型的口头记忆续推。这里靠两层落盘：

- **卡目录与 `summary.md` 是跨会话的唯一事实源**：无论切几次会话，新会话恢复工作时都以变更卡目录里的落盘状态为准，而不是上一轮聊天。
- **交接哨兵是一次性信标**：建议分段时，Owner 写一个交接哨兵文件，记录三件事——目标卡目录、下一步该做的阶段、一句交接注，外加写入时间戳。哨兵**不绑定具体会话**，它的使命就是把"续跑指令"从旧会话传给下一个新会话。

交接哨兵的关键约束是**单次消费加时效**，否则它会变成噪音：

- 新会话启动时，如果检测到一个新鲜的哨兵（默认写入 24 小时内，可配置），就把它读成一段"接着做某卡的某阶段"的续推提示。
- 读取的同时哨兵立即失活（被标记为已消费）。用户在新会话里随便输入什么之后，哨兵就不再起作用，也不会劫持再之后的会话——这样一个过期或陈旧的哨兵不会反复把无关会话拽回旧任务。
- 哨兵失活后，续跑完全回到常规恢复路径：先读 `summary.md` 确认阶段和阻塞，再读当前阶段产物。哨兵只给方向，落盘状态才是权威。

这套机制在本项目里被称作 `v0.8.5 · 分段机制 DF-017`。它把原先"靠人记得该在哪切、切完手动交接"的纯人工纪律，升级成半机械化：本地实测中，被切分的长卡主循环缓存成本再省 **20–27%**，而交接不丢、恢复不靠记忆。

## 例子

一张文档改写卡可能没有代码 diff，但仍然需要变更卡。它至少要记录：改写范围、哪些文件完成、验收规则、审查意见、未完成批次和下一步。如果没有这些记录，下一轮 Agent 可能重复改已完成文章，或误以为所有 25 篇都已达标。

再看一个恢复场景。上一轮做到“测试写完，但测试评审发现 mock 过度”。新的 Owner 读到 summary 后，不应该从实现阶段继续，也不应该直接跑 CI 宣布通过。正确恢复是定位测试评审报告，补真实契约或集成验证，再更新 test report，之后才进入 CI。恢复的关键是尊重已落盘的未决风险。

反例是 summary 写成：“今天改了很多文件，基本完成，明天继续。”这句话无法告诉后续执行者当前阶段、哪些证据可信、哪些风险未闭合、用户是否确认。好的 summary 应该像交接单，而不是日记。

## 维护者注意事项

维护变更卡机制时，优先保护 `summary.md` 的恢复价值。它不必记录每个细节，但必须记录目标、阶段、状态、关键证据、阻塞、下一步和用户确认。任何让这些信息分散到聊天里的做法，都会削弱持久化记忆。

如果变更卡命名或索引规则变化，应同步更新历史索引和恢复说明。否则卡仍在，但维护者找不到，等同于证据链断裂。

如果某张卡引用了 M0-M5 产物、Roadmap 卡或外部决策，应记录引用关系。上游产物变化时，Owner 才能判断当前卡是否需要级联重开，而不是继续基于旧前提推进。

## 边界与反模式

- 不要把 `summary.md` 写成流水账；它应服务恢复和判断。
- 不要只在聊天里记录用户确认；关键 HITL 要能被后续会话看到。
- 不要把历史变更卡当新项目模板导出；它们是本项目审计历史。
- 不要让多张卡共用模糊状态；每张卡必须独立可恢复。
- 不要在收尾时忘记状态翻牌；否则工作看似完成但流程仍悬空。

## 读完后你应该能做什么

- 能说明为什么变更卡是过程审计链。
- 能写出可恢复的 `summary.md` 应包含什么。
- 能判断哪些任务需要建卡。
- 能识别状态漂移、证据串卡和聊天记忆依赖风险。
