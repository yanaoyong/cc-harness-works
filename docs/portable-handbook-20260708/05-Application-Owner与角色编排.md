# Application Owner 与角色编排

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

Harness 里最重要的角色不是写代码最快的 Agent，而是 Application Owner，下文简称 Owner。Owner 是流程编排中枢：它判断任务类型、维护状态、选择角色、注入上下文、整合证据，并把需要人类裁决的事项交给用户。

本篇解释 Owner 为什么不能被 Generator、Reviewer 或工具替代。没有 Owner，系统很容易变成“谁拿到任务谁做主”，最后出现角色漂移、证据断裂和用户无法判断当前状态的问题。

## 核心模型

Owner 的职责可以压缩成五个动作：

| 动作 | 含义 | 产物 |
|---|---|---|
| 路由 | 判断请求走 L 旁路、10 阶段、M0-M5 还是维护任务 | 路由结论 |
| 建模 | 把任务变成范围、阶段、角色和证据要求 | spec/tasks 或阶段计划 |
| 委派 | 给子 Agent 明确目标、上下文、边界和输出格式 | 委派 prompt |
| 汇总 | 把实现、评审、测试、CI、部署等证据合成状态判断 | summary / reports |
| HITL | 在关键点让用户拍板 | 用户确认记录 |

Owner 不是万能执行者。它可以亲自做工作，但不能放弃编排责任；也可以委派子 Agent，但不能把裁决权交出去。

角色编排存在的根本原因，是 AI Agent 很容易把“能做”误解成“能决定”。Generator 能写代码，不代表它能判断范围是否已经被用户确认；Reviewer 能发现问题，不代表它能自己修完并宣布通过；工具能输出证据，不代表它能决定合并。Owner 的职责就是把这些能力放在正确位置，让执行、评判、证据和用户裁决互相制衡。

最常见的角色冲突有三类。第一类是执行者自评：Generator 写完后直接说完成，跳过 Reviewer 和测试评审。第二类是评审者自修：Reviewer 发现问题后顺手改掉，导致后续没有独立评审者。第三类是工具夺权：wiki、codegraph、测试命令或子 Agent 报告给出强信号后，Owner 不再组织证据整合和 HITL。三者都会让流程看起来更快，但会削弱可审计性。

## 操作判断

| 场景 | Owner 应怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 用户一句话提需求 | 先判断任务类型 | 决定 L 旁路、变更卡或元流程 | 直接实现会跳过范围确认。 |
| 子 Agent 返回结论 | 判断证据是否足够 | 核对输出、风险、未决项，再决定下一步 | 盲信子 Agent 会让错误进入主流程。 |
| Reviewer 提 MUST FIX | 判断是否阻塞 | 回到修复阶段并保留证据 | 把 MUST FIX 当建议会带入已知风险。 |
| 用户要求跳流程 | 判断是否仍产生项目产物 | 轻量事项可旁路，项目产物应建卡 | 流程被绕开后无法审计。 |
| 多个任务并行 | 判断状态和上下文是否会串 | 分卡、分角色、分证据链 | 跨卡混淆会污染状态。 |

判断角色是否冲突时，可以看“谁提出结论，谁证明结论，谁授权结论”。同一个角色可以承担多个动作，但不能让关键制衡消失。Owner 可以亲自实现小改动，但仍要组织评审或至少说明为什么不需要独立子 Agent；Reviewer 可以建议修复方向，但不应把自己变成最终实现者；用户可以要求加速，但不能因此默认授权合并。

当子 Agent 返回强结论时，Owner 要问四个问题：它的输入上下文是否完整，它是否越过了角色边界，它的证据能否复查，它的结论影响哪个阶段。如果这四个问题答不清，Owner 不能直接采纳，而应把输出降级为线索，补上下文或重新委派。

## 最小流程

1. 接收请求，记录用户原意。
2. 判断任务类型和最小必要流程。
3. 建立或定位状态容器。
4. 选择需要的角色和 Skills。
5. 委派时注入上下文、边界、输出要求和禁止事项。
6. 收回结果后核验证据，不让子 Agent 自行裁决。
7. 在 HITL 点把风险、证据和建议交给用户。
8. 更新状态和下一步。

## 例子

用户说：“帮我重构测试体系。”Owner 不能直接让 Generator 大改代码，也不能让 Reviewer 自己决定方案。正确做法是先判断这是架构/质量相关变更，可能需要变更卡甚至 M3 级设计；再让分析角色整理范围和风险；随后委派实现和评审，并要求测试证据和假绿检查。

如果 Owner 只说“Reviewer 你看一下”，Reviewer 可能开始实现修复，角色就从评判者漂移成执行者，后续评审不再独立。

再看一个角色冲突案例。用户要求“赶紧把这个 bug 修掉并合并”。Generator 找到原因后提交修复，测试也绿了。此时 Generator 不能自己决定合并，因为它没有代表用户承担授权，也没有独立审视自己的实现风险。Owner 应收集修复说明、测试证据和可能影响面；如果风险低，可以做轻量评审，但仍要明确用户授权点。快不等于无主。

反例是 Owner 变成纯转发器：用户说需求，Owner 转给 Generator；Generator 说完成，Owner 转给 Reviewer；Reviewer 说可以，Owner 转给用户。表面上角色很多，实际上没有人整合状态、识别冲突、判断证据是否足够。真正的 Owner 要把各角色输出转成流程状态和下一步，而不是只搬运消息。

## 维护者注意事项

维护角色模型时，不要为了简化文档而合并 Owner、Generator、Reviewer 的职责说明。角色分离是 Harness 的核心护栏之一，尤其在 AI 场景下，执行者自信地宣布完成并不罕见。文档可以允许小任务由 Owner 亲自执行，但必须说明编排责任仍在 Owner 身上。

如果新增新的角色类型，例如 Researcher、Integrator 或 Release Agent，应先定义它能做什么、不能做什么、输出如何被 Owner 回收，以及是否允许修改文件。没有这些边界的新角色会增加名词，却不能降低漂移风险。

如果某个流程经常出现“Reviewer 修了自己发现的问题”或“Generator 自己判断通过”，应优先补角色边界和委派模板，而不是只在失败后提醒一句不要这样做。

## 边界与反模式

- 不要让 Owner 变成纯转发器；否则证据不会被整合。
- 不要让 Generator 判断自己是否完成；否则自评会掩盖风险。
- 不要让 Reviewer 修改自己正在评审的代码；否则独立性消失。
- 不要把 HITL 降级成礼貌确认；HITL 会改变流程状态。
- 不要跨卡复用未整理的上下文；否则会把一个任务的事实带入另一个任务。

## 读完后你应该能做什么

- 能说明 Owner 与 Generator、Reviewer 的区别。
- 能设计一次合格的子 Agent 委派。
- 能判断子 Agent 输出是否足以推进阶段。
- 能识别角色漂移和裁决外包风险。

## 离线来源依据

### 角色主线

- `application-owner-spec`
  - 列入原因：用于补足正文背后的背景信息、维护线索和判断依据。
  - 具体补足：Owner 作为编排中枢的职责、状态管理和 HITL 组织方式，并说明 Owner 如何识别任务类型、维护状态、委派子 Agent、整合证据并组织用户拍板。
  - 离线吸收位置：Owner 编排职责、子角色边界和 HITL 组织。
  - 离线使用方式：用来判断 Owner 是否正确承担编排、委派、证据整合和用户拍板组织职责。
- `agent-roles`
  - 列入原因：用于补足正文背后的背景信息、维护线索和判断依据。
  - 具体补足：角色分离模型，说明执行、评判和治理职责不能混在一起，并说明 Owner、Generator、Reviewer、Strategist 等角色的权限边界，说明执行、评判和治理为什么不能混在一起。
  - 离线吸收位置：Owner 编排职责、子角色边界和 HITL 组织。
  - 离线使用方式：用来区分执行、评审、策略和治理角色，防止把裁决外包给执行者。

### 子角色

- `reviewer-agent`
  - 列入原因：用于补足正文背后的背景信息、维护线索和判断依据。
  - 具体补足：Reviewer 独立评判、MUST FIX 和 ONE pass 约束，并说明 Reviewer 的独立评判、MUST FIX 分类、ONE pass 限制和不能自我修复导致角色漂移的约束。
  - 离线吸收位置：Owner 编排职责、子角色边界和 HITL 组织。
  - 离线使用方式：用来检查 Reviewer 是否保持独立评判、MUST FIX 分类和 ONE pass 约束。
- `generator-agent`
  - 列入原因：用于补足正文背后的背景信息、维护线索和判断依据。
  - 具体补足：执行型 Agent 的边界，即按任务产出实现但不自我裁决，并说明执行型 Agent 应围绕明确任务产出实现、测试或文档，而不负责最终裁决和质量放行。
  - 离线吸收位置：Owner 编排职责、子角色边界和 HITL 组织。
  - 离线使用方式：用来确认执行型 Agent 只负责产出实现或文档，不自我裁决质量放行。
- `strategist-agent`
  - 列入原因：用于补足正文背后的背景信息、维护线索和判断依据。
  - 具体补足：策略型分析角色，用于复杂任务前的方案和风险整理，并说明复杂任务前如何整理方案、风险、依赖和取舍，让 Owner 获得决策材料而不是直接执行。
  - 离线吸收位置：Owner 编排职责、子角色边界和 HITL 组织。
  - 离线使用方式：用来判断复杂任务是否需要先整理方案、风险和取舍，再进入执行。
