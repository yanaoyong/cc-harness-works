# Skills 体系与阶段触发

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

Skills 是 Harness 的阶段性 SOP 层。Rules 告诉 Agent “不能越过什么边界”，Skills 告诉 Agent “在这个阶段具体怎么做”。

Skills 不应该全部常驻加载。正确做法是由 Owner 根据任务和阶段选择最小必要 Skill，读完后把产物写回变更卡或元流程实例。

## 核心模型

| Skill 类型 | 作用 | 例子 | 边界 |
|---|---|---|---|
| 10 阶段 Skill | 支撑需求分析、编码、评审、测试、CI、部署 | request-analysis、coding、code-review | 只服务单卡阶段 |
| 元流程 Skill | 支撑 M0-M5 | vision、requirements、architecture、roadmap | 服务项目级规划 |
| 知识/旁路 Skill | 支撑 wiki、codegraph、M0.5 | research-discovery、wiki-engine | 提供证据不裁决 |
| profile Skill | 支撑具体技术栈 | Python、React/Vite、FE integration | 不应污染 core |

Skill 的价值在于把阶段内做法固定下来，同时避免上下文常驻过胖。

Skills 和 Rules 的区别必须守住。Rules 规定长期边界，例如不能绕过 HITL、不能破坏用户改动；Skills 提供阶段做法，例如需求分析怎么拆 spec/tasks、代码评审怎么写 findings、测试怎么检查假绿。如果把 Skill 当规则常驻，会让上下文变胖；如果把 Rules 写进 Skill 又不常驻，关键边界可能在不加载该 Skill 时消失。

Skill 误用常见于三个场景。第一，加载过早：还没确认任务类型就加载 coding Skill，导致 Agent 直接实现。第二，加载过多：为了保险把所有 Skill 都塞进上下文，结果重要步骤被噪音淹没。第三，加载错层：把 profile Skill 当 core Skill，让某个技术栈习惯污染所有项目。Owner 的阶段判断就是为了避免这些误用。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 阶段 1 需求分析 | 需要把自然语言转成 spec/tasks | 加载 request-analysis 类 Skill | 不加载会让需求边界模糊。 |
| M3 架构设计 | 需要 C4、接口、ADR、Rules | 加载 architecture/interface/ADR Skill | 直接实现会跳过架构决策。 |
| 前后端契约不一致 | 需要跨层联调 SOP | 加载 FE integration 或契约测试 Skill | 单端测试会漏 mismatch。 |
| 技术栈特有命令 | 属于 profile 能力 | 加载对应 profile Skill | 写入 core 会污染其他项目。 |
| 只是普通咨询 | 不需要 Skill | 轻量回答或 L 旁路 | 过度加载会浪费上下文。 |

选择 Skill 时，可以按“阶段、产物、风险”三步判断。当前阶段决定候选 Skill；预期产物决定需要哪些步骤；主要风险决定是否需要额外 Skill。例如阶段 1 需要 request-analysis，因为产物是 spec/tasks；如果涉及接口契约，还可能补 interface design；如果只是解释概念，不需要任何 Skill。不要按“这个 Skill 名字看起来相关”来加载。

Skill 输出必须回写到流程产物。需求分析 Skill 的结果应进入 spec/tasks；代码评审 Skill 的结果应进入 review report；测试 Skill 的结果应进入 test report；Roadmap Skill 的结果应进入 Roadmap 卡。如果输出只留在聊天里，下一次会话无法恢复，Skill 就没有真正服务流程。

## 最小使用步骤

1. Owner 判断当前阶段和任务类型。
2. 选择最小必要 Skill，而不是把所有 Skill 预加载。
3. 读取 Skill 的步骤、输入和产出要求。
4. 按 Skill 产出阶段产物。
5. 将产物写回变更卡、元流程实例或报告。
6. 如果 Skill 元数据、注册方式和实际文件不一致，先修一致性再依赖它。

## 例子

用户要求“规划一个新模块”。如果直接加载 coding Skill，Agent 会过早进入实现。正确做法是判断这是元流程任务：先用需求和范围 Skill 收敛 M1/M2，再用架构 Skill 做 M3，最后用 roadmap Skill 做 M5。

另一个失败场景是 frontmatter 漂移：Skill 文件存在，但元数据字段、manifest 注册和实际发现方式不一致。结果是 Owner 以为能调用，运行时却找不到。维护 Skills 时必须同时检查文件内容和发现机制。

再看 Rules/Skills 混用的例子。测试守护中“total=0 不能算通过”应是规则，因为它是长期质量边界；“如何写单元测试、如何组织断言、如何运行 CI”应是 Skill，因为它是阶段做法。把前者只写进测试 Skill，会导致不加载该 Skill 时边界消失；把后者写进常驻规则，会让所有会话背着过多操作步骤。

反例是“为避免漏掉步骤，把所有 Skills 全部读一遍”。这会消耗上下文，也会让 Agent 同时看到互不相关的 SOP。正确做法是先由 Owner 判断阶段，再加载最小必要 Skill；如果中途阶段变化，再加载新的 Skill，而不是一开始全量注入。

## 维护者注意事项

维护 Skills 时，要同时维护内容、元数据和发现机制。只有文件存在不够；frontmatter、manifest、插件注册和实际调用路径都要一致。否则 portable 手册说“加载某 Skill”，运行时却无法发现。

新增 Skill 前，先判断它是否真的复用。一次性任务步骤应留在变更卡；反复出现、阶段明确、产物稳定的做法才值得沉淀成 Skill。Skill 太多会导致选择成本上升，Owner 反而更难判断该加载哪个。

如果某个 Skill 经常被误用，应补触发条件和反例，而不是只改名称。好的 Skill 文档不仅说明“怎么做”，也说明“什么时候不该用”。

## 边界与反模式

- 不要把 Skill 当成硬门禁；它是阶段 SOP，不是自动裁决器。
- 不要全部常驻加载；否则上下文会变胖，重点反而降低。
- 不要让 profile Skill 污染 core；技术栈适配应按项目加载。
- 不要只引用 Skill 名称而不读具体步骤；名称不能替代 SOP。
- 不要让 Skill 产物停留在聊天里；必须回写到流程产物。

## 读完后你应该能做什么

- 能区分 Rules 和 Skills。
- 能判断某阶段应该加载什么 Skill。
- 能识别 core/profile Skill 边界。
- 能发现 Skill 元数据和注册漂移风险。
