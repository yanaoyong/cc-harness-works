# 插件化分发、模板导出与可移植性

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

Harness 要迁移到其他项目，不能把本仓所有历史一锅端复制出去。插件化和模板导出要区分：哪些是通用引擎，哪些是技术栈 profile，哪些是项目配置，哪些只是历史实例。

本篇解释 core/profile、开发态/安装态/模板态/分发态，以及 portable 文档如何避免依赖本仓路径。

## 核心模型

| 语境 | 目标 | 应包含 | 不应包含 |
|---|---|---|---|
| 开发态 | 维护插件源码 | core/profile 源资产 | 安装后的本地状态 |
| 安装态 | 在消费项目运行 | `.harness/`、运行时桥接 | 开发仓历史 |
| 模板态 | 离线复制骨架 | `_TEMPLATE`、项目基线 | 旧项目变更卡 |
| 分发态 | marketplace/plugin 安装 | manifest、版本、资产清单 | 未声明资产 |

core 承载通用流程、Rules、Skills、hooks 和组件；profile 承载技术栈绑定，例如 Python、React/Vite 或 FE integration。

插件迁移的关键问题不是“文件能不能复制”，而是复制后谁拥有它、谁更新它、它是否携带旧项目状态。一个文件如果代表通用纪律，应由 core 维护；如果代表技术栈命令，应由 profile 维护；如果代表消费项目选择，应在安装后生成或由项目配置覆盖；如果代表历史证据，只能留在原项目。迁移计划要先解决归属，再解决路径。

迁移前可以给每个文件打四个标签：资产类型、运行语境、状态属性和更新责任。资产类型包括 rule、skill、hook、template、doc、fixture、example；运行语境包括开发态、安装态、模板态、分发态；状态属性包括源资产、生成物、本机状态、历史记录；更新责任包括 core、profile、项目、用户。打完标签后，分发清单自然会比手工挑文件稳定。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 文件是通用流程 | 属于 core | 放入 core plugin | 缺失会导致安装后不可用。 |
| 文件绑定技术栈 | 属于 profile | 放入 profile plugin | 写进 core 会污染其他项目。 |
| 文件是历史变更卡 | 属于项目沉淀 | 不导出为模板 | 新项目继承旧历史。 |
| 文件是本机状态 | 不可分发 | 排除 | 跨项目状态泄漏。 |
| 文档要对外发布 | 需要 portable | 内嵌关键背景，去掉仓内依赖 | 外部读者读不懂。 |

判断模板厚度时，要看新项目需要多少上下文才能启动。thin 模板只给最小目录、常驻契约和安装入口，适合已有工程接入；medium 模板带常用 Rules、Skills、质量门禁和 profile，适合快速建立标准流程；thick 模板带示例卡、示例验收和完整演练，适合教学或样板项目。模板越厚，越要警惕把旧项目历史当成通用能力。

判断 manifest 是否可信，要做三类对账：manifest 声明的资产是否都存在；实际分发目录是否存在未声明资产；安装后落盘结果是否符合清单边界。对账时特别关注 `.git`、本机状态、旧变更卡、临时报告、缓存、测试输出、私有路径和密钥样式文本。可移植性不是“能打包”，而是“打包后不会泄漏、不漂移、可恢复”。

## 最小流程

1. 为每类资产标注语境：开发态、安装态、模板态、分发态。
2. 将通用引擎放入 core，将技术栈适配放入 profile。
3. 导出模板前剥离历史变更卡、本机状态和演示残留。
4. 用 manifest 明确分发资产清单。
5. 首次运行提供 bootstrap 和 restore 兜底。
6. 对 portable 文档做离线验收，确保不依赖本仓 wiki。

## 迁移后验收链

插件迁移完成后，不应只检查“包能安装”，还要证明它没有泄漏旧项目、没有丢失恢复路径，并且能在新项目里跑出最小证据链。建议按以下顺序验收：

| 验收项 | 检查什么 | 通过证据 | 失败处理 |
|---|---|---|---|
| manifest 对账 | 声明资产都存在，未声明资产没有混入 | manifest 清单、实际文件清单、差异说明 | 修 manifest 或移除多余资产 |
| 空白项目安装 | 不依赖原仓路径或历史状态 | 干净项目安装记录、生成文件列表 | 回到资产归属重新拆包 |
| first-run 恢复 | 首次会话能生成脚手架和常驻契约 | bootstrap 输出、入口文件、恢复说明 | 补 setup/restore 兜底 |
| 状态目录隔离 | 新项目不读取旧项目状态 | 项目本地状态路径、隔离验证 | 回到 state dir 设计和 ADR 判断 |
| 历史泄漏扫描 | 旧变更卡、私有路径、缓存、报告不进入包 | 扫描结果或人工清单 | 排除历史实例并重建包 |
| 最小质量门禁 | 能跑一张最小卡并留下证据 | 范围、测试/检查、CI 或等价验证、HITL 记录 | 限制发布，补质量链 |
| 回滚验证 | 升级失败时能回到上一版或停在可诊断状态 | 版本记录、回滚步骤、失败诊断 | 补版本策略和恢复手册 |

这条验收链把“可分发”拆成七个问题：资产有没有对齐，环境能不能独立安装，首次运行能不能自恢复，状态是否隔离，历史是否泄漏，质量门禁是否能跑，失败是否能回滚。任何一项缺失，都应标成发布风险，而不是用“安装成功”覆盖。

## 例子

`test-guard` 这类通用测试纪律适合 core；React/Vite 的具体命令适合 profile；某次 `feat-wiki-auto-ingest` 的变更卡属于历史实例，不应进入新项目模板；`.codegraph/` 或本机状态目录属于派生状态，不应分发。

更具体的迁移操作可以这样做：

1. 列出候选文件清单，先不要移动文件。
2. 逐项标注 core、profile、project、history 四类归属。
3. 把通用规则、通用技能和通用质量纪律放入 core。
4. 把 React/Vite、Python、FE integration 等技术栈命令和验收样例放入对应 profile。
5. 把项目名、仓库路径、业务域配置改成安装后占位或项目覆盖。
6. 排除历史变更卡、真实失败记录、本机状态、缓存和旧报告。
7. 用空白项目安装，验证首次运行能生成脚手架、恢复常驻契约、查询状态并执行最小质量门禁。

反例是把一个已经运行很久的项目目录直接打包成模板。这样会把旧 Roadmap、旧状态、旧验收报告、旧路径甚至私有知识带到新项目，导致 Agent 在新项目里引用不存在的历史。正确做法是把旧项目当素材库，从中提炼规则、技能和示例，再通过 manifest 明确哪些进入分发包。

## 维护者注意事项

维护插件和模板时，要坚持“源资产只在一个地方修改”。如果 core、profile、模板目录和安装态镜像都能手改同一份规则，版本很快会漂移。推荐做法是明确开发源、生成镜像和安装结果，并在发布前做清单对账。

插件迁移还要记录兼容性和回滚。每次升级应说明支持的运行时版本、影响的 hooks/commands/Rules/Skills、是否需要重新 bootstrap，以及失败时如何恢复到上一版。没有回滚说明的分发包，在消费项目出问题时只能靠人工猜测。

## 边界与反模式

- 不要把本项目历史沉淀导出成通用模板。
- 不要把栈绑定写死进 core。
- 不要忽视开发态和安装态路径差异。
- 不要让 manifest 与实际资产漂移。
- 不要在 portable 文档中保留本仓专属路径作为必需条件。

## 读完后你应该能做什么

- 能区分 core、profile、模板和历史实例。
- 能判断一个文件是否应该进入分发包。
- 能说明 first-run bootstrap、state dir 本地化和 CLAUDE restore 如何提升可移植性。
- 能为 portable 文档制定离线验收规则。
