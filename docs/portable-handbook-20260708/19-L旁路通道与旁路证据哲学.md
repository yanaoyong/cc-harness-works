# L 旁路通道与旁路证据哲学

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

不是所有用户请求都应该建卡，也不是所有证据都应该进入硬门禁。L 旁路解决轻量请求和旁路证据的边界：什么时候可以轻量处理，什么时候必须升格为流程，工具证据如何影响判断但不夺取裁决权。

## 核心模型

| 类型 | 说明 | 是否进入门禁 |
|---|---|---|
| 轻量咨询 | 不产生项目产物的问题解释、状态查询、建议 | 否 |
| 旁路证据 | wiki、codegraph、M0.5 等提供的背景和线索 | 否，作为判断输入 |
| 变更卡 | 会修改项目产物的工作 | 是 |
| HITL 裁决 | 用户对范围、风险、合并的确认 | 是 |

旁路的核心是“证据不夺权”。工具可以强烈影响 Owner 判断，但不能自动决定阶段通过。

L 旁路有两层含义。第一层是轻量请求旁路：用户只是问概念、查状态、要建议，不产生项目产物，就不必强行建卡。第二层是证据旁路：wiki、codegraph、M0.5、`rg` 等工具可以在主流程之外提供背景和线索，但这些线索进入 Owner 判断后，仍要回到变更卡、质量门禁或 HITL。两层都强调“轻量”和“辅助”，不是绕开治理。

“证据不夺权”可以用一句话记住：旁路证据可以改变 Owner 的建议，不能直接改变流程状态。wiki 结果可以让 Owner 建议回到规则权威源；codegraph 结果可以让 Owner 要求补测试；M0.5 结果可以让 Owner 提议某方案；但阶段是否通过、范围是否接受、合并是否授权，仍要由流程和用户确认。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 用户问概念 | 不产生项目产物 | L 旁路回答 | 强行建卡会过重。 |
| 用户要求改文件 | 产生项目产物 | 升格为变更卡 | 旁路改文件无审计。 |
| codegraph 给影响面 | 旁路证据 | 写入判断，仍回源码/测试核对 | 工具误报会变裁决。 |
| M0.5 给调研结论 | 旁路证据 | 交 Owner/HITL 判断 | 调研替用户拍板。 |
| 咨询中延伸出任务 | 边界变化 | 明确升格点 | 悄悄开发会破坏流程。 |

判断是否仍可旁路，要看是否产生持久影响。如果只是解释“为什么 portable 不能只留来源名”，可以旁路回答；如果开始修改 25 篇文档，就产生项目产物，应升格为变更卡。查询 wiki 或 codegraph 本身不一定建卡，但如果查询结果导致修改规则、改代码、调整 Roadmap，就必须进入相应流程。

判断旁路证据是否越权，要看它是否被用来替代门禁。例如“codegraph 没有调用方，所以不用测试”“M0.5 推荐方案 A，所以用户已经同意”“wiki 说规则存在，所以当前卡失败”都是越权。正确写法是“这些证据提示下一步应补哪些检查或交给谁确认”。

## 最小流程

1. 判断是否产生项目产物。
2. 不产生项目产物时可走 L 旁路，保持轻量。
3. 一旦需要修改文件、生成长期文档或改变状态，升格为变更卡或元流程。
4. 使用 wiki/codegraph/M0.5 证据时，标注来源和置信边界。
5. 让 Owner 整合证据，并在需要时交 HITL。

## 例子

用户问：“为什么这篇 portable 文档不能只保留来源名？”这是轻量咨询，可以直接解释自足性原则。但用户接着说“那你把 25 篇都改了”，这就会产生项目产物，必须进入变更管理和证据链。

再看一个综合案例。用户问：“短链过期链接应该返回 404 还是 410？”这本身可以先走 L 旁路：Owner 可以解释两者语义差别，并说明需要看项目契约。如果用户要求“查一下项目之前怎么约定”，wiki 是旁路证据；如果要看当前代码如何处理过期，codegraph 和源码是旁路证据；如果要比较外部 HTTP 语义，M0.5 是旁路调研。最终选择错误码并修改接口契约，就必须升格到 M3/变更卡和 HITL。

反例是工具链自动把 M0.5 调研结论写进接口契约。调研可以说明 410 更符合语义，但不能替项目选择兼容性策略。用户可能因为旧客户端兼容选择 404，也可能接受 410 并安排迁移。选择权不在调研工具手里。

## 维护者注意事项

维护 L 旁路时，要把升格点写清楚。轻量咨询、状态查询、证据收集可以旁路；一旦开始修改文件、改变规则、生成长期文档、改变阶段状态或影响合并，就应进入正式流程。

工具集成时，不要把旁路结果接成硬门禁表达式。可以让工具结果触发提醒、建议、风险标注或待确认项，但不要让工具结果自动通过或拒绝阶段。

如果团队发现旁路被频繁用来偷偷改项目，说明入口判断太弱。应补 Owner 路由规则和变更卡创建流程，而不是继续扩大旁路权限。

## 边界与反模式

- 不要把 L 旁路当成绕流程改项目的后门；否则项目产物会缺少变更卡、评审和验收证据。
- 不要把工具证据接成硬门禁；否则 wiki、codegraph 或调研结果会越过 Owner 和 HITL 直接裁决。
- 不要在轻量咨询中悄悄生成长期项目产物；否则后续维护者找不到状态和来源。
- 不要忽略升格点；否则用户以为仍是咨询，实际已经进入开发或维护工作。
- 不要让旁路结果没有置信说明；否则工具未命中、索引过期或调研不完整会被误读成事实。

## 读完后你应该能做什么

- 能判断请求是否适合 L 旁路。
- 能说明旁路证据为什么不夺权。
- 能识别咨询升格为变更卡的时刻。
- 能把工具证据写成判断输入而不是最终裁决。
