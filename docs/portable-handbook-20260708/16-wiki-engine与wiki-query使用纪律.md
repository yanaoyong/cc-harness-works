# wiki-engine 与 wiki-query 使用纪律

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

wiki-engine 是 Harness 的文档知识库组件，`wiki-query` 是面向 Agent 的单命令查询入口。portable 读者不需要拥有这套组件，也能理解它背后的纪律：什么时候需要文档知识综合、什么时候应该回到全文搜索、什么时候要把知识摘要继续追溯到原始证据。

本文中的 wiki 可替换为任何文档知识库或知识索引能力。重点不是组件名，而是查询、摄取、新鲜度和信任边界。

本篇重点不是教命令，而是解释“文档知识如何被安全消费”。

## 核心模型

| 问题类型 | 首选方式 | 原因 | 边界 |
|---|---|---|---|
| 规则、历史、ADR、失败案例 | wiki-query 或本手册正文 | 需要跨篇综合 | 不能替代事实源 |
| 字符串、错误消息、配置键 | `rg` 或全文搜索 | 需要找全字面出现 | wiki 提炼会有损 |
| 子 Agent 背景注入 | Owner 先摘要再委派 | 降低迷路和漏读 | 不应只给路径 |
| 未命中 | 回退搜索并说明低置信 | 未命中不是不存在 | 不能臆答 |
| 修改规则 | 追溯到权威文件 | 防止二手摘要漂移 | 不能只改手册 |

wiki 的两跳思想是：第一跳给概念入口，第二跳给原始证据。portable 版保留这个思想，但把概念解释写进正文，把来源只作为维护线索。

wiki-query 的价值在于“综合”，不是“找全”。当问题是“这条规则为什么存在”“这个机制和哪些失败案例相关”“某个阶段的边界是什么”，wiki 能把分散文档压缩成可消费背景。它不适合回答“某个字符串出现了几次”“某个函数被谁调用”“当前实现是否正确”。这些问题需要全文搜索、codegraph、源码阅读和测试。

证据不夺权在 wiki 场景里尤其重要。wiki 摘要即使准确，也只是二手知识。它可以告诉 Owner 某条规则的历史背景和来源线索，但不能直接修改规则、关闭 MUST FIX、跳过测试或授权合并。涉及规则、架构、合并和用户承诺时，wiki 的第二跳应该把读者带回权威文件和流程。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 问“为什么有这条规则” | 需要历史背景 | 读本手册相关正文；有原仓时再查来源 | 只看当前规则会误删护栏。 |
| 问“这个词出现在哪” | 需要字面找全 | 用全文搜索 | 用知识摘要会漏改写后的文本。 |
| 委派 Reviewer | 需要上下文注入 | Owner 先整理摘要、范围、文件和风险 | 只说“去看 wiki”会让子 Agent 漂移。 |
| wiki 未命中 | 不是事实否定 | 回退搜索并标注置信边界 | 直接说“不存在”会误导。 |
| 写 portable 版 | 需要正文自足 | 把来源结论内嵌，来源只写列入原因和离线依据说明 | 留来源名会让离线读者读不懂。 |

判断是否该用 wiki，可以先问：我是在寻找“概念背景”还是“原文字面”？如果用户问“为什么 Reviewer 不能自修”，wiki 很适合，因为答案来自角色边界和失败案例；如果用户问“Reviewer 这个词在哪些文件里出现”，就应使用全文搜索。如果用户问“Reviewer 类被哪些模块调用”，那是代码结构问题，应走 codegraph 或源码。

wiki 未命中也要谨慎解释。未命中可能是查询词不对、页面未摄取、来源太新、预算不够，或者该知识确实不存在。正确做法是说明低置信边界，并回退到全文搜索或人工阅读，而不是把未命中写成“项目没有这个规则”。

## 最小流程

1. 判断问题是概念综合、字面搜索、代码影响面还是裁决。
2. 概念综合可用 wiki 或 portable 正文；字面搜索用 `rg`；代码影响面用 codegraph 或源码。
3. 委派前由 Owner 摘要关键背景，不把寻路工作丢给子 Agent。
4. 得到知识摘要后，判断是否需要追溯事实源。
5. 如果结论会影响规则、架构或合并，回到对应流程和 HITL。

## 例子

用户问：“为什么测试通过还不能合并？”这不是代码调用问题，而是质量规则和失败案例问题。正确做法是解释 false-green、测试评审、CI 和 HITL 的证据分工，再把需要的检查落回当前卡。

如果只告诉子 Agent “去 wiki 看测试规范”，它可能找到一堆页面但不知道本次卡需要什么结论。Owner 应先给出问题、范围、预期输出和边界。

具体来说，合格的委派背景可以写成：“当前卡要判断测试通过是否可信；已知风险是 false-green；请只使用 wiki 背景整理测试评审为什么存在、哪些证据不能被 exit 0 替代、哪些问题需要回到测试报告，不要判断本卡是否可以合并。”这样 wiki 提供背景，子 Agent 提供整理，Owner 仍负责裁决。

反例是：“wiki-query 说测试守护规则存在，所以这张卡必须失败。”规则存在不等于当前卡失败。Owner 还要看当前测试命令、测试数量、skip、mock、变更影响面和用户验收。wiki 只能帮助解释为什么要检查这些证据。

## 维护者注意事项

维护 wiki 查询纪律时，要持续检查“输出是否适合 Agent 消费”。如果输出太长、来源不清或未命中不可解释，Agent 会回到猜测或放弃使用。查询工具越容易用，越需要把信任边界写清楚。

摄取新知识时，要区分“摘要页”和“权威源”。摘要页可以帮助发现和理解，权威源决定规则、架构和历史事实。portable 手册也一样：正文要自足，但修改规则时不能只改手册。

如果发现子 Agent 经常被要求“自己去 wiki 找背景”，应改委派模板。Owner 应先把 wiki 结果整理成任务相关摘要，再交给子 Agent；否则知识工具会变成寻路负担。

## 边界与反模式

- 不要把 wiki 提炼页当唯一事实；修改规则时仍要回到权威文件。
- 不要用 wiki 替代全文搜索；提炼会丢失字面召回。
- 不要把 wiki-query 当硬门禁；它是证据工具，不是裁决器。
- 不要让子 Agent 自行寻路；委派需要摘要和目标。
- 不要在 portable 版只留下来源名；正文必须自足。

## 读完后你应该能做什么

- 能判断什么时候需要文档知识综合。
- 能区分 wiki、全文搜索、codegraph 的适用边界。
- 能为子 Agent 准备知识背景摘要。
- 能把 linked 来源转写成 portable 正文。
