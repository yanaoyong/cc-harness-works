# codegraph 使用边界与查询路由

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

codegraph 用来回答代码结构问题：符号在哪里、谁调用谁、改动影响哪些节点、索引是否新鲜。它不是文档知识库，也不是最终裁决器。

本篇给出离线可用的查询路由 runbook，帮助读者判断什么时候该用 codegraph，什么时候该用全文搜索或直接读源码。

## 核心模型

| 问题 | 首选工具 | 为什么 |
|---|---|---|
| 字符串、错误消息、配置键 | `rg` | 需要字面找全 |
| 函数/类/模块调用关系 | codegraph | 需要符号和边 |
| 规则和历史决策 | wiki 或手册正文 | 需要文档综合 |
| 最终行为正确性 | 测试/CI/运行验证 | 工具只能给线索 |

codegraph 的结论是证据线索，最终仍要回源码核对。

codegraph 的强项是结构，不是语义裁决。它能帮助你发现某个函数的调用方、某个模块的依赖、某次改名可能影响的节点；但它不知道用户是否接受范围，不知道测试是否可信，也不知道某个设计为什么在 ADR 中被选择。codegraph 输出越像“影响面”，越容易被误读成“完整事实”，所以必须保留源码核对和置信说明。

在知识层对比里，codegraph 经常和 `rg` 混用。`rg` 找字面，codegraph 找结构。比如你要找所有 `EXPIRED_LINK` 错误码出现处，先用 `rg`；你要看跳转解析函数被哪些入口调用，用 codegraph；你要知道为什么过期链接要有单独错误码，回到 wiki/手册；你要证明行为正确，回到测试。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 同名函数很多 | 需要消歧 | 先 query，确认路径和 node id | 聚合结果会误导影响面。 |
| 索引可能过期 | 需要新鲜度判断 | 先 status/doctor，必要时 sync | 旧索引会漏新调用。 |
| 动态调用较多 | codegraph 可能不全 | 结合源码和测试 | 静态图会低估影响面。 |
| 文档规则问题 | 不适合 codegraph | 回到手册或 wiki | 代码图谱无法回答历史原因。 |
| runtime 缺失 | 工具不可用 | 降级 rg，并说明置信边界 | 臆造调用关系最危险。 |

判断 codegraph 结果是否能使用，要看三件事。第一，索引是否新鲜；过期索引只能作为低置信线索。第二，符号是否消歧；同名函数或类会让影响面聚合失真。第三，源码是否核对；静态图无法完整理解动态注册、反射、配置驱动或运行时拼接。三者缺一时，结论都应降级。

codegraph 未命中也不能直接写成“没有影响”。未命中可能来自索引过期、语言解析不支持、动态调用、符号名变化或查询方式错误。正确结论应写成：“当前 codegraph 查询未发现直接调用方；已补充 `rg` 和源码核对；仍存在动态调用不确定性。”这才是证据，而不是裁决。

## 最小流程

1. 先判断问题是否真的是代码结构问题。
2. 查索引状态，确认是否新鲜。
3. 用 query 消歧符号。
4. 再查 callers、callees、impact 或 affected。
5. 回源码核对关键路径。
6. 把命令、索引状态、源码核对点和不确定性写入证据链。

## 例子

你要改 `create_change`，担心影响命令入口。正确做法是先查符号，确认它指的是哪个文件中的函数；再看 callers，找出命令、hook 或测试入口；最后打开源码核对条件分支和动态调用。不能只因为 codegraph 结果里没有某个调用方，就断言没有影响。

再看短链服务例子。你要修改 `resolveShortCode` 的过期判断。codegraph 可以帮助确认它被 HTTP 跳转入口、测试、审计记录调用；`rg` 可以找出错误码字面；源码核对可以确认过期分支是否在跳转前执行；测试证明过期链接不会跳转。即使 codegraph 显示调用方只有一个 HTTP handler，也不能因此跳过契约测试，因为行为正确性属于测试证据，不属于结构证据。

反例是用 codegraph 回答“为什么这个规则存在”。代码图谱可能找到规则加载路径，但不会告诉你规则来自哪次失败或 ADR。这个问题应回到 wiki 或 portable 正文。工具路由错误会产生看似技术化、实则无关的答案。

## 维护者注意事项

维护 codegraph 文档时，要把“索引状态”放在流程前面。没有新鲜度说明的调用关系报告，会让读者误以为结果是当前源码事实。

如果 codegraph 输出难读或调用成本高，Agent 会绕开它。采纳改进不只是增加功能，也包括让查询路由、失败输出和结果解释更贴近 Owner 的判断流程。

codegraph 结果进入评审报告时，建议固定写四项：查询对象、索引状态、发现的影响面、源码核对点。这样 Reviewer 和 Owner 都能看出证据边界。

## 边界与反模式

- 不要把同名符号聚合结果直接当事实。
- 不要在索引过期时无提示信任结果。
- 不要用 codegraph 替代源码核对。
- 不要用 codegraph 回答文档规则或历史决策。
- 不要因为工具不可用就编造调用关系。

## 读完后你应该能做什么

- 能判断问题应由 `rg`、codegraph、手册还是测试回答。
- 能执行 query -> callers/callees/impact -> 源码核对的流程。
- 能说明索引新鲜度为什么影响可信度。
- 能把 codegraph 结果写成证据而不是裁决。
