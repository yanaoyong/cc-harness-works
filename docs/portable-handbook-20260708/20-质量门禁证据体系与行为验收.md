# 质量门禁、证据体系与行为验收

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

Harness 判断“完成”不能靠 Agent 自述，而要靠可核查证据。本篇解释一张卡需要哪些证据，哪些证据证明什么，哪些证据不能互相替代。

它与 `21-测试守护契约测试与假绿治理.md` 的分工是：本篇看全链路完成证据，`21` 专门检查测试证据是否可信。

## 核心模型

| 证据 | 证明什么 | 不能证明什么 |
|---|---|---|
| spec/tasks | 范围和验收边界清楚 | 实现正确 |
| code review | 结构、契约、风险被审视 | 运行一定通过 |
| unit test | 局部行为符合预期 | 跨层契约一致 |
| test review | 测试不是摆设 | 全量基线绿 |
| CI | 当前环境全量基线成立 | 用户价值可用 |
| deployment | 运行面或等价验证成立 | 所有边界覆盖 |
| acceptance | 行为遵行或真实使用场景成立 | 可替代所有测试 |
| HITL | 用户接受范围、风险或合并 | 技术正确性自动成立 |

质量门禁的原则是：证据可以支持裁决，但裁决仍由 Owner 组织并由用户在 HITL 点确认。

一条完整验收链不是“证据越多越好”，而是每类证据回答不同问题。spec/tasks 证明目标和边界被确认；review 证明结构和风险被审视；测试证明局部行为；测试评审证明测试不是摆设；CI 证明全量基线；部署或运行验证证明用户路径可消费；acceptance 证明行为纪律或真实场景；HITL 证明用户接受范围、风险或合并。缺哪一环，就要说明该风险如何被替代覆盖，否则不能只靠另一环补叙事。

验收链还要能复查。一个合格的证据条目至少包含：执行对象、命令或检查方式、结果、解释、剩余风险和归属阶段。例如“跑了测试”不够；应写“在项目根目录执行 `npm test`，总数 84，失败 0，skip 1，skip 原因是平台限定；本次修改覆盖 parser 和 serializer；未覆盖端到端部署，因本卡不涉及运行时路径”。这种写法让后续 Reviewer 和用户能看懂证据边界。

证据记录可以使用这个最小格式：

- 命令/检查：执行了什么命令、脚本、人工检查或行为验收。
- 范围：覆盖哪些文件、功能、路径或用户场景。
- 结果：通过、失败、skip、数量、输出位置或截图位置。
- 异常：环境缺失、skip 原因、未覆盖范围或替代验证。
- 剩余风险：哪些风险仍需 HITL、后续卡或回填证据处理。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| Reviewer 无 MUST FIX | 只证明评审风险清零 | 继续看测试、CI、部署 | 评审不能替代运行验证。 |
| CI 绿 | 证明当前基线通过 | 检查测试总数和环境前置 | CI 假绿会误导合并。 |
| 用户想合并 | 需要全链路证据 | 汇总范围、评审、测试、CI、部署和风险 | 用户看不到风险会误授权。 |
| 工具命令能跑 | 只证明工具可用 | 做行为验收，看 Agent 是否真的使用 | 工具存在不等于被采纳。 |
| 旧卡缺证据 | 需要回填 | 补 acceptance/backfill 记录 | 历史账不补会复发。 |

组织验收链时，Owner 应先列风险，再配证据，而不是先堆命令。例如一次规则改动的主要风险可能是“Agent 不会采纳新规则”，这时单元测试帮助有限，更重要的是构造真实提示词、观察 Agent 是否触发规则、检查失败时是否能恢复。一次数据迁移的主要风险可能是“旧数据被破坏”，这时 review 和 unit test 不够，还需要迁移前后样本、回滚步骤和幂等验证。

如果某类证据缺失，要明确是阻塞、替代还是风险接受。阻塞表示没有它不能继续，例如 CI 红且原因未知；替代表示用更接近风险的证据覆盖，例如没有真实部署环境时用同构容器和端到端脚本；风险接受表示用户明确知道剩余风险并仍决定继续。三者不能混写成“后续观察”，否则验收链会失去裁决力。

## 最小流程

1. 确认范围已被 HITL-1 冻结。
2. 核对实现是否覆盖 tasks。
3. 检查 Reviewer 结论和 MUST FIX 状态。
4. 检查测试是否真实执行并覆盖关键行为。
5. 检查 CI 或等价全量验证。
6. 检查运行面或部署验证。
7. 汇总剩余风险，交用户 HITL-5。
8. 收尾时同步 summary 和证据链。

## 治理硬门：DF-015 四子门

前面讲的是“要拿什么证据”，这一节讲“证据和授权本身如何防伪造”。长回合或后台会话里，模型自律会衰减，只写在文字里的纪律拦不住“自问自答虚构用户确认”或“未经授权就合并”。为此项目在 `v0.8.1 · 治理硬门 DF-015` 把四条纪律从文字层升级成 hook 层的结构性强制（`v0.8.3 · 授权门提号三态` 再收严其中的授权判定）。四条硬门共用同一地基——用户授权台账：每轮用户输入的原文加 ISO8601 时间戳被 append-only 落盘，只被真实用户输入触发，模型无法伪造。

| 子门 | 作用 | 拦什么 | 放行条件 |
|---|---|---|---|
| T1 授权台账 | 其余三门的地基 | — | 真实用户输入才写入 |
| T2 merge·push 门 | 拦危险 git 操作 | 合并 / 推送无授权 | 台账近期授权词命中，按 gate 独立消费 |
| T3 委派 guard | 强制委派质量 | 委派 generator/reviewer/strategist 缺四要素 | 四要素齐全放行并追加委派台账 |
| T4 翻牌硬门 | 拦“没委派就报完成” | 产出型阶段（1/3/5）翻 PASSED 无委派台账 | 有委派记录才放行 |

授权纪律三要点决定这套门能否真的防住伪造：

- 逐字：授权语必须是用户逐字输入的原话才进台账。用弹窗选项作答（在选项里点选，而非亲手打字）不进台账，因此不能拿它放行合并或推送——授权必须是一句用户亲手打的话。
- 单次消费：单条授权在同一个门只消费一次，推送和合并各自独立追踪；一句话可以同时包含推送、合并、tag 多个门的授权词，各门各自命中、各自记一次，用完即失效，避免“一次授权反复放行”。
- 三态：授权门核对 PR 号时按三态判定——号对了放行，提空或号说不清都向拒绝收敛。早期两态把“没提号”和“号说不清”混为一谈，收严后任何歧义一律拦。所有证据类失败（台账读不到、号对不上）统一向拒绝收敛，宁可误拦合法操作也不放过伪造。

T3 的拒绝不是死拦：缺哪个要素就回一段修复配方，补齐后重试即可。T4 只在产出型阶段（1/3/5）生效，阶段行是 N/A、非 1/3/5、或走 M 元流程时不触发。

## 例子

一篇 portable 文档改写完成后，不能只说“已经写完”。合格证据包括：文件列表、无 wiki 依赖表达的检查、目录内链接检查、来源依据是否写清列入原因和离线使用方式、审查意见是否处理、用户是否接受。这些证据分别证明不同风险面。

再看一个全栈接口例子。需求是把订单详情新增 `estimatedDeliveryDate` 字段。验收链应包含：需求卡确认字段语义和空值策略；后端 review 检查模型、序列化和权限；后端测试覆盖有值、无值和老订单；契约测试固定响应字段和错误格式；前端集成验证真实响应被渲染；CI 证明全量基线；部署或预览环境截图证明用户路径可见；HITL 说明剩余风险，比如历史订单数据需要另开迁移卡。这条链里任一证据都不能替代其他证据。

反例是“CI 绿，所以合并”。如果 CI 没有跑前端集成，或者 mock 仍使用旧字段，那么 CI 绿只能证明现有自动化没发现问题，不能证明真实用户路径已经完成。Owner 需要把“CI 覆盖了什么、没覆盖什么”写出来，再决定是否补验收或向用户声明风险。

## 维护者注意事项

维护质量门禁时，优先检查证据链的“裁决问题”是否清楚。每个门禁都应对应一个问题：范围是否明确、实现是否覆盖、测试是否可信、运行是否可消费、用户是否接受。无法对应问题的证据会变成仪式感，应该删减或转写。

当新增门禁时，也要给出失败时的恢复路径。例如要求部署验证，就要说明没有部署权限时能否用预览环境、容器演练或人工截图替代；要求行为验收，就要说明观察对象、输入提示、预期行为和记录方式。门禁只规定“必须有”，不规定“缺失如何处理”，会在紧急情况下被绕开。

## 边界与反模式

- 不要用“看起来没问题”替代门禁证据。
- 不要让一种证据替代全部证据；评审、测试、CI、部署、HITL 分工不同。
- 不要把工具能跑当成 Agent 会用；行为验收要看真实场景。
- 不要让 acceptance 成为补叙事的文案；它必须能回到证据链。
- 不要在 CI 红时只写归因说明，除非它确实是环境基线并已记录。

## 读完后你应该能做什么

- 能判断一张卡缺哪类完成证据。
- 能解释为什么 CI 绿不等于可合并。
- 能区分质量门禁和测试守护。
- 能设计一个行为遵行验收。
