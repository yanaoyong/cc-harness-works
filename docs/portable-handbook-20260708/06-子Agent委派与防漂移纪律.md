# 子 Agent 委派与防漂移纪律

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

子 Agent 能提高并行探索和独立评审质量，但也会带来漂移：读错上下文、越过角色边界、自己发明目标、或者把不完整证据当结论。本篇给出离线可用的委派纪律。

委派不是“把任务丢出去”。合格委派必须包含目标、上下文、边界、输出格式、禁止事项和回收方式。

## 核心模型

一次委派应具备六个要素：

| 要素 | 内容 | 缺失后果 |
|---|---|---|
| 目标 | 子 Agent 要回答什么或产出什么 | 输出发散 |
| 上下文 | 当前卡、阶段、相关文件、已知事实 | 误读任务 |
| 边界 | 哪些文件/行为不能碰，哪些结论不能下 | 越权修改 |
| 角色 | Reviewer、Generator、Explorer、Strategist 等 | 评审者变执行者 |
| 输出 | 需要报告、patch、风险列表还是判断 | 难以整合 |
| 回收 | Owner 如何核对和使用结果 | 子 Agent 变裁决者 |

委派漂移通常不是子 Agent “能力不够”，而是输入包太松。子 Agent 会根据它看到的上下文自我补全目标；如果 Owner 没有说明当前阶段、文件范围、禁止事项和输出格式，它就可能把审查任务做成修复任务，把调查任务做成方案决策，把局部问题扩成全仓重构。委派越开放，漂移空间越大。

委派还需要区分“探索”“执行”“评审”“策略”四种模式。探索型委派回答具体问题；执行型委派改明确范围；评审型委派只找问题和风险；策略型委派整理方案和取舍。模式不同，允许的文件权限、输出格式和停止条件都不同。最危险的委派是同时要求“看看、改改、判断能不能合并”，因为它混合了所有角色。

## 操作判断

| 场景 | 是否适合委派 | 推荐动作 | 风险 |
|---|---|---|---|
| 需要独立评审 | 适合 | 明确只评审、不修改，要求 MUST FIX 分级 | Reviewer 自循环修复会破坏独立性。 |
| 需要并行查背景 | 适合 | 给具体问题和资料边界 | 泛泛“去看看”会浪费上下文。 |
| 需要改同一文件 | 谨慎 | 只给一个执行者，其他人读不写 | 并行写同文件会冲突。 |
| 任务目标仍不清 | 不适合执行委派 | 先由 Owner 或 Strategist 澄清 | 子 Agent 会自造目标。 |
| 结果影响合并 | 可委派调查，不可委派裁决 | Owner 汇总后交 HITL | 子 Agent 不能替用户拍板。 |

判断是否委派时，可以先问：我希望子 Agent 补的是信息、实现、评审还是策略？如果只是为了省时间，但任务边界尚未明确，委派会放大混乱；如果问题已经可切分，委派能提高独立性和并行度。好的委派通常很窄：查一个影响面、审一个目录、改一组文件、评一个测试策略。坏的委派通常很宽：帮我看看、优化一下、整体评估。

回收结果时也要防漂移。Owner 不应只读结论，还要看证据路径、未决问题、被排除的范围和是否违反角色边界。一个看似正确的输出，如果修改了不允许修改的文件，或者把“可能风险”写成“可以合并”，仍然不能直接进入主流程。

## 最小流程

1. Owner 先判断为什么需要子 Agent。
2. 写清任务目标和停止条件。
3. 注入必要上下文，不让子 Agent 自己在仓库里漫游。
4. 指定角色边界和是否允许修改文件。
5. 要求输出结构化结果，例如 findings、risks、changed files、open questions。
6. Owner 回收结果，核对证据，决定是否进入下一阶段。

## 例子

Owner 要审查 portable 手册是否仍依赖 wiki。合格委派应写：“只审查 `docs/portable-handbook-20260708`，不要修改文件；检查是否有仓内 wiki 路径、外部跳读要求或把理解责任交给来源名的表达；输出 P0/P1/P2 问题和文件行号。”

糟糕委派是：“帮我看看 portable 版。”这样子 Agent 可能泛泛评价、修改文件、或遗漏真正的验收标准。

再看一个代码场景。Owner 要确认一个函数改名会影响哪些调用方。合格委派是：“只做影响面调查，不修改文件；查询 `foo()` 的定义、直接调用方、测试覆盖和可能的 public API 入口；输出文件列表、风险等级和需要 Owner 再核对的源码位置。”这样子 Agent 不会顺手重构，也不会替 Owner 判断是否合并。

反例是给两个执行型子 Agent 同时写同一批文件。即使两者都能力很强，也会产生覆盖、冲突和证据归属不清。一个子 Agent 改了测试，另一个改了实现，最后 CI 失败时很难判断失败来自谁的假设。并行委派的前提是写范围和责任边界可分离。

## 维护者注意事项

维护委派模板时，优先检查是否包含六要素：目标、上下文、边界、角色、输出、回收。任何模板如果只有目标，没有停止条件和禁止事项，都容易在复杂任务中漂移。

如果出现过子 Agent 自循环修复、跨卡上下文混用或输出不可整合，应把失败案例转成模板约束。例如 Reviewer 模板明确“只审查不修改”；Explorer 模板明确“只回答指定问题”；Worker 模板明确“只写指定文件范围”。

委派模板不应要求子 Agent 自己去寻找所有背景。Owner 可以让子 Agent 做搜索，但必须给出问题、范围和判断标准。否则“探索”会变成“迷路”。

## 边界与反模式

- 不要只说“去看 wiki/代码”；否则子 Agent 会自行选择错误路径。
- 不要让 Reviewer 同时修复自己发现的问题；否则失去独立评审。
- 不要把子 Agent 的结论原样当最终结论；Owner 必须整合和判断。
- 不要给多个子 Agent 同一写范围；否则冲突和覆盖风险上升。
- 不要把用户裁决交给子 Agent；HITL 必须由 Owner 组织。

## 读完后你应该能做什么

- 能写出包含目标、上下文、边界、角色、输出的委派 prompt。
- 能判断什么时候该委派，什么时候应由 Owner 本地处理。
- 能识别 Reviewer 自循环、跨卡漂移和上下文不足。
- 能把子 Agent 输出整合回主流程。

## 离线来源依据

### 角色与评审

- `agent-roles`
  - 列入原因：用于补足正文背后的背景信息、维护线索和判断依据。
  - 具体补足：角色分离和不同 Agent 职责边界，并说明 Owner、Generator、Reviewer、Strategist 等角色的权限边界，说明执行、评判和治理为什么不能混在一起。
  - 离线吸收位置：委派输入包、独立评审约束和防漂移反模式。
  - 离线使用方式：用来区分执行、评审、策略和治理角色，防止把裁决外包给执行者。
- `reviewer-agent`
  - 列入原因：用于补足正文背后的背景信息、维护线索和判断依据。
  - 具体补足：评审角色的独立性、MUST FIX 和 ONE pass 纪律，并说明 Reviewer 的独立评判、MUST FIX 分类、ONE pass 限制和不能自我修复导致角色漂移的约束。
  - 离线吸收位置：委派输入包、独立评审约束和防漂移反模式。
  - 离线使用方式：用来检查 Reviewer 是否保持独立评判、MUST FIX 分类和 ONE pass 约束。
- `strategist-agent`
  - 列入原因：用于补足正文背后的背景信息、维护线索和判断依据。
  - 具体补足：策略分析型委派的适用场景，并说明复杂任务前如何整理方案、风险、依赖和取舍，让 Owner 获得决策材料而不是直接执行。
  - 离线吸收位置：委派输入包、独立评审约束和防漂移反模式。
  - 离线使用方式：用来判断复杂任务是否需要先整理方案、风险和取舍，再进入执行。

### 委派注入

- `wiki-query-ingest-discipline`
  - 列入原因：用于补足项目知识库的查询、摄取、信任边界和维护纪律。
  - 具体补足：委派前应注入背景摘要，而不是只给路径的纪律，并说明查询何时适用、摄取如何保持新鲜度、矛盾结论如何回到权威源、以及子 Agent 背景注入为什么不能只给路径。
  - 离线吸收位置：委派输入包、独立评审约束和防漂移反模式。
  - 离线使用方式：用来判断知识库查询、摄取、冲突裁决和子 Agent 背景注入是否保持证据工具边界。
- `codegraph-query-routing-precision`
  - 列入原因：用于补足代码结构查询、影响面分析、索引可信度和源码核对边界。
  - 具体补足：代码结构类委派前应先路由和消歧的要求，并说明查询前如何判断问题类型、先消歧符号、再看调用或依赖影响面，并最终回到源码核对。
  - 离线吸收位置：委派输入包、独立评审约束和防漂移反模式。
  - 离线使用方式：用来决定代码结构问题如何先路由、消歧、看影响面，并最终回到源码核对。
- `acceptance-WB-SA1-delegation-injection`
  - 列入原因：用于说明正文判断如何被验收、复查或转化成可观察行为证据。
  - 具体补足：“委派注入是否真的改善行为”的验收视角，并说明如何验收委派背景注入是否真的减少漂移，包括观察子 Agent 输出是否围绕范围、风险和证据。
  - 离线吸收位置：委派输入包、独立评审约束和防漂移反模式。
  - 离线使用方式：用来验收委派背景注入是否真的降低子 Agent 漂移，而不是只增加说明文字。

  - 离线吸收位置：委派输入包、独立评审约束和防漂移反模式。
  - 离线使用方式：用来识别 Reviewer 自循环修复会如何破坏独立评审边界。
- `failure-record-004-owner-crosscard-parallel-drift`
  - 列入原因：用于保留真实失败教训，把抽象规则落到具体风险和反模式上。
  - 具体补足：跨卡并行上下文漂移的失败案例，并说明跨卡并行时上下文混用、状态串线和证据归属错误的具体风险。
  - 离线吸收位置：委派输入包、独立评审约束和防漂移反模式。
