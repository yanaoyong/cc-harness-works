# 测试守护、契约测试与假绿治理

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

测试命令 exit 0 不等于测试有效。测试可能没跑、跑错路径、skip 过多、mock 过度、环境缺失被吞，或者错误实现也能通过。本篇专门处理测试证据可信度。

## 核心模型

| 假绿类型 | 现象 | 应对 |
|---|---|---|
| total=0 | 命令成功但没跑测试 | 检查测试总数 |
| 路径错 | 跑了无关测试目录 | 核对目标和发现路径 |
| skip 过多 | 关键测试被跳过 | 查 skip 原因 |
| mock 过度 | 只验证 mock 自己 | 加契约或集成验证 |
| 环境缺失被吞 | 缺工具却绿 | 环境前置显式化 |
| mutant 杀不死 | 错误实现也通过 | 补断言或测试路径 |
| 权限/快照遗漏 | 本地变化没进版本控制 | 检查 diff 和权限位 |

假绿排查要按顺序做，不能从“测试绿”直接跳到“逻辑正确”。第一步确认命令确实在目标仓库、目标包、目标路径执行；第二步检查 total、passed、failed、skipped，而不是只看 exit code；第三步核对测试发现规则是否包含新文件；第四步检查断言是否验证业务语义；第五步临时制造一个应失败的错误实现，确认测试会红；第六步恢复代码并确认 diff 干净。这个顺序能把“没跑”“跑错”“跑浅”“错了也绿”分开。

契约测试专门处理单侧测试无法证明的问题。后端单测可以证明序列化函数在 mock 输入下工作，前端单测可以证明组件在 mock 响应下渲染，但它们都不能证明真实响应字段、错误结构、枚举语义和空值策略一致。只要修改跨进程、跨包、前后端或插件接口，就要考虑契约测试、schema 固定、fixture 回放或真实集成路径。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 新增核心逻辑 | 需要证明测试能杀错 | 做 kill-mutant | 测试只覆盖 happy path。 |
| 修改 API 字段 | 跨层契约可能破 | 加契约测试或 FE integration | 前后端 mock 不一致。 |
| CI 绿但 total 异常 | 假绿信号 | 修测试发现路径 | 完全没跑测试也会合并。 |
| 本地缺工具 | 不是通过 | 记录阻塞或安装依赖 | 把缺工具当绿。 |
| 快照更新 | 可能掩盖回归 | 核对 diff 和语义 | 错误结果被刷新成期望。 |

排查 `total=0` 时，先看测试框架输出中是否明确发现测试文件，再看配置里的 include/exclude、工作目录和包管理器过滤条件。常见错误是只跑了根目录脚本，但实际测试在子包；或者新增文件名不符合发现模式。修复后要用一个会失败的临时断言确认新文件确实被执行。

排查 mock 过度时，重点看断言是否只验证“调用了 mock 函数”，而没有验证真实输入输出。一个有价值的测试应至少固定关键字段、错误分支和边界语义。例如价格计算不能只断言 `calculatePrice` 被调用，而要断言折扣、税费、舍入和空值策略的结果。涉及外部接口时，mock 要从真实契约样本生成，而不是随手写一个方便通过的对象。

## 最小流程

1. 记录测试命令、工作目录和 exit code。
2. 检查 total/passed/failed/skipped。
3. 确认测试目标覆盖本次变更路径。
4. 检查断言是否验证业务语义，不只是 mock 调用。
5. 做 kill-mutant：错误实现应红，正确实现应绿。
6. 还原 mutant，确认 diff 干净。
7. 涉及跨层契约时，加契约或集成验证。

## 例子

改了价格接口字段名，后端单测通过，但前端仍按旧字段读取。单测只能证明后端局部行为，不能证明契约一致。正确做法是补契约测试或 FE integration，确认真实消费方和后端响应一致。

再看短链过期链接。后端把 `expiresAt` 从秒级时间戳改成 ISO 字符串，后端单测和前端 mock 都能绿，因为前端 mock 仍写旧格式。假绿排查应先查看 API fixture 或网络响应，再用契约测试固定字段类型和时区语义，最后用前端集成测试确认过期提示仍按真实响应工作。如果没有契约测试，这类问题通常会在预览环境或线上才暴露。

kill-mutant 可以这样使用：把价格计算中的 `>` 临时改成 `>=`，或把过期判断中的当前时间偏移删除。若测试仍然通过，说明现有断言没有覆盖边界；需要补边界用例。验证完成后必须还原 mutant，并用 `git diff` 或等价方式确认没有把错误实现留下。这个动作不是为了制造复杂流程，而是为了证明测试真的能抓住关键错误。

## 维护者注意事项

维护测试守护时，要把“测试可信度”写进验收，而不是只在失败后口头提醒。每个高风险变更至少应说明测试发现范围、关键断言、skip 原因、契约覆盖和是否做过 kill-mutant。低风险文档或配置变更可以简化，但也要说明为什么无需完整测试链。

当测试体系长期变慢时，不要用跳过关键测试解决。应先拆分快速单元、契约、集成、端到端几类套件，再为每类定义触发条件。否则短期速度会提升，长期会产生大量假绿，最后把风险推给人工验收。

## 边界与反模式

- 不要只看 exit 0，不看 total 和 skip；否则完全没跑测试或关键测试被跳过也会被当成通过。
- 不要用 mock 掩盖真实契约；否则后端和前端都绿，真实联调仍然失败。
- 不要把本地环境缺失解释成通过；否则 CI 或其他机器上会暴露同一缺口。
- 不要让 kill-mutant 后的错误实现留在代码里；否则验证动作本身会制造回归。
- 不要用快照刷新替代语义判断；否则错误输出会被保存成新的期望结果。

## 读完后你应该能做什么

- 能识别常见 false-green 信号。
- 能设计 kill-mutant 验证。
- 能判断何时需要契约或集成测试。
- 能把测试证据写成可信报告。

  - 离线吸收位置：false-green 类型、kill-mutant、契约测试和联调证据。
  - 离线使用方式：用来设计能验证业务语义、覆盖关键分支且不只测试 mock 的单元测试。
- `skill-unit-test-ci`
  - 列入原因：用于把阶段性 SOP 转写成离线读者也能执行的步骤、输入输出和角色责任。
  - 具体补足：CI 验证和环境前置检查方法，并说明 CI 验证如何确认命令、环境、测试数量、失败输出和本地/远端差异。
  - 离线吸收位置：false-green 类型、kill-mutant、契约测试和联调证据。
  - 离线使用方式：用来核对 CI 命令、环境、测试数量和失败输出是否可信。
