# 最小工作流：从一次请求到一张变更卡

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

本篇回答一个非常实际的问题：用户给 Agent 一句话以后，Harness 怎样决定是否建卡、怎样把自然语言变成可执行范围、怎样留下证据，最后怎样安全收尾。

这里的“变更卡”不是待办列表，而是一项工作从提出、确认、实现、评审、测试、CI、部署到合并的审计容器。没有变更卡，工作很容易只存在于聊天上下文里；上下文一断，状态、范围和证据都会丢。

读完本篇后，你应该能用最小路径解释一张普通增量卡怎么跑，也能判断一个请求是否太轻，不需要进入完整流程。

## 核心模型

最小工作流有一个窄腰：无论任务多复杂，都必须回到“路由、建卡、确认范围、实现、独立评审、验证、用户授权、状态收尾”这几个稳定控制点。

| 控制点 | 解决的问题 | 主要责任人 | 典型产物 |
|---|---|---|---|
| 路由 | 判断是否需要流程 | Owner | 路由结论 |
| 建卡 | 让状态和证据有容器 | Owner | `summary.md` |
| 需求分析 | 把自然语言转成范围和任务 | Owner / 分析 Skill | `spec.md`、`tasks.md` |
| HITL-1 | 让用户确认做什么和不做什么 | 用户 + Owner | 用户确认记录 |
| 实现 | 按任务修改代码或文档 | Generator / Owner | diff、coding report |
| 评审 | 找结构、契约和风险问题 | Reviewer | review report |
| 测试/CI/部署 | 证明行为、基线和运行面成立 | Owner / Skills | test、CI、deployment report |
| HITL-5 与收尾 | 让用户授权合并并沉淀状态 | 用户 + Owner | merge、summary、知识维护 |

关键点是：Owner 不等于写代码的人。Owner 的核心职责是保持流程和证据链完整，让用户能理解当前状态并做决策。

最小工作流存在的原因，是把一次自然语言请求从“聊天内容”变成“工程对象”。聊天内容容易丢失上下文，也很难证明某个范围、某个测试或某个合并授权是否真的发生过。变更卡把这些信息固定下来：用户原话、范围确认、实现差异、评审发现、测试结果、CI 状态、运行验证和最终授权都能被追溯。

这条工作流还有一个重要边界：它不是把所有任务都变复杂，而是给会产生项目产物的任务提供最低审计结构。真正的最小化不是删掉评审或测试，而是在不牺牲证据链的前提下缩小范围、减少不必要阶段内容、明确哪些验证可以等价替代。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 用户提出新增功能 | 会产生项目产物，应进入变更卡 | 建卡，先写 spec/tasks，再让用户确认范围 | 直接编码会让需求边界不可追踪。 |
| 用户问一个解释性问题 | 不产生项目产物 | 走轻量回答或 L 旁路 | 强行建卡会把简单协作变重。 |
| 用户要求紧急修 bug | 仍会修改项目产物 | 建卡，但可缩小范围和证据链，只覆盖本次修复影响面 | 不建卡会让修复原因、测试证据和回归风险丢失。 |
| 用户中途追加需求 | 判断是澄清、范围变更还是新任务 | 小澄清写入当前卡；新范围另建卡或触发上游修订 | 在同一张卡无限扩范围会破坏评审和验收。 |
| 会话中断后恢复 | 以落盘状态为准 | 先看 `summary.md`，再看阶段产物和报告 | 只靠记忆恢复会误判当前阶段。 |

实际判断时，可以把请求拆成三个问题。第一，是否会修改项目产物；如果不会，通常先走轻量回答。第二，修改是否有明确范围；如果没有，先做需求澄清或进入 M0-M5。第三，完成后是否需要用户授权、评审或测试证据；如果需要，就必须有变更卡承载这些证据。这个判断可以很快，但不能省略。

对紧急修复尤其要谨慎。紧急并不等于无流程。可以缩短文档说明，可以聚焦最小影响面，可以只运行相关验证加必要 CI，但仍要记录修复原因、改动范围、验证命令和剩余风险。否则一次紧急修复会变成未来排查时的黑洞。

## 最小流程

1. **接收请求**：记录用户原话，避免后续把解释当成原始需求。
2. **做路由判断**：轻量咨询走 L 旁路；产生项目产物的增量工作走 10 阶段变更卡；项目级方向不清时走 M0-M5。
3. **建立变更卡**：创建或定位工作容器，让 `summary.md` 记录当前阶段、状态和下一步。
4. **写 spec/tasks**：把需求拆成可确认范围、任务、验收点和不做项。
5. **HITL-1 确认**：用户确认范围后再实现。没有确认，后续评审和测试都可能验证错目标。
6. **实现与记录**：按任务修改代码或文档，记录关键设计选择和未完成项。
7. **独立评审**：Reviewer 关注结构、契约、风险和遗漏，不替代测试。
8. **测试与 CI**：证明局部行为和全量基线成立，防止 total=0、skip 过多或 mock 过度等假绿。
9. **部署或运行验证**：证明用户价值在运行面或等价环境中可消费。
10. **HITL-5 与收尾**：用户授权合并，Owner 更新状态、归档证据，并按需要维护知识层。

## 例子

用户说：“给 URL 短链服务增加批量价格查询接口。”

正确路线不是立刻写 endpoint。Owner 先判断它会产生项目产物，因此需要变更卡。阶段 1 先写清接口路径、请求体、响应体、错误码、金额单位、权限边界、测试要求和不做范围。用户确认后，才进入实现。

实现完成后，Reviewer 检查契约和风险；测试证明服务层和路由层行为；CI 证明全量基线；部署验证证明调用方能消费。最后用户授权合并。整个过程留下的不是“我做完了”一句话，而是一条可复盘证据链。

容易犯的错误是把“批量价格查询”顺手扩成“价格缓存重构”或“前端页面改造”。这类新增范围如果没有被 HITL-1 确认，会让评审和测试验证一个用户未批准的目标。

再看一个中断恢复例子。某张卡已经完成实现和代码评审，但测试评审发现 mock 过度，需要补契约测试。会话中断后，新的执行者不能从“代码已经写完”继续，而应从 `summary.md` 和测试评审报告恢复状态：当前阶段应回到测试修复或阶段 7 整合，补完契约测试后再重新跑相关验证。这样做能避免重复实现，也能避免跳过已知风险。

反例是把变更卡当成聊天摘要。只写“用户要批量接口，代码已改，测试通过”，没有范围、不做项、评审发现、测试命令和用户授权。这样的卡无法支持恢复，也无法支持审查。真正的变更卡不一定很长，但必须能回答：当前阶段是什么，下一步是什么，哪些证据已经存在，哪些风险还没闭合。

## 维护者注意事项

维护最小工作流时，优先保护路由、HITL、独立评审、测试可信度和收尾状态这几个控制点。其他实现细节可以随工具变化，但这些控制点一旦缺失，变更卡就会退化成待办列表。

如果新增一种报告或阶段产物，应说明它进入哪一个控制点，证明什么，失败时回到哪里。不要只新增文件名；读者需要知道它在证据链中的作用。

如果未来压缩流程以适应小任务，应使用“轻量卡”或“范围缩小”的方式，而不是默默删掉关键证据。轻量化的目标是降低操作成本，不是降低可审计性。

## 边界与反模式

- 不要在 HITL-1 前扩大需求范围；否则后续证据会验证一个用户未确认的目标。
- 不要把 `summary.md` 当成可选文档；否则会话恢复时无法判断当前阶段。
- 不要让 Reviewer 自己修代码；否则评审角色和执行角色会混在一起。
- 不要用“测试通过”替代代码评审；测试证明行为，评审证明结构和风险被审视。
- 不要把 L 旁路咨询悄悄变成开发工作；一旦产生项目产物，就应升格为变更卡。

## 读完后你应该能做什么

- 能判断一个用户请求是否需要建变更卡。
- 能说出一张卡从请求到收尾的最小证据链。
- 能解释 `summary.md` 为什么是状态恢复的核心。
- 能识别范围膨胀、角色混淆、假绿和无授权合并等风险。
