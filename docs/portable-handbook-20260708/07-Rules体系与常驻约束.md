# Rules 体系与常驻约束

> Portable 版说明：本篇可脱离仓内 wiki 单独阅读。正文已经内嵌关键结论；末尾“离线来源依据”只记录这些结论原先来自哪些规范、决策或验收材料。

## 本篇解决什么问题

Rules 是 Harness 的长期约束层。它们回答“Agent 做事时哪些边界不能越过”，例如流程、目录、编码、架构、测试和知识层纪律。

如果没有 Rules，每次会话都要重新解释工程规范；如果 Rules 写得太多太散，又会挤占上下文并制造第二事实源。本篇说明如何理解、使用和维护 Rules。

## 核心模型

Rules 分三层：

| 层 | 作用 | 例子 |
|---|---|---|
| 常驻规则 | 每次会话都必须记住的身份、流程和安全边界 | Owner 责任、HITL、不得破坏用户改动 |
| 阶段规则 | 某阶段或某类任务才加载的具体约束 | 测试守护、API 设计、代码评审 |
| 项目定制规则 | M3/M4 为具体项目生成的覆盖规则 | 项目架构、目录、技术栈约束 |

好的 Rules 应该来自反复出现的失败模式，而不是某次临时偏好。

Rules 的价值在于稳定默认行为，但 Rules 也是最容易被误用的层。把太多内容写成常驻规则，会让上下文变重，Agent 反而抓不住重点；把某个项目的技术选择写进 core，会污染其他项目；把一次变更卡里的临时判断升级为规则，会让未来任务继承错误前提。Rules 应该是长期护栏，不是经验垃圾桶。

判断一条规则是否成立，要看它是否可判定、可执行、可维护。可判定意味着读者能判断是否违反；可执行意味着 Agent 能据此改变行为；可维护意味着规则有权威位置和更新路径。像“写好代码”不是好规则；“测试命令 exit 0 但 total=0 不能视为通过”才是好规则，因为它可检查、可执行、可复查。

### 常驻层怎样进入上下文

常驻规则不只是"写下来"，还要真的被装进每次会话的上下文，而"怎样装载"本身是一次成本治理改造的核心。早期做法是让会话入口的引导文件用一种整文件内联指令（import 式指令），把整个参考层（编排中枢定义 + 工程结构、开发流程、编码规范三份规则）常驻进上下文；结果是主 Agent 和被委派的子 Agent 都被迫全量背上这套参考层，子 Agent 的固定底座因此高达 58–69k token。

`v0.8.0 · 常驻契约注入 ADR-018` 改造了这条装载路径（它细化了更早的注入决策 ADR-017，并部分推翻了"import 式指令为主装载路径"的旧结论），把常驻内容按性质拆成两类分别装载：

- **决策时刻契约**（首个动作契约、各 HITL 停点、子 Agent 委派要素、隔离工作树禁用、跨卡并行禁令、Git 授权边界等）恒留在最小引导文件里，随每次上下文重建完整重载。它们体积小、必到场，即使注入机制失效也不缺席。
- **参考层**（编排中枢定义 + 三份规则）改由会话启动钩子在会话起手时把权威源全文拼接后注入上下文，带一个固定标头，覆盖冷启动、恢复、清屏、上下文压缩四种启动场景，并且**只面向主会话**——子 Agent 不再自动携带这三份规则。
- 如果注入块因钩子失效或上下文压缩而丢失，每轮用户输入时会检测一个会话哨兵，缺失就兜底补注入一次（机械自愈）；钩子被彻底禁用时，还有一个手动注入命令作为降级路径。

这次改造把子 Agent 的固定底座从 58–69k 降到 22–35k 量级（中位约下降 **49%**），而且常驻内容比早期更完整——早期那种"底座小"是靠引导文件不完整换来的，这次用机制把它正规化了。

它对 Rules 供给有一个必须记住的连带影响：因为子 Agent 不再自动携带三份规则，委派编码或评审子 Agent 时，必须把本任务用到的硬约束（金额单位、超时降级、禁写密钥等）直接摘录进委派指令。这份摘录从原来的"双保险"升级成规则供给的**唯一路径**——漏摘录，子 Agent 就没有编码规范可依。

## 操作判断

| 场景 | 应该怎么判断 | 推荐动作 | 风险 |
|---|---|---|---|
| 某条约束每次都要提醒 | 可能适合常驻规则 | 写成短而强的默认约束 | 常驻过多会挤占上下文。 |
| 某约束只在测试阶段有用 | 阶段规则更合适 | 放到测试 Skill 或测试守护规则 | 常驻会造成噪音。 |
| 项目有特殊架构边界 | 项目定制规则 | 从 M3/M4 产物生成覆盖规则 | 写进 core 会污染其他项目。 |
| 一次卡内临时做法 | 不应升级通用规则 | 写在变更卡上下文里 | 临时经验变规则会长期误导。 |
| 规则和实现冲突 | 先找权威源和决策背景 | 更新权威规则，再更新手册 | 多处复制会导致口径漂移。 |

Rules 误用通常有四种。第一，规则膨胀：把所有建议都常驻，导致关键约束被淹没。第二，规则夺权：用规则替代用户确认，例如“规则允许就自动合并”。第三，规则错层：把项目定制写进通用 core。第四，规则漂移：手册、运行时规则和 Skill 各写一份，修改时只改其中一处。遇到这些问题，应先判断权威源，再决定是删、移、合并还是升级。

如果一条规则来自失败案例，应保留失败模式，而不只是保留结论。比如“不要让 Reviewer 自修”背后是独立评审失效；“不要把安装态当开发态”背后是修改会被覆盖；“不要把 total=0 当测试通过”背后是假绿。保留失败原因，读者才知道规则不是风格偏好。

## 最小使用步骤

1. 判断约束属于常驻、阶段还是项目定制。
2. 检查它是否来自真实失败、长期边界或架构决策。
3. 用简短可执行语言写规则，避免散文式口号。
4. 明确规则消费者：Owner、Generator、Reviewer、测试、CI 还是模板。
5. 更新相关 Skill、模板或运行时入口，避免规则写了但没人加载。
6. 用评审、测试或验收证明规则能被遵守。

## 例子

项目曾出现路径语境混乱：有人把权威源层、消费方镜像层、运行时层三态混为一谈。这个问题不是一次性解释能解决的，因此应沉淀为规则和手册原则：所有路径都必须带语境。

另一个例子是假绿测试。命令 exit 0 但 total=0 或断言没有杀死 mutant，这类失败会反复出现，因此应进入测试守护规则，而不是只留在某张卡的总结里。

反例是把“这次为了赶时间只跑局部测试”写成通用规则。局部测试可以作为某张卡的临时风险接受，但不能升级为长期默认。正确做法是在变更卡里记录本次为什么只跑局部、剩余风险是什么、是否需要后续补 CI，而不是让未来所有任务都继承这个例外。

再看项目定制规则。某个 React/Vite 项目规定组件目录、状态管理和 API client 位置，这些规则对该项目很有价值，但不应写入 Harness core。它们应来自 M3/M4 的项目产物，进入 profile 或项目定制层。这样换到 Python CLI 项目时，不会莫名继承前端目录约束。

## 维护者注意事项

维护 Rules 时，应定期做“规则瘦身”。检查哪些规则已经被 Skill 覆盖，哪些规则只是一次性建议，哪些规则写得太抽象。常驻规则越少越有力，阶段规则越贴近场景越有效。

修改规则前，要先找权威源和相关决策。不要只改 portable 手册；手册是解释层，真正影响 Agent 行为的可能是运行时规则、插件资产或项目定制文件。修改后还要检查对应 Skill、模板和验收是否需要同步。

新增规则时，建议附带一个违反场景和一个验收方式。没有违反场景，规则容易变口号；没有验收方式，规则难以证明被遵守。

## 边界与反模式

- 不要把某次卡的临时做法升级成通用规则；否则后续项目会继承错误约束。
- 不要在多个地方复制同一条规则全文；否则修改时会出现口径漂移。
- 不要让 Rules 替代 HITL；规则能约束行为，不能替用户拍板。
- 不要把具体技术栈写进 core 通用规则；应放到 profile 或项目定制层。
- 不要写无法执行的口号；规则必须能指导判断或验收。

## 读完后你应该能做什么

- 能区分常驻规则、阶段规则和项目定制规则。
- 能判断一条新规则是否应该进入 core。
- 能把失败案例转成可执行约束。
- 能识别规则复制、规则膨胀和规则夺权风险。

### 常驻装载机制

- `v0.8.0 · 常驻契约注入 ADR-018`
  - 列入原因：用于补足常驻规则"怎样被装进上下文"的机制取舍，以及这次改造带来的成本与规则供给连带影响，属于常驻约束层的运行边界与历史决策。
  - 具体补足：常驻内容按性质二分装载的机制——决策时刻契约恒留最小引导文件、参考层改由会话启动钩子全文注入且只面向主会话；覆盖冷启动/恢复/清屏/压缩四种启动场景；注入丢失时的哨兵检测兜底自愈与手动注入降级路径；子 Agent 固定底座中位约下降 49%（从 58–69k 降到 22–35k 量级）；以及子 Agent 不再自动携带三份规则后，委派摘录升为规则供给唯一路径这一硬约束。
  - 离线吸收位置：核心模型下的“常驻层怎样进入上下文”小节，以及边界与反模式中关于委派摘录必须带硬约束的判断。
  - 离线使用方式：读者可据此判断常驻规则的物理装载是否正常（主会话有注入块、子 Agent 只靠委派摘录），并在委派编码/评审子 Agent 时确认硬约束是否已摘录进委派指令。
