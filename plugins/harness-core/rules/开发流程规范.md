# 规则 · 开发流程规范

> 稳定不变约束。告诉 Agent「流程层面的标准是什么」。阶段内部 SOP 见 `../skills/`，阶段细则见 `docs/stage-01-Harness体系建设/02-体系设计/07-十阶段流程详细规范.md`。

## 1. 适用范围

适用于需求从接收到交付的全过程流程纪律；不替代各阶段 Skill 的具体步骤。

| DF-012 | 报告类产物预算：评审/编码/单测**报告类产物**（`*_review_vN` / `coding_report_vN` / `unit_test_report_vN`）须满足——(a) **内容硬约束**：只含「结论 + 证据表 + MUST FIX 清单」，禁止复述 spec、禁止大段贴代码、禁止过程流水账；(b) **软目标**：~100 行自检线，超出须自检多余部分是证据/MUST FIX 而非叙述；(c) **铁律**：证据完整性 > 简洁，任何 MUST FIX 或必需证据不得为压缩篇幅而省略；(d) **不做硬行数截断**（100 行为气味阈值非铡刀）。**豁免**：定义性产物 `spec.md` / `tasks.md` 不设行数限制 | MUST 人工〔(a)(c) 强制〕+ SHOULD〔(b) 软目标〕 | 人工评审（阶段4/6 对照报告内容结构） |
| DF-013 | **阶段10 分支内翻牌（merge 前）**：阶段10 在 **HITL-5 用户授权合并之后、执行 `gh pr merge` 之前**，在变更分支上追加一个翻牌 commit，把本卡 `summary.md` 终态翻牌——`| 总体状态 |` → `PASSED`、`| 当前阶段 |` → `10`；merge SHA 字段写**占位 `{{merge SHA 待回填}}`**（merge 动作尚未发生、SHA 不存在故无法预填，见硬约束），PR 号在阶段7 已知可写实。该翻牌 commit 随 PR 一并合并进 `main`，使 main 上该卡内容天然 PASSED，从源头消除翻牌债务（不再依赖事后补刷）。**时序锚点**：HITL-5 授权之后、merge 之前——不可更早（更早翻 = 提前宣告 PASSED，不合规）。**占位即合规终态**：merge SHA 事后回填为 SHOULD（可选 · 由 autoflip hook 提示 / flip_merged.sh 兜底 / 人工补），不构成闭合阻塞；占位串 `{{merge SHA 待回填}}` 为**纯人读约定、无机械消费方**（hook/脚本均不识别它）。**守界**：翻牌 commit 仍在分支内随 PR 合并（守 DF-007 不直推 main）；翻牌是维护副作用、不进任一阶段门禁判定式（守 ADR-005）。与 DF-008 互引——本条管「branch→PR→merge 链的 merge 前一刻」翻牌时序，DF-008 管分支-PR-merge 策略，不重复定义。**R-1 诚实声明**：本环境无机械 pre-merge 钩子点（合并经 `gh pr merge` / GitHub 网页），DF-013 是流程性 MUST 纪律、靠 Owner 自律；漏翻由 autoflip hook（状态安全网）+ flip_merged.sh（兜底）缓解，非根除。 | MUST 人工 | 人工（阶段10 流程核对 summary 终态已翻 + merge SHA 占位在场） |

### 2.1 失败回退路由（Python 项目表述）

| 失败现象 | 回退到 | 说明 |
|---|---|---|
| CI/pytest **用例数 0** 或 `total_tests == 0` | 阶段 5（单测编写） | 对应原「0/0」 |
| **语法/导入错误**、pytest **收集失败**、`pip install` 失败 | 阶段 3（编码实现） | 对应原「编译错误」 |
| **mypy** 类型检查失败（若项目启用且门禁纳入） | 阶段 3 或 4 | 按项目约定 |
| 需求与 spec 不符 | 阶段 1（需求分析） | 不变 |
| 阶段 4 **代码级 MUST FIX** | 阶段 3（v2）+ 受影响单测返工 | 4∥5 并行下的回退（DF-010）：仅返工 v2 改动波及的测试，不受影响测试保留 |

> 历史文档中的「编译错误」均指 **构建或测试无法通过**，在 Python 仓中以上表为准。

### 2.2 DF-009 门禁降级 / N-A 叠加条款（卡型两类拆分 · 反假绿安全边界 · 不破冻结判定式本体）

> 本小节是 DF-009 的**叠加子条款**——只在 DF-009 冻结判定式 `exit==0 && total>0 && passed==total`（BL-INV-1）**之外**补「`total` 天然不可 >0 的卡如何过阶段8」一类场景，**不删、不改、不旁路削弱**该判定式本体。它把此前 W4+ 批次出现的**逐卡 ad-hoc 临时豁免**收编为统一规范条款（替代逐卡随手豁免 · 防豁免泛滥）。
>
> **核心原则**：阶段8 门禁按卡型**严格拆两类**——卡是否产生可机械计数的测试（`total` 能否天然 >0）是唯一判别轴；**禁止**把两类混为一谈、更**禁止**把本可计数的验证型卡引向降级旁路。

**① 纯文档卡（无任何可运行测试 → 阶段8 门禁判定 N/A）**

- **判别**：本卡产物全为规则 / 文档 / summary / roadmap / 改进记录等 markdown，**不引入也不改动任何可运行测试**，`total` 天然不可能 >0（不是"测试没写"，而是"本质上无可计数测试对象"）。
- **判定**：阶段8 门禁判 **N/A（不适用）**——冻结判定式因 `total>0` 天然不可满足而对本卡不适用，**不视为门禁失败、不触发回退阶段5**。
- **不得空过（MUST）**：判 N/A 的卡**必须在 `ci_result.md` 附替代正向证据**，至少其一——(a) 人工复核留痕（改了哪些文件、逐项自检结论）；(b) 文档自洽核对（交叉引用一致 / 无双源漂移 / 锚点在场）；(c) `git diff` 复核摘要（只动该动区块、无丢段/串行/乱码）。**N/A ≠ 空过**：无替代正向证据的 N/A = 不合规（等同跳过验收）。

**② 验证型卡（守护断言即可计数测试 → 走既有冻结判定式，不属降级）**

- **判别**：本卡引入 / 改动**可计数的 pytest/vitest 测试**——典型如带 `test_*_invariants.py` 守护断言的规则卡、带结构守护的脚本卡，`total>0` 天然成立。
- **判定**：**走 DF-009 既有冻结判定式 `exit==0 && total>0 && passed==total`**（经 `testCommand()` → `outputParser()` → `eval_gate_contract.sh`），与常规代码卡同判、**不属降级、不走 N/A**（守 BL-INV-1）。
- 例：守护断言本身就是门禁可消费的客观计数对象，绝不能因为"卡只改了 .harness/ 制品"就误判为纯文档卡而降级。

**③ 反假绿安全边界（关键 · 防削弱 BL-INV-1 反假绿意图）**

- 门禁降级（判 N/A）**仅在 `total` 天然不可 >0 时触发**，且**必须附替代正向证据**（见①不得空过）——两条件缺一不可。
- **禁止**把**可机械计数的验证型卡**（守护即测试、`total` 本可 >0）引向 N/A 降级旁路：那等于给本应跑冻结判定式的卡开后门，直接削弱 BL-INV-1 的反假绿意图（"有可计数测试却不计数"= 假绿温床）。
- 卡型判别有歧义（既有文档改动又疑似可加守护）时，**默认归②验证型卡**（举证责任偏向"应计数"），由阶段4 评审取证后再定；**不得**为图省事默认归①降级。

**④ 与逐卡临时豁免的关系**：本条款是 W4+ 批次"逐卡 ad-hoc 门禁豁免"的**规范化替代**——卡型一经按①/②判定即按统一规则处置，**不再逐卡随手写临时豁免**。确需偏离本条款（如既非纯文档亦无法计数的特殊卡）才回落 §5 例外与豁免流程（须 Reviewer + HITL 批准 + summary 留痕）。

## 3. 正例 / 反例

- 正例：阶段2 评审 APPROVED 且 HITL-2 确认后才进入阶段3。
- 正例：阶段8 绑定层 `testCommand()` 全绿且 `total_tests > 0`（经 `outputParser()` 还原三元组 + `eval_gate_contract.sh` 判 PASS）后进入阶段9（pytest/vitest 两 profile 同判定式）。
- 正例：阶段8 门禁经 `testCommand()` + `outputParser()` + `eval_gate_contract.sh` 栈无关判定，换 profile 只换测试命令/解析器、判定式本体冻结（`exit==0 && total>0 && passed==total`）（符合 DF-009 / BL-INV-1）。
- 反例：阶段8 门禁规则文本写死单一框架命令（如硬编码 `pytest -q` 为唯一门禁命令），新增 profile 须改流程层（违反 DF-009 / BL-INV-2 流程层零栈特定串）。
- 正例：阶段7 在 `change/chore-git-workflow-hygiene-20260611` 分支按阶段多次提交 → 经显式授权 push 开 PR → HITL-5 前经显式授权以 merge commit 合并进 `main`（符合 DF-007/DF-008）。
- 正例：切分支前先 `git status` 确认工作树干净，`git checkout` 单独执行后立即 `git branch --show-current` 核对再继续下一步（符合 DF-011）。
- 正例：阶段10 HITL-5 用户授权合并后、`gh pr merge` 前，在变更分支上追加翻牌 commit——summary `| 总体状态 |`→`PASSED`、`| 当前阶段 |`→`10`、merge SHA 字段写占位 `{{merge SHA 待回填}}`（SHA 尚不存在）、PR 号写实；该 commit 随 PR 合并进 main，main 上内容天然 PASSED（符合 DF-013）。
- 反例：编码评审有 MUST FIX 却直接推送（违反 DF-002/DF-003）。
- 反例：pytest 未跑或 `ci_result.md` 仅填 `SUCCESS` 但用例数为 0（违反 DF-002 / 阶段8 门禁）。
- 反例：未建分支、未开 PR，直接 `git push origin main`（违反 DF-007/DF-008）。
- 反例：`git checkout main && git merge change/xxx && git push`——前半段 checkout 因工作树不干净失败时，后续写操作落在非预期分支（违反 DF-011 第②条）。
- 反例：`code_review` 报告大段复述 spec 原文 + 整段粘贴被评审代码 + 流水账式过程记录，结论与证据被淹没（违反 DF-012 (a)）。
- 反例：对 `summary.md` 用携带大段 `old_string` 的 `Edit` 做整篇刷新，高密度中文+表格下工具调用 malformed 被解析器拒绝 → 编辑根本未执行、状态未落盘却被误以为已生效（违反 DF-006 写法纪律：整篇刷新应 `Write` 整文件 + 写后 `git diff` 复核）。
- 反例：阶段10 不在分支内翻牌，等 PR 合并进 main 后才事后补刷 summary 终态（产生翻牌债务：合并瞬间 main 上该卡仍非 PASSED，`list_flows` 误报为在途）；或分支内翻牌时强行编造一个未发生的 merge SHA 填入（SHA 是 merge 产物、合并前不存在，编造 = 伪造溯源）——两者均违反 DF-013（应分支内翻 + merge SHA 写占位）。

## 4. 校验方式

- 机械（DF-002/006 + DF-007 危险命令与直推 main 拦截部分 + DF-011 第②条复合命令拦截）：门禁判定式与 hooks（`docs/stage-01-Harness体系建设/03-质量与改进/08-质量门禁与反馈回路规范.md` / `docs/stage-01-Harness体系建设/04-运行手册/12-运行时集成-ClaudeCode桥接规范.md`）。
- 人工（DF-001/003/004/005/008/009/010 + DF-007 显式授权核对部分 + DF-011 第①/③条及第②条人工兜底）：阶段流转与评审时核对。
- 人工（DF-012）：评审阶段（4/6）对照报告内容结构（是否只含结论+证据表+MUST FIX、是否复述/贴码/流水账、是否对 spec/tasks 误用行数限制）。DF-012 (b) 软目标为 SHOULD，不机械拦截；(a)(c) 由评审人工把关。

### 6.1 git 暂存 SSOT 陷阱（关联 DF-006 / DF-007 / DF-008）

> 病根都是「index（SSOT）≠ 你以为的工作树内容」——**推送 / CI 跑的是 index、不是工作树**；唯有核验提交的真实内容才逮得住。statusline 卡（2026-06-22）实测踩中。

- **陷阱①（冻结旧 blob）**：`git update-index --chmod=+x <file>` 会把**当时的 blob 冻结进 index**；若之后继续 `Edit` 工作树却忘了重新 `git add`，index 卡在旧（可能损坏的）中间版（`AM` 态：`git ls-files -s` 的 staged blob ≠ `git hash-object` 工作树 blob）。阶段3 generator 自测跑的是工作树（正确）→ 推送的却是 index（损坏版）→ 险些把坏版带进 main。
- **陷阱②（pathspec 缺失整条中止）**：`git add A B C` 中**任一 pathspec 不存在 → 报 fatal、整条命令中止、什么都不暂存**。例如 `git add 新路径 旧路径(已 git mv 不存在) ...` → 整条 abort → 提交只含此前预暂存内容（漏掉真实改动），推出一个不完整提交。`R100` rename（应有内容改动却显示 100% 相同）是"编辑没进暂存"的危险信号。
- **How（核验暂存版而非工作树版）**：① 任何 `git update-index --chmod` 或分步暂存后，提交前必 `git diff --cached --name-status` + 对关键文件 `git show :<path> | <验证>` 实跑**暂存版**；② 多文件 `git add` 后核对暂存清单逐一在场，留意 `R100`（应有改动却 100% 相同 = 编辑没进暂存）；③ push 前由 reviewer / 单测**独立**揪 SSOT 脱节最可靠（statusline 卡即靠 reviewer + 单测两独立子 Agent 互证 index 损坏）。
- 关联还原纪律见 `.harness/rules/测试守护规范.md`（kill-mutant 还原纪律）、行为遵行验收见 `.harness/rules/验收执行规范.md`（行为遵行验收）。

### 6.2 git / 分支 / worktree / re-sync 卫生（7 则 · 逐则锚对口 DF）

> 权威定义在 §2 的 DF-007 / DF-008 / DF-011 / DF-013；本小节是反复踩出的操作教训，**从属于既有 DF 条目、不另立新规**。

1. **并行 job 须 worktree + 命名修正**（rm-102 / rm-105 · 关联 DF-011 / DF-008）：共享工作树下 `git checkout -b` **无法隔离**并行 job（互相踩工作树）→ 必须用 worktree。但 `EnterWorktree` 自动命名为 `worktree-change+<name>`，**不符 DF-008** 的 `change/<type>-<slug>-<date>` → 须 `git branch -m` 改名。注意：worktree 隔离**禁用**于 Harness 10 阶段产物（见 `CLAUDE.md`「子 Agent 委派 · worktree 隔离禁用契约」）。
2. **分支 / 目录前缀须在注册表内**（rm-120 · 关联 DF-008）：分支与变更目录前缀必须是 `list_flows` 认识的（**仅 `feat` / `fix` / `chore`**）；用 `refactor-` 等未注册前缀 → 被 hook 标 `unknown`。
3. **re-sync 落后基线：不只防冲突，还致假绿 + total_tests 写错**（c004 / rm-112 / rm-113 · 关联 DF-007 / DF-008）：push 前若落后 `origin/main`，re-sync 不只是防文本冲突——① 守护断言可能在缺失内容上**空满足**（假绿）；② CI 绝对计数会**写错**（须 merge 后全量复跑取真数，delta 不变但绝对数变）。`mergeable=CLEAN` **只保证无文本冲突、不保证语义安全** → 必 merge + 全量复跑确证。re-sync 后还常顺手暴露 / 抢修跨卡遗留缺陷（如执行位），须诚实记 idempotent。
4. **CLAUDE.md ∥ myCLAUDE.md 字节孪生**（rm-107 / rm-116 · 关联 DF-006 邻域）：两文件相关段须**字节完全相同** → 任何注入须同步改两处，改完 `diff -q` 仍报 IDENTICAL 才算对。（注：本条仅为被移植的知识内容，不要求每张卡同步改孪生体。）
5. **summary `当前阶段` 字段须清洁 `10`**（rm-109 · 关联 DF-006 / DF-013）：`当前阶段` 字段必须是清洁的 `10`；写成 verbose 串（如 `10 PASSED（…merge SHA…）`）会破坏 `=="10"` 解析致误分类。merge SHA 等详情放阶段记录行，不塞进字段值。
6. **roadmap 双层翻牌**（chore-flip-summary-roadmap / roadmap-flip-debt · 关联 DF-006 / DF-013）：roadmap 有两层——聚合层 `project/roadmap.md` vs 模块级 `m5_roadmap`；翻牌须刷**两层**，且聚合层"总览 ∥ 详情"两处都刷。**frontmatter `status` ≠ per-card 状态字段** → 盘点须逐卡 grep（只看 frontmatter 会误判已翻牌）。聚合层补 §C/§D 是新撰写 ≠ 翻 stale（别混进翻牌卡范围）。
7. **pre_bash_guard 误拦的干净绕法**（autosync · 关联 DF-007 / DF-011）：隔离临时 git 仓做功能测试时，`pre_bash_guard` 按命令文本可能误拦 → 干净绕法 = 把命令写成 `.sh` 再 `bash file.sh`（逃生口 `HARNESS_GIT_GUARD_BYPASS=1`，审计痕迹 = 前缀本身）。

### 6.3 阶段 8-10 收尾再合并（关联 DF-008 / DF-013）

> 阶段8（`ci_result`）、阶段9（`deployment` 报告）、阶段10（`summary` 终态 / 指标 / HITL-5 留痕）**都是在阶段7 push 之后才产出**。若在产出这些收尾文档之前就合并 PR，它们进不了那个 PR → main 上变更卡 SSOT 停留在"阶段7"，必须再开回填 PR（且按"合并授权一次性·每次 to-main 须重新经 HITL-5 授权"，还要用户再授权一次合并）。

- **复现（反例）**：PR #16→#17（封板 SHA 回填）、PR #18→#19（fix-sqlite-thread-safety 阶段7-10 收尾回填）——都是"功能已合并，但卡自身收尾记录滞后"。
- **正例**：2026-06-14 PR #20 首次正确应用——阶段 8-10 收尾全提交进分支、HITL-5 后再回填 summary 终态、最后才合并 → 单 PR 完成、零回填。RM-103（PR #27）同样正确：HITL-5 授权后**先**把 summary 终态 PASSED commit+push 进分支、**再**合并 → 终态随合并入 main，零回填。
- **升级教训（PR #28）**：同会话一个仅"一行状态翻牌"的轻量 chore 又中招——summary 留在「阶段10 PENDING / 总体状态 IN_PROGRESS」就合并了，终态翻牌没进 main。**教训：连一行状态翻牌的轻量 chore 也要在合并前把 summary 终态写进分支，别因为"只是个 chore"就松手**；不值得单开回填 PR 的一行终态修正，可随下卡起跑顺带 ride-along 刷（沿用共享 roadmap 翻牌延后纪律）。
- **How**：阶段7 push 开 PR 后，继续在**同一分支**跑阶段8（CI）→ 阶段9（部署验证）→ 阶段10（summary 终态），每步 commit + push 到该分支；**等收尾文档全部 commit 进分支、PR 内容完整后，再在 HITL-5 请求合并授权并合并**——一个 PR 含全部 1-10 阶段产物，无需回填 PR。
