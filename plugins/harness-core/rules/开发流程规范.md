# 规则 · 开发流程规范

> 稳定不变约束。告诉 Agent「流程层面的标准是什么」。阶段内部 SOP 见 `../skills/`，阶段细则见 `docs/stage-01-Harness体系建设/02-体系设计/07-十阶段流程详细规范.md`。

## 1. 适用范围

适用于需求从接收到交付的全过程流程纪律；不替代各阶段 Skill 的具体步骤。

| DF-012 | 报告类产物预算：评审/编码/单测**报告类产物**（`*_review_vN` / `coding_report_vN` / `unit_test_report_vN`）须满足——(a) **内容硬约束**：只含「结论 + 证据表 + MUST FIX 清单」，禁止复述 spec、禁止大段贴代码、禁止过程流水账；(b) **软目标**：~100 行自检线，超出须自检多余部分是证据/MUST FIX 而非叙述；(c) **铁律**：证据完整性 > 简洁，任何 MUST FIX 或必需证据不得为压缩篇幅而省略；(d) **不做硬行数截断**（100 行为气味阈值非铡刀）。**豁免**：定义性产物 `spec.md` / `tasks.md` 不设行数限制 | MUST 人工〔(a)(c) 强制〕+ SHOULD〔(b) 软目标〕 | 人工评审（阶段4/6 对照报告内容结构） |
| DF-013 | **阶段10 原分支终态与合并证据分离**：阶段 8/9 完成并取得 **HITL-5 最终交付确认**后，在原变更分支把本卡 `summary.md` 翻为唯一闭合终态——`| 总体状态 |` → `PASSED`、`| 当前阶段 |` → `10`，形成终态 commit。阶段 7 `关键产出（链接）` 单元格的稳定 PR 身份（SPI）唯一写法为 `PR #<n> · https://github.com/<owner>/<repo>/pull/<n>`；不得写 PR head、终态 commit SHA 或 merge SHA。merge SHA 是按需只读派生合并证据（DME），不是闭合字段；闭合唯一判定为 `PASSED + 10`。终态 commit 后须取得独立 push 授权并推送，运行时核对 GitHub PR head 等于本地终态 commit SHA，并等待该 head 的 required checks 完成；head/push/checks/授权任一步失败即停止，修复产生新 commit 时重新核对。随后须取得独立 merge 授权方可合并；HITL-5、CCA、push、merge 各 gate 互不替代。merge 后仅按需只读查询 DME，禁止为 SHA 回填修改 summary、创建卡/commit/push/PR。历史已记录的真实 SHA 保留，旧占位不构成债务，不批量迁移。与 DF-008 互引——本条管终态 commit→push/head/checks→merge→只读 DME 时序，DF-008 管分支-PR-merge 策略。 | MUST 人工 | 人工（核对 PASSED+10、SPI 二段式、独立授权、PR head/required checks 与 merge 后零回填） |
| DF-015 | **HITL 载体 + 记录真实性硬门（feat-hitl-authz-hardening · 治理缺口①修复）**：本条把「HITL 停点必须由真实用户输入闭合、不得由模型自答虚构」从文字纪律升为 MUST，共三句可独立 grep：**(a) HITL 载体强制**——HITL-1~5 五个停点必须以 **AskUserQuestion 工具提问**、或**结束本回合等待用户下一条真实输入**为载体闭合；**禁止呈现停点后自答**（模型在同一回合内既问又替用户拍板 = 伪造确认）。**(b) bg/headless 会话硬停**——后台（background）/ 无头（headless）会话触及任一 HITL 停点时**一律硬停、结束回合**，**禁止代答推进**下一阶段（长回合/bg 会话模型自律必然衰减，唯有硬停可防漂移）。**(c) summary 记录须附证据**——`summary.md` 写「HITL-n 通过 / 用户确认」等字样时**必须附用户原话摘录或 AskUserQuestion 答案摘录**（无证据的「用户拍板」= SSOT 伪证，评审须打回）。**机械协同**：push/merge 授权由 pre_bash_guard.sh 授权硬门（凭 user_prompt_state_inject.sh 落盘的用户授权台账）机械兜底；产出型阶段（1/3/5）翻牌由 pretool_summary_flip_guard.sh 凭委派台账机械兜底。**R-1 诚实声明**：本条(a)(c) 无机械 pre-HITL 钩子点、靠 Owner 自律 + 评审核对；(b) 的 bg 会话硬停亦为流程性 MUST，机械护栏仅覆盖其下游 push/merge/翻牌动作、不覆盖 HITL 应答本身。**R-2 台账真实性定性（code_review_v1 S-2）**：授权/委派台账由 hook 在真实用户输入时落盘、正常流程模型不触碰它；但本护栏是**纪律性护栏而非对抗性安全沙箱**——模型经 Bash 重定向 / Write 独立追加伪造台账条目的行为面**客观存在**，属越出纪律护栏的对抗面（不在本卡威胁模型内，审计可经 transcript 复核发现），故不得断言「模型无法伪造」。机械兜底的作用是把无意漂移的绕过面从文字纪律抬升为需蓄意越界才能突破的硬门，而非对抗性完备。 | MUST 人工（机械部分兜底下游动作） | 人工（评审核对 HITL 载体真实性 + summary 附证据）+ 机械（授权硬门 / 翻牌硬门 hook 兜底下游） |
| DF-014 | **长卡上下文分段纪律（阶段边界分段 · 非中途 · DF-013 同体例）**：单卡上下文在**阶段7 起手**前后累积超阈值（**~200k token** 为上位人工判据；机械侧双阈值 T2 160k / T6 200k 细则见 **DF-017**）时须**分段续跑**——**新会话（启动序列重入）优先**、`/compact` 备选（分段手段优先级机械细则：同窗 `/clear` 优先于新窗口、二者均优于 `/compact`，见 **DF-017**）；分段点**只落在 `summary.md` 刚覆盖式刷新的阶段边界**（**首选阶段 6/7 之间**、次选 HITL-2 后），**禁止阶段中途分段**（中途分段丢在制品未落盘的工作态、易致状态错位）。**「新会话优于 compact」理由**：新会话经启动序列从磁盘真态（`.harness/changes/*/summary.md` + `git status`）重建上下文，比 `/compact` 有损压缩更可靠；两者**前提相同** = SessionStart 注入随新会话 / compact 重触发重建常驻契约（标头 `[harness:resident_contract]` · 四 matcher `startup/resume/clear/compact` · 见 ADR-018）。**与 DF-017 互引**：DF-014 是上位人工纪律（何时该分段、切点落哪个阶段边界的人工判断），DF-017 承载其机械化落地细则（双阈值 env 触发 / 交接哨兵跨会话续推 / 先分段后授权 / 禁中途分段 / `/clear` 手段优先级），两者交叉引用不重复定义（体例同 DF-013↔DF-008）。**R-1 诚实声明**：本条是流程性 MUST 纪律、靠 Owner 自律；本环境无机械 pre-context-limit 钩子点，超限判断由 Owner 观测上下文用量触发。 | MUST 人工 | 人工（分段时核对：分段点落 summary 刚刷新的阶段边界、非阶段中途；优先新会话重入） |

### 2.2 DF-009 门禁降级 / N-A 叠加条款（卡型两类拆分 · 反假绿安全边界 · 不破冻结判定式本体）

> 本小节是 DF-009 的**叠加子条款**——只在 DF-009 冻结判定式 `exit==0 && total>0 && passed==total`（BL-INV-1）**之外**补「`total` 天然不可 >0 的卡如何过阶段8」一类场景，**不删、不改、不旁路削弱**该判定式本体。它把此前 W4+ 批次出现的**逐卡 ad-hoc 临时豁免**收编为统一规范条款（替代逐卡随手豁免 · 防豁免泛滥）。
>
> **核心原则**：阶段8 门禁按卡型**严格拆两类**——卡是否产生可机械计数的测试（`total` 能否天然 >0）是唯一判别轴；**禁止**把两类混为一谈、更**禁止**把本可计数的验证型卡引向降级旁路。

**① 纯文档卡（无任何可运行测试 → 阶段8 门禁判定 N/A）**

- **判别**：本卡产物全为规则 / 文档 / summary / roadmap / 改进记录等 markdown，**不引入也不改动任何可运行测试**，`total` 天然不可能 >0（不是"测试没写"，而是"本质上无可计数测试对象"）。
- **判定**：阶段8 门禁判 **N/A（不适用）**——冻结判定式因 `total>0` 天然不可满足而对本卡不适用，**不视为门禁失败、不触发回退阶段5**。
- **不得空过（MUST）**：判 N/A 的卡**必须在 `ci_result.md` 附替代正向证据**，至少其一——(a) 人工复核留痕（改了哪些文件、逐项自检结论）；(b) 文档自洽核对（交叉引用一致 / 无双源漂移 / 锚点在场）；(c) `git diff` 复核摘要（只动该动区块、无丢段/串行/乱码）。**N/A ≠ 空过**：无替代正向证据的 N/A = 不合规（等同跳过验收）。

**② 验证型卡（守护断言即可计数测试 → 走既有冻结判定式，不属降级）**

- **判别**：本卡引入 / 改动**可计数的 pytest/vitest 测试**——典型如带 `test_*_invariants.py` 守护断言的规则卡、带结构守护的脚本卡，`total>0` 天然成立。
- **判定**：**走 DF-009 既有冻结判定式 `exit==0 && total>0 && passed==total`**（经 `testCommand()` → `outputParser()` → `eval_gate_contract.sh`），与常规代码卡同判、**不属降级、不走 N/A**（守 BL-INV-1）。
- 例：守护断言本身就是门禁可消费的客观计数对象，绝不能因为"卡只改了 .harness/ 制品"就误判为纯文档卡而降级。

**③ 反假绿安全边界（关键 · 防削弱 BL-INV-1 反假绿意图）**

- 门禁降级（判 N/A）**仅在 `total` 天然不可 >0 时触发**，且**必须附替代正向证据**（见①不得空过）——两条件缺一不可。
- **禁止**把**可机械计数的验证型卡**（守护即测试、`total` 本可 >0）引向 N/A 降级旁路：那等于给本应跑冻结判定式的卡开后门，直接削弱 BL-INV-1 的反假绿意图（"有可计数测试却不计数"= 假绿温床）。
- 卡型判别有歧义（既有文档改动又疑似可加守护）时，**默认归②验证型卡**（举证责任偏向"应计数"），由阶段4 评审取证后再定；**不得**为图省事默认归①降级。

**④ 与逐卡临时豁免的关系**：本条款是 W4+ 批次"逐卡 ad-hoc 门禁豁免"的**规范化替代**——卡型一经按①/②判定即按统一规则处置，**不再逐卡随手写临时豁免**。确需偏离本条款（如既非纯文档亦无法计数的特殊卡）才回落 §5 例外与豁免流程（须 Reviewer + HITL 批准 + summary 留痕）。

## 5. 例外与豁免

- 流程偏离（如紧急修复简化阶段）须经人工在 HITL 点明确批准，并在 `summary.md` 记录理由与补救措施。
- **门禁例外密度收紧（防逐卡临时豁免泛滥）**：单张变更卡的门禁例外 / 豁免**累计 >2 项**时，须**回溯需求定义**——判断本卡是否应**改判卡型**（纯文档卡 N/A / 验证型卡走冻结判定式 / 常规代码卡，见 §2.2）而非靠逐项临时豁免堆叠。逐卡随手豁免堆积是 W4+ 批次收尾纪律滑坡的征兆之一（病因见 `.harness/improvement/failure-record-w4-closure-debt-20260629.md`）；正解 = 一次性按卡型规范处置（§2.2），而非例外清单越列越长。
