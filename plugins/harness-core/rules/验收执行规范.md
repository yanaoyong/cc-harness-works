## 1. 验收套件刷新前须枚举全部同族守护

**铁律**：刷新一个验收套件（如 `.harness/acceptance/research-phase/`）时，它常被**不止一个**结构守护测试守护，且这些守护分属**不同卡**。漏掉任一守护 → 改套件文件集后计数/正则断言漂移，以阶段8 全套件回归暴露（净回归 +1），而单卡视野根本看不到兄弟卡建的守护。

**实例（chore-acceptance-m05-tier-refresh / PR#121）**：research-phase 套件由**两个**同族守护守——`test_research_phase_acceptance_structure.py`（cases 守护，RM-114 建）+ `test_rp_result_stubs.py`（results 空壳守护，chore-rp-result-stubs-20260623 建）。本卡把 cases/+results/ 从 9→12，只在 spec/HITL-1 申报了第一个守护，**漏了第二个** → 阶段8 全套件回归核才暴露 `test_rp_result_stubs::test_ac1`（钦定 9）由绿转红（净回归 +1）。

**Why**：守护测试硬编码"钦定全集/计数基线"（`EXPECTED_*` + 文件名正则如 `RP-0[1-9]`）。改套件文件集必同步**所有**引用它的守护，否则计数/正则断言漂移。单卡视野只看到自己那张卡建的守护，看不到兄弟卡建的。

**How to apply**：

1. **阶段1 划范围前先枚举全部守护**：`grep -rl "<套件路径片段>" tests/`（如 `grep -rl "acceptance/research-phase" tests/harness_tools/`）枚举**全部**守护，逐个进白名单。漏一个 → 阶段8 计数回归。
2. **改文件名正则基线时连带拓宽**：如 `RP-0[1-9]`→`RP-(0[1-9]|1[0-2])`，否则新文件的潜在失败被早期断言遮蔽（masked），一旦前序文件修了才爆。
3. **区分"本卡引入"vs"既存失败"**：对照**前一张相关卡的 ci_result 既存基线**（m05 PR#120 记 11 failed）——回到同一既存数 = 净回归 0。base 复现法见 `.harness/rules/开发流程规范.md` §6 操作教训（阶段8-10 收尾再合并）；masked 失败见 `.harness/rules/测试守护规范.md`（假绿反模式 / masked 失败）。
4. **兄弟卡的既存失败不在本卡 delta 则不修**：如 RP-01/02 的 RAb 回填问题，留痕指向后续卡，不在本卡修。

## 2. 旁路工具行为遵行验收必须在仓库根起会话跑

**铁律**：旁路工具（codegraph / wiki · ADR-005）的「行为遵行 / 真实采纳」验收案例（CGR/WB 第五维），**必须在仓库根（项目级 CLAUDE.md 可加载处）起 Claude 会话跑**；**绝不能 `cd` 进 /tmp fixture 后再起会话**。

**Why**：harness 的 codegraph 纪律（CG-Q/CG-P/CG-S/CG-Q3）是**项目级**的——住在 `/workspace/myharness/CLAUDE.md`（10 处命中）+ `.harness/` + 项目级 hooks（`user_prompt_state_inject.sh` 等）。已核实：**无 `~/.claude/CLAUDE.md` 全局兜底**、`~/.claude/settings.json` 对 codegraph/prompt_state 零命中。从 `/tmp/tmp.X` 起的会话加载不到这些 → 等于在测一个**没装被测纪律的「裸 Claude」**。而 RM-115/116 恰恰就是那套纪律 → 裸 Claude 不用 cg 是预期、对纪律零证明力 = **假阴**。

**How to apply**：

1. **行为采纳跑 = 仓库根纯终端起 `claude` + 拿仓里真实、有跨文件调用方的真符号**（如 `get_discount_rate`：service+test 跨 A/B 两套）问"谁调用+牵连哪些文件"，看 agent 是否路由 cg。**不要造 /tmp fixture**——fixture 既剥纪律、又因在仓外 cg 索引/find 都够不到 → 双重假阴。
2. **编辑器卫生**：纯终端起会话、**输入里无 `⧉ Selected` 行**（VS Code 选区若高亮到 cg 引用会反向污染"自主路由"观察，见 RM-117 CGR-01 带污染保留）。
3. **实证（2026-06-21 RM-117 Xb 重分析）**：CGR-02 在 /tmp 连跑 3 次都"不用 cg"判可疑 LOW；改在仓内拿 `get_discount_rate` 跑，agent 全套 CG-Q/CG-P/CG-S/CG-Q3 打满（cg 主力 + grep 交叉核验 + 揪出 cg 漏召的字符串 mock）→ **缺口证伪 = 测试方法假象**，不 REOPEN。唯一纪律在场的首跑也早援引 "per CG-Q routing"。
4. **剧本须避先验污染**：行为案例的 prompt / spec / system prompt **不得内嵌被测工具的引用**（如剧本里写「请查 wiki」「per CG-Q」、或 strategist.md 内嵌 ADR 引用且源可直接 grep）→ agent 会绕过被测工具直接答，遵行率虚高（c-wiki WB-Q1 内嵌 ADR 引用 1/3 vs 干净两跳 WB-Q2 3/3）。fence 内的 `[wiki 侧]/[cg 侧]` 标注也是泄露 → 移出 fence。用 `_TEXT_BLOCK` 正则只抽 text 块做断言。
5. **推论：纯功能/静态案例可任意 cwd 跑**：退出码、diff、grep 静态核验（如 CGR-03~09）不依赖纪律加载、可任意 cwd 跑；**只有"行为采纳"案例受此约束**。
6. **dogfood 真跑能炸出静态验收照不到的实现 bug**：b 卡 dogfood 实跑两跳剧本直接揪出 R4 审计脚本漏检第一跳 `wiki/...md` 页路径的假阴——行为遵行维度的价值正在于此（静态验收只验"能用 + 条文在位"，照不到"是否被真调用"）。
