## 2. 规则条目（后端分层 · ES-001~007）

> 以下 ES-001~007 为**后端 Python 分层 + 通用工程约束**，语义自模板原版 v1.1.0 沿用不变。前端分层对等条目见 §2.4（ES-FE-1~5）。

| 编号 | 规则 | 级别 | 校验方式 |
|---|---|---|---|
| ES-001 | 分层自上而下：**api（表现）→ service（业务）→ repository（数据）→ adapter（外部适配）**；依赖**单向向下** | MUST 机械 | import 方向检查 / 评审 |
| ES-002 | 禁止逆向依赖（下层不得 import 上层，如 repository 不得 import api） | MUST 机械 | 依赖方向检查 |
| ES-003 | 新增代码必须归入 §2.1（后端）/ §2.3（前端）约定目录；不得散落在仓库根或随意新建顶层包 | MUST 人工 | 评审 |
| ES-004 | 包/模块/文件 **`snake_case`**（Python）；测试放 `tests/`；Pydantic 模型放 `model` 或 `schemas`。前端命名见 ES-FE-4 | SHOULD | ruff + 评审 |
| ES-005 | 新增对外 API 或独立模块须在 Wiki 或变更 `spec.md` 登记 | SHOULD | 评审 |
| ES-006 | 公共能力优先复用项目内已有包，禁止重复造轮子（防熵增） | SHOULD | 评审 |
| ES-007 | **外部可执行组件**（含 bin 脚本 / install.sh / references / tests 等可执行资产，区别于 `.harness/skills/` 的纯 SOP markdown）须落 **`.harness/components/<component>/`**；该专区是外部组件的约定落点，**不触碰 ES-003**（落点在 `.harness/` 内、非仓根顶层包）。组件以符号链接 `.claude/skills/<component>` → `.harness/components/<component>/` 注册供 claude 识别（约定段见 §2.2） | MUST 人工 | 评审（落点核对） |

### 2.1 后端标准目录树（Python 业务 / Demo）

> 对比测试与新建 Python 业务服务均以此为准；路径相对项目根（如 `demo/price-service/` 或 `harnessdemo/price-service/`）。

```text
{project-root}/
├── README.md
├── pyproject.toml              # 或 requirements.txt（须锁定依赖方式之一）
├── src/{package_name}/         # 例：src/price_service/
│   ├── __init__.py
│   ├── main.py                 # FastAPI app 工厂 / 启动入口
│   ├── api/                    # 表现层：路由、入参校验、委托 service
│   │   └── *_router.py
│   ├── service/                # 业务层
│   ├── repository/             # 数据访问层
│   ├── adapter/                # 外部系统适配（可选，无则可省略目录）
│   └── model/                  # 领域模型 / Pydantic schemas（按需）
└── tests/
    ├── conftest.py
    └── test_*.py
```

**分层与目录对应**：

| 逻辑层 | 目录 | 允许依赖 |
|---|---|---|
| api | `api/` | service、model（schemas） |
| service | `service/` | repository、adapter、model |
| repository | `repository/` | model、标准库/ORM |
| adapter | `adapter/` | model、httpx 等客户端 |

### 2.2 `.harness/components/` 外部可执行组件专区（ES-007 约定）

> 此专区不属于 §2.1「Python 业务/Demo 标准目录树」（那是业务分层树），而是 **Harness 模板仓自身制品**的一类（对齐 §1 适用范围「Harness 模板仓自身按制品树」）。承接 ADR-004（组件落点 = `.harness/components/<comp>/`）。

```text
.harness/
├── components/                       # 外部可执行组件专区（ES-007 / ADR-004）
│   ├── wiki-engine/                  # 文档 wiki 引擎（bin/install.sh/references/tests）
│   │   ├── bin/                      # wiki-rescan / wiki-lint / wiki-extract / wiki-ingest-cheap（仓库根直接执行 · 有执行位）
│   │   ├── install.sh                # 安装/回滚（只加不改宿主跟踪文件 · git diff 仅新增）
│   │   ├── references/schema.md      # 提炼页 schema + 提示词模板
│   │   └── tests/
│   └── codegraph/                    # 代码索引组件（同此约定 · ADR-004 两组件统一）
├── skills/                           # 纯 SOP markdown（10 阶段 / 元流程做法说明书）—— 与 components/ 正交
└── rules/  changes/  agents/  ...    # 其余制品
```

**约定要点**：

| 约定 | 内容 | 锚点 |
|---|---|---|
| **语义正交** | `.harness/components/`（外部可执行组件）≠ `.harness/skills/`（纯 SOP markdown）≠ `.claude/skills/`（claude 注册入口 · 符号链接） | ADR-004 |
| **统一容纳** | wiki-engine + codegraph 均落 `.harness/components/`（两组件统一 · ADR-004 / C-006①） | ADR-004 |
| **注册方式** | `.claude/skills/<comp>` → `.harness/components/<comp>/` 符号链接，与其他 skill 一致；install.sh 建链接可回滚 | c4-container.md ⑧ / R-011 |
| **产物根分离** | wiki **产物根仍 `wiki/`**（wiki 内容产物，**非组件落点**），不受组件落点变更影响（S-005） | scope §7④ |

**`.harness/scripts/` 脚本登记（ES-005 · 与 `.harness/components/` 正交）**：

> `.harness/scripts/` 与 `.harness/components/` 的「外部可分发可执行组件」（含 install.sh/references/tests 的自足组件包）语义正交——本目录收纳的是**扁平单文件脚本**，**非** ES-007 组件专区。
>
> **本目录同时容纳两类脚本，登记时必须逐条标注归属类别（ES-005 · fix-script-dist-and-gate-fixes-20260713 修正）**：
>
> - **【甲】随 core 分发脚本的镜像**：**权威源在 `plugins/harness-core/scripts/`（见 ES-008）**，经 `session-start.sh` 的 `_mirror_pairs`（`plugins/harness-core/scripts` → `.harness/scripts`）送达**消费方项目**。此类脚本被消费方 SOP 直接点名（skill / slash 命令 / rules / 配置模板注释），消费方项目**必须**收到；本仓的 `.harness/scripts/<X>` 只是其运行时镜像，两侧内容须逐字节一致。
> - **【乙】纯本仓自用脚本**：只服务本仓开发态/发布态（导出模板、发布 sync、成本观测、身份验证等），**不进插件包**、不随分发送达消费方。
>
> **历史修正（根因留痕）**：本节旧表述把 `.harness/scripts/` **整体**归为一类，并把若干**实为【甲】类分发镜像**的脚本举例为该类代表——**这一整体定性与事实矛盾**：`list_changes.sh` 是随插件分发的 `/list-changes` slash 命令的实现载体（`plugins/harness-core/commands/list-changes.md` 直接 `bash .harness/scripts/list_changes.sh`），`flip_merged.sh` 由 DF-013 与 `session_start_autoflip.sh` 的用户提示点名，`init_identity.sh` 由分发出去的 `HARNESS_CONFIG.yaml` 模板明文指示消费方运行——三者**均须随插件送达消费方**。**照旧表述办事的人不会想到要把这些脚本放进 plugin 包**——这正是 `fix-script-dist-and-gate-fixes-20260713` 修复的分发缺口的根因。故本节改为甲/乙二分并逐条标注类别。
>
> **纪律**：新增脚本及其配套配置须在下表登记落点与职责，**并显式标注甲/乙类别**；标【甲】者**必须同时落 `plugins/harness-core/scripts/`（权威源）**，并把该文件扩进 `tests/harness_tools/test_plugin_display_parity.py` 的 `PARITY_PAIRS` / `SCAFFOLD_SCRIPTS` 两份清单（见 ES-008）。

### 2.4 FE 分层 ↔ 目录映射（ES-FE · 对应 ES-001~004 后端范式）

> ES-FE-1~5 是前端侧对等条目，与 ES-001/002/003/004 同范式（后端权威仍在 §2 / §2.1）。

| 编号 | 规则 | 级别 | 校验 |
|---|---|---|---|
| ES-FE-1 | FE 分层自上而下：**components（UI）→ containers（容器）→ hooks（逻辑）→ services（数据/外部）**；依赖**单向向下**（components 不得直接 import services 越层调后端，须经 containers/hooks 编排） | MUST 机械 | import 方向检查 / 评审 |
| ES-FE-2 | 禁止逆向依赖：services 不得 import containers/components；hooks 不得 import containers | MUST 机械 | 依赖方向检查 |
| ES-FE-3 | UI 组件（components/）保持纯展示——无副作用、无直接 API 调用；业务逻辑在 hooks/containers，外部调用在 services（对应后端 CODE-003 router 瘦身范式） | MUST 人工 | 评审 |
| ES-FE-4 | 文件命名：组件 `PascalCase.tsx`，hooks `useXxx.ts`，services `xxxService.ts`，类型放 `model/`；测试 `*.test.tsx` | SHOULD | 评审 + lint |
| ES-FE-5 | services 层外部调用须设超时 + 降级（对齐后端 CODE-002/R-002 语义 · 见 `项目编码规范.md` 栈特定层 FE-3） | MUST 人工 | 评审 |

| FE 逻辑层 | 目录 | 允许依赖 |
|---|---|---|
| components（UI） | `components/` | model、utils |
| containers（容器） | `containers/` | components、hooks、services、model |
| hooks（逻辑） | `hooks/` | services、model、utils |
| services（数据/外部） | `services/` | model、utils、HTTP 客户端 |
