## 2. 规则条目（后端分层 · ES-001~007）

> 以下 ES-001~007 为**后端 Python 分层 + 通用工程约束**，语义自模板原版 v1.1.0 沿用不变。前端分层对等条目见 §2.4（ES-FE-1~5）。

| 编号 | 规则 | 级别 | 校验方式 |
|---|---|---|---|
| ES-001 | 分层自上而下：**api（表现）→ service（业务）→ repository（数据）→ adapter（外部适配）**；依赖**单向向下** | MUST 机械 | import 方向检查 / 评审 |
| ES-002 | 禁止逆向依赖（下层不得 import 上层，如 repository 不得 import api） | MUST 机械 | 依赖方向检查 |
| ES-003 | 新增代码必须归入 §2.1（后端）/ §2.3（前端）约定目录；不得散落在仓库根或随意新建顶层包 | MUST 人工 | 评审 |
| ES-004 | 包/模块/文件 **`snake_case`**（Python）；测试放 `tests/`；Pydantic 模型放 `model` 或 `schemas`。前端命名见 ES-FE-4 | SHOULD | ruff + 评审 |
| ES-005 | 新增对外 API 或独立模块须在 Wiki 或变更 `spec.md` 登记 | SHOULD | 评审 |
| ES-006 | 公共能力优先复用项目内已有包，禁止重复造轮子（防熵增） | SHOULD | 评审 |
| ES-007 | **外部可执行组件**（含 bin 脚本 / install.sh / references / tests 等可执行资产，区别于 `.harness/skills/` 的纯 SOP markdown）须落 **`.harness/components/<component>/`**；该专区是外部组件的约定落点，**不触碰 ES-003**（落点在 `.harness/` 内、非仓根顶层包）。组件以符号链接 `.claude/skills/<component>` → `.harness/components/<component>/` 注册供 claude 识别（约定段见 §2.2） | MUST 人工 | 评审（落点核对） |

### 2.1 后端标准目录树（Python 业务 / Demo）

> 对比测试与新建 Python 业务服务均以此为准；路径相对项目根（如 `demo/price-service/` 或 `harnessdemo/price-service/`）。

```text
{project-root}/
├── README.md
├── pyproject.toml              # 或 requirements.txt（须锁定依赖方式之一）
├── src/{package_name}/         # 例：src/price_service/
│   ├── __init__.py
│   ├── main.py                 # FastAPI app 工厂 / 启动入口
│   ├── api/                    # 表现层：路由、入参校验、委托 service
│   │   └── *_router.py
│   ├── service/                # 业务层
│   ├── repository/             # 数据访问层
│   ├── adapter/                # 外部系统适配（可选，无则可省略目录）
│   └── model/                  # 领域模型 / Pydantic schemas（按需）
└── tests/
    ├── conftest.py
    └── test_*.py
```

**分层与目录对应**：

| 逻辑层 | 目录 | 允许依赖 |
|---|---|---|
| api | `api/` | service、model（schemas） |
| service | `service/` | repository、adapter、model |
| repository | `repository/` | model、标准库/ORM |
| adapter | `adapter/` | model、httpx 等客户端 |

### 2.3 前端标准目录树（React + Vite + TypeScript）

> 首个 FE profile（D-前端 · S-005）；与后端 8 分层 Spec 对等。路径相对前端项目根（如 `demo/price-web/` 或 FE example 根）。承接 m3.3 工程结构-fullstack.md（DRAFT）。

```text
{fe-project-root}/
├── package.json               # 锁定依赖（npm/pnpm 之一）
├── vite.config.ts
├── tsconfig.json
├── index.html
├── src/
│   ├── main.tsx               # 应用入口（挂载 React root）
│   ├── App.tsx
│   ├── components/            # UI 层：纯展示组件（无业务逻辑、无副作用）
│   ├── containers/            # 容器层：编排 UI + 调 hooks/services，承接页面级状态
│   ├── hooks/                 # Hooks 层：可复用逻辑（useXxx · 状态/副作用封装）
│   ├── services/              # 服务层：对后端 API 的调用封装（fetch/axios + 超时/降级）
│   ├── model/                 # 类型/领域模型（TS interface / type · 与后端 schema 对齐）
│   └── utils/                 # 通用工具（无业务语义）
└── tests/                     # 或 src/**/*.test.ts(x) 就近放置
    └── *.test.tsx             # vitest 用例
```

### 2.4 FE 分层 ↔ 目录映射（ES-FE · 对应 ES-001~004 后端范式）

> ES-FE-1~5 是前端侧对等条目，与 ES-001/002/003/004 同范式（后端权威仍在 §2 / §2.1）。

| 编号 | 规则 | 级别 | 校验 |
|---|---|---|---|
| ES-FE-1 | FE 分层自上而下：**components（UI）→ containers（容器）→ hooks（逻辑）→ services（数据/外部）**；依赖**单向向下**（components 不得直接 import services 越层调后端，须经 containers/hooks 编排） | MUST 机械 | import 方向检查 / 评审 |
| ES-FE-2 | 禁止逆向依赖：services 不得 import containers/components；hooks 不得 import containers | MUST 机械 | 依赖方向检查 |
| ES-FE-3 | UI 组件（components/）保持纯展示——无副作用、无直接 API 调用；业务逻辑在 hooks/containers，外部调用在 services（对应后端 CODE-003 router 瘦身范式） | MUST 人工 | 评审 |
| ES-FE-4 | 文件命名：组件 `PascalCase.tsx`，hooks `useXxx.ts`，services `xxxService.ts`，类型放 `model/`；测试 `*.test.tsx` | SHOULD | 评审 + lint |
| ES-FE-5 | services 层外部调用须设超时 + 降级（对齐后端 CODE-002/R-002 语义 · 见 `项目编码规范.md` 栈特定层 FE-3） | MUST 人工 | 评审 |

| FE 逻辑层 | 目录 | 允许依赖 |
|---|---|---|
| components（UI） | `components/` | model、utils |
| containers（容器） | `containers/` | components、hooks、services、model |
| hooks（逻辑） | `hooks/` | services、model、utils |
| services（数据/外部） | `services/` | model、utils、HTTP 客户端 |

### 2.5 全栈聚合视图（前后端分层对应）

> 全栈项目同仓时的聚合组织（如 `{root}/api`（BE）+ `{root}/web`（FE）），前后端各自遵分层、经 HTTP 契约联调（fe-integration · S-007）。承接 S-004（全栈聚合视图）。

```text
{fullstack-root}/
├── HARNESS_CONFIG.yaml        # 项目身份单源（S-003 · stack 可多栈）
├── api/  (或 harnessdemo/price-service/)   # 后端：本文 §2.1 Python 分层树
│   └── src/{pkg}/{api,service,repository,adapter,model}/
├── web/  (或 demo/price-web/)              # 前端：本文 §2.3 React+Vite 目录树
│   └── src/{components,containers,hooks,services,model}/
└── tests/ 或各自项目内 tests/
```

**全栈分层对应表**（前后端范式对等 · S-004 全栈聚合）：

| 关注点 | 后端（Python） | 前端（React+Vite） |
|---|---|---|
| 表现/编排 | api/（router） | components + containers |
| 业务逻辑 | service/ | hooks + containers |
| 数据/外部访问 | repository/ + adapter/ | services/ |
| 领域模型 | model/（Pydantic schemas） | model/（TS interface · 与后端 schema 对齐） |
| 测试 | tests/（pytest） | tests/ 或就近 *.test.tsx（vitest） |

> **联调契约一致性**：前端 `model/` TS 类型须与后端 `model/` schema 对齐（金额 int 分单位等通用约束跨栈一致 · 见 `项目编码规范.md` 通用层 GEN-1）；前后端契约漂移由 fe-integration 流程收敛（S-007）。

## 3. 正例 / 反例

**后端（Python）**：
- 正例：`service/price_service.py` 调用 `repository/price_repository.py`（向下依赖）。
- 反例：`repository/price_repository.py` 中 `from price_service.api import ...`（逆向依赖，违反 ES-002）。
- 反例：在 `api/prices_router.py` 内实现折扣计算与批量查库（违反 ES-001 + CODE-003，应委托 service）。

**前端（React+Vite）**：
- 正例：`containers/PriceQueryContainer.tsx` 调 `hooks/usePriceQuery.ts`，后者调 `services/priceService.ts`（单向向下 · ES-FE-1）。
- 反例：`components/PriceCard.tsx` 内直接 `fetch('/api/prices')`（违 ES-FE-3——UI 越层调后端，应经 container/hook + service）。
- 反例：`services/priceService.ts` 中 `import PriceCard from '../components/...'`（违 ES-FE-2 逆向依赖）。
- 反例：FE 调后端未设超时（违 ES-FE-5，对应后端 CODE-002）。
