### 6.1 git 暂存 SSOT 陷阱（关联 DF-006 / DF-007 / DF-008）

> 病根都是「index（SSOT）≠ 你以为的工作树内容」——**推送 / CI 跑的是 index、不是工作树**；唯有核验提交的真实内容才逮得住。statusline 卡（2026-06-22）实测踩中。

- **陷阱①（冻结旧 blob）**：`git update-index --chmod=+x <file>` 会把**当时的 blob 冻结进 index**；若之后继续 `Edit` 工作树却忘了重新 `git add`，index 卡在旧（可能损坏的）中间版（`AM` 态：`git ls-files -s` 的 staged blob ≠ `git hash-object` 工作树 blob）。阶段3 generator 自测跑的是工作树（正确）→ 推送的却是 index（损坏版）→ 险些把坏版带进 main。
- **陷阱②（pathspec 缺失整条中止）**：`git add A B C` 中**任一 pathspec 不存在 → 报 fatal、整条命令中止、什么都不暂存**。例如 `git add 新路径 旧路径(已 git mv 不存在) ...` → 整条 abort → 提交只含此前预暂存内容（漏掉真实改动），推出一个不完整提交。`R100` rename（应有内容改动却显示 100% 相同）是"编辑没进暂存"的危险信号。
- **How（核验暂存版而非工作树版）**：① 任何 `git update-index --chmod` 或分步暂存后，提交前必 `git diff --cached --name-status` + 对关键文件 `git show :<path> | <验证>` 实跑**暂存版**；② 多文件 `git add` 后核对暂存清单逐一在场，留意 `R100`（应有改动却 100% 相同 = 编辑没进暂存）；③ push 前由 reviewer / 单测**独立**揪 SSOT 脱节最可靠（statusline 卡即靠 reviewer + 单测两独立子 Agent 互证 index 损坏）。
- 关联还原纪律见 `.harness/rules/测试守护规范.md`（kill-mutant 还原纪律）、行为遵行验收见 `.harness/rules/验收执行规范.md`（行为遵行验收）。

### 6.2 git / 分支 / worktree / re-sync 卫生（7 则 · 逐则锚对口 DF）

> 权威定义在 §2 的 DF-007 / DF-008 / DF-011 / DF-013；本小节是反复踩出的操作教训，**从属于既有 DF 条目、不另立新规**。

1. **并行 job 须 worktree + 命名修正**（rm-102 / rm-105 · 关联 DF-011 / DF-008）：共享工作树下 `git checkout -b` **无法隔离**并行 job（互相踩工作树）→ 必须用 worktree。但 `EnterWorktree` 自动命名为 `worktree-change+<name>`，**不符 DF-008** 的 `change/<type>-<slug>-<date>` → 须 `git branch -m` 改名。注意：worktree 隔离**禁用**于 Harness 10 阶段产物（见 `CLAUDE.md`「子 Agent 委派 · worktree 隔离禁用契约」）。
2. **分支 / 目录前缀须在注册表内**（rm-120 · 关联 DF-008）：分支与变更目录前缀必须是 `list_flows` 认识的（**仅 `feat` / `fix` / `chore`**）；用 `refactor-` 等未注册前缀 → 被 hook 标 `unknown`。
3. **re-sync 落后基线：不只防冲突，还致假绿 + total_tests 写错**（c004 / rm-112 / rm-113 · 关联 DF-007 / DF-008）：push 前若落后 `origin/main`，re-sync 不只是防文本冲突——① 守护断言可能在缺失内容上**空满足**（假绿）；② CI 绝对计数会**写错**（须 merge 后全量复跑取真数，delta 不变但绝对数变）。`mergeable=CLEAN` **只保证无文本冲突、不保证语义安全** → 必 merge + 全量复跑确证。re-sync 后还常顺手暴露 / 抢修跨卡遗留缺陷（如执行位），须诚实记 idempotent。
4. **CLAUDE.md ∥ myCLAUDE.md 字节孪生**（rm-107 / rm-116 · 关联 DF-006 邻域）：两文件相关段须**字节完全相同** → 任何注入须同步改两处，改完 `diff -q` 仍报 IDENTICAL 才算对。（注：本条仅为被移植的知识内容，不要求每张卡同步改孪生体。）
5. **summary `当前阶段` 字段须清洁 `10`**（rm-109 · 关联 DF-006 / DF-013）：`当前阶段` 字段必须是清洁的 `10`；写成 verbose 串（如 `10 PASSED（…merge SHA…）`）会破坏 `=="10"` 解析致误分类。merge SHA 等详情放阶段记录行，不塞进字段值。
6. **roadmap 双层翻牌**（chore-flip-summary-roadmap / roadmap-flip-debt · 关联 DF-006 / DF-013）：roadmap 有两层——聚合层 `project/roadmap.md` vs 模块级 `m5_roadmap`；翻牌须刷**两层**，且聚合层"总览 ∥ 详情"两处都刷。**frontmatter `status` ≠ per-card 状态字段** → 盘点须逐卡 grep（只看 frontmatter 会误判已翻牌）。聚合层补 §C/§D 是新撰写 ≠ 翻 stale（别混进翻牌卡范围）。
7. **pre_bash_guard 误拦的干净绕法**（autosync · 关联 DF-007 / DF-011）：隔离临时 git 仓做功能测试时，`pre_bash_guard` 按命令文本可能误拦 → 干净绕法 = 把命令写成 `.sh` 再 `bash file.sh`（逃生口 `HARNESS_GIT_GUARD_BYPASS=1`，审计痕迹 = 前缀本身）。

### 6.3 阶段 8-10 收尾再合并（关联 DF-008 / DF-013）

> 阶段8（`ci_result`）、阶段9（`deployment` 报告）、阶段10（`summary` 终态 / 指标 / HITL-5 留痕）**都是在阶段7 push 之后才产出**。若在产出这些收尾文档之前就合并 PR，它们进不了那个 PR → main 上变更卡 SSOT 停留在"阶段7"，必须再开回填 PR（且按"合并授权一次性·每次 to-main 须重新经 HITL-5 授权"，还要用户再授权一次合并）。

- **复现（反例）**：PR #16→#17（封板 SHA 回填）、PR #18→#19（fix-sqlite-thread-safety 阶段7-10 收尾回填）——都是"功能已合并，但卡自身收尾记录滞后"。
- **正例**：2026-06-14 PR #20 首次正确应用——阶段 8-10 收尾全提交进分支、HITL-5 后再回填 summary 终态、最后才合并 → 单 PR 完成、零回填。RM-103（PR #27）同样正确：HITL-5 授权后**先**把 summary 终态 PASSED commit+push 进分支、**再**合并 → 终态随合并入 main，零回填。
- **升级教训（PR #28）**：同会话一个仅"一行状态翻牌"的轻量 chore 又中招——summary 留在「阶段10 PENDING / 总体状态 IN_PROGRESS」就合并了，终态翻牌没进 main。**教训：连一行状态翻牌的轻量 chore 也要在合并前把 summary 终态写进分支，别因为"只是个 chore"就松手**；不值得单开回填 PR 的一行终态修正，可随下卡起跑顺带 ride-along 刷（沿用共享 roadmap 翻牌延后纪律）。
- **How**：阶段7 push 开 PR 后，继续在**同一分支**跑阶段8（CI）→ 阶段9（部署验证）→ 阶段10（summary 终态），每步 commit + push 到该分支；**等收尾文档全部 commit 进分支、PR 内容完整后，再在 HITL-5 请求合并授权并合并**——一个 PR 含全部 1-10 阶段产物，无需回填 PR。

### 6.4 会 detach/commit/推送的脚本自测须隔离（关联 DF-007）

> 病根 = K4 经典「长任务 detach 后终结不了、跨回合失控」。某摄取脚本按设计 `setsid` detach 成独立后台进程，阶段3 自测**非 dry-run 真跑**对准主仓 → 跨回合持续跑约 30 分钟、逐批真摄取并 `git commit` 多个批提交落在 local main，从被污染 main 拉出的变更分支也继承了它们；Owner 靠 `git status`/`ps` 核验才揪出活进程 + git 污染（所幸只 commit 未 push，远端未污染）。

- **禁止对真实仓非 dry-run 直跑**：自测**会 ① 后台 detach ② 写/commit/推送真仓**的脚本时，**必须 `--dry-run` 或指向 tmp 隔离 git 仓**（`tmp_path` + 假私有仓）；**禁止对主仓 / 真实远端非 dry-run 直跑**。
- **委派须显式禁止**：委派 generator 自测这类"会 detach / 会 commit"的被测脚本时，prompt 必须**显式禁止对主仓非 dry-run 真跑**。
- **核验后台副作用**：提交前用 `ps -ef | grep <脚本名>` 查活进程 + 查 `git log` 找机器提交，确认无失控 detach 进程与污染提交。
- **适用性**：一切具后台 detach / 自动 commit / 自动 push 副作用的脚本自测通用；失控污染处置 = kill 父+子进程 → `git rebase --onto <干净基线>` 剔除失控提交 → `git branch -f` 把本地分支 reset 回干净基线。
