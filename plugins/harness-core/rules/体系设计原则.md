# 规则 · 体系设计原则

> 本规范移植自跨卡沉淀的体系设计判断 · 旁路工具权威 = ADR-005 · 本文是设计判断准则非门禁。
> 这些原则原先只活在用户主目录的自动记忆里，随仓移植会丢；本卡（chore-memory-porting-rules-20260627）将其固化为按需加载规范（`l1_resident: false`），供设计/评审阶段判断时检索。

## 1. 适用范围

适用于 Harness 体系（`.harness/` 制品、hook、元流程、变更流程）的**设计与评审判断**；不是机械门禁，不进任一阶段门禁判定式。遵循原则②（单源）：本文只搬"设计判断准则"本身，凡涉及 ADR-005 / DF-013 / DF-007 等既有权威源处只给指针、不复制其定义。

## 2. 规则条目（6 条设计判断原则）

> 每条保留"原则 + 病因/Why + 实例编号"语义结构；实例编号（doc-wiki-v2 / rm-119 / autoflip OQ-3 / A-2 等）为跨卡溯源锚点，照搬自源沉淀。

### 2.1 维护侧 hook ≠ 消费侧强制 hook（命脉红线 · doc-wiki-v2 / rm-119 / rm-122 / rm-123）

wiki / codegraph 集成的命脉红线：**维护侧 hook**（如 SessionStart detect-only 报新鲜度 delta）必须**非阻塞 exit0、不进门禁、不拦 Read**；**绝不能**做成**消费侧强制 hook**（拦 Read 校验慢 3–4×，codegraph L6 曾否决此设计）。两者正交并存、全程守 ADR-005（旁路工具不进任一阶段门禁判定式、不夺裁决权 · 权威定义见 ADR-005，本文不复制）。

- **Why**：维护侧 hook 是"提示新鲜度/增量"的零阻塞副作用；一旦升格为消费侧强制 hook（每次 Read 都校验），既拖慢主路径（实测慢 3–4×），又把旁路工具拽进门禁、破坏 ADR-005 旁路语义。
- **判断要点**：设计任何 hook 时先问"它是维护侧（检测/提示）还是消费侧（拦截/校验）"——维护侧守 exit0 非阻塞，消费侧默认否决。

### 2.2 单源原则：定义只放一处、其余只放指针（rm-105 / m5-discipline / result-template）

定义性文本只放**单一权威源**，其余落点只放指针（防双源漂移）。物理 delta 卡只校验"存在"、**不重写权威源**。否决「在存在性层加格式检查」这类**层级漂移**（格式属另一层，不应混进存在性校验）。

- **Why**：同一定义出现在两处即埋下双源漂移——两处迟早不一致，且无法判定谁是真。指针式引用让权威源唯一可信。
- **判断要点**：新增落点前先问"这是定义还是指针"；若是定义且别处已有，改为指针。校验类卡（存在性 delta）不得顺手加格式/内容校验（层级越界）。
- **本文自证**：本规范即遵守此原则——对 ADR-005 / DF-013 等只指针不复制（见 §2.4、§1）。

### 2.3 B-on-self 零物理落地（doc-wiki / doc-wiki-v2 metaflow）

在模板仓自身挂元流程模块（B-on-self）时，**全程零物理落地**——M0–M5 产出的是设计/基线确认文档，真正实施留给下游 K 卡。M4 对「既有成熟组件重构」= 基线确认文档，**不碰 `.harness/rules`**；但组件自带的离线套件须实跑（M4「等价收集检查」非 N/A）。被 REOPEN 的阶段回 PASSED 后进首次 M4，是**正常状态转移、非 REOPEN**。

- **Why**：模板仓对自身做元流程时，若 M 阶段就动物理文件，会与下游 K 卡的真正实施重叠/冲突；元流程产出应止于设计文档。
- **判断要点**：B-on-self 场景下，M 阶段产物只应是文档（vision/scope/architecture/基线确认），物理改动归 K 卡。M4 "等价收集检查"对自带套件的组件不可标 N/A——离线套件必须实跑。

### 2.4 DF-013：占位即合规终态（chore-flip-stage10）

阶段10 翻牌：merge SHA 是合并产物（先有鸡、后有蛋问题）+ 本环境无 pre-merge 机械钩子点 → **占位 `{{merge SHA 待回填}}` 即合规终态**。HITL-5 授权后、merge 前在分支内翻牌（总体状态→PASSED / 当前阶段→10），随 PR 合并使 main 天然 PASSED，根治翻牌债务。

- **Why**：SHA 在合并那一刻才生成，翻牌发生在合并前，故无法预填真 SHA；编造 SHA = 伪造溯源。本环境又无 pre-merge 机械钩子点，无法靠机器兜底，只能靠占位约定 + Owner 自律。
- **判断要点**：评审阶段10 终态时，看到 `{{merge SHA 待回填}}` 占位**即视为合规**，不要求合并前填实 SHA。
- DF-013 完整定义（翻牌时序 / 占位串语义 / 守界）详见 `.harness/rules/开发流程规范.md` DF-013，本文不复制其定义。

### 2.5 正交关注点解耦（autoflip OQ-3）

把正交关注点（如「消噪」与「还债」）捆在一起改是病根。消噪可放在**检测层零 mutation**；还债走合规路径（本地 commit 不 push 会致 main 分叉、破 DF-007 + autosync ff，无合规传播路径 → 改走 detect-only + 显式落库的 flip-PR）。

- **Why**：两个正交关注点捆在一处时，一个的约束会污染另一个的解法（如"还债"想直接 commit 却破了 DF-007 不直推 main）。拆开后各走各的合规路径。
- **判断要点**：发现一个改动同时承载 ≥2 个正交目的时，先拆分 locus——消噪/检测放零副作用层，状态落库走合规 PR 路径。DF-007 不直推 main 的定义见 `.harness/rules/开发流程规范.md` DF-007，本文不复制。

### 2.6 HITL「确定做」vs 证据存疑：升「是否做」≠ 删「待证伪」（doc-wiki-v2 REOPEN M3.3 · A-2）

当 HITL 拍板「确定做」一个**证据尚倾向反对**的决策（如 A-2 路由收窄·证据倾向不选纯收窄·因 Class-S 主场也接近 0 采纳），诚实范式 = **升「是否做」+ 保「效果待证伪」**：去掉「暂定/可回退/前置闸」框架（决策已定），但把存疑证据**降格为「已知风险·效果由后续验证裁决」而非删除、更不粉饰成「一定有效」**（守「采纳率证伪而非假设」立场）。

三层正交：

- **「确定做」= 决策层**（HITL 已拍板）
- **「仍 nudge 不强制」= 机制层**（机制仍是提示性 nudge，不升级为强制门禁）
- **「效果待证伪」= 认识论层**（效果好坏交后续验证裁决，不预设有效）

委派 strategist 升格时须**显式重申后两层**，防「确定做」被误读为「强制」或「假设必有效」。效果证伪交独立验收复测卡（分 locus 解 confound），不靠分卡隔离风险。

- **判断要点**：HITL 决定做一件证据存疑的事时，正确动作是把决策框架从"是否做（暂定可回退）"升为"确定做"，**同时保留**存疑证据为"已知风险待验证"——不是删掉存疑证据、也不是粉饰成必然有效。
