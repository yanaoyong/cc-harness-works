## 1. 适用范围

适用于本仓一切**守护测试 / 回归测试**的编写与验收——尤其是守护 `.harness/` 体系制品（rules / skills / agents / hooks / 脚本）结构与约束不退化的那类测试。不约束业务功能测试的写法（那归 `unit-test-write` skill）。

## 2. 守护/回归测试假绿（false-green）7 类反模式

> 守护测试最大的风险不是写不出，而是**写成永远绿**（false-green）。下列反模式在本仓多次复现，全部 = 测试存在但对真缺陷零证明力。
>
> **铁律：每条守护必须 kill-mutant 双向验（先验真红再验转绿），不能只看「现在是绿」。**（kill-mutant 做法见 §3。）

### 2.1 执行位 chmod 未进 index（最高频 · rm-101 源头 / c-wiki / m5-discipline / rm-120 / rm-122 复现）

本仓 `core.fileMode=false` → git 忽略工作树执行位。迁移/新增可执行文件（`bin/*`、`*.sh`）必须 `git update-index --chmod=+x` 把 `100755` 写进 index，并 `git ls-tree` 核对 mode。否则：fresh checkout / worktree 里 `os.access(X_OK)` 守护真红，但 **CI 原地跑（沿用本地执行位）漏掉**，只有全量 worktree 跑才暴露。同型也指向 git 暂存 SSOT（index≠工作树，见 `.harness/rules/开发流程规范.md` §6 操作教训〔git 暂存 SSOT〕）。

### 2.2 守护硬编码主仓绝对路径（chore-cascade-eval / autosync / doc-wiki / rm-121）

conftest 的 `REPO_ROOT` 或守护测试写死主仓绝对路径 → 在 worktree 内跑时测的是**主仓旧脚本**而非改动后的脚本 = 假绿。改用 `_find_repo_root()` 探测 / `Path(__file__).resolve().parents[N]`。**Owner 验收门禁结果时必须核「测的是不是真目标（改动后的文件）」**，不只看 pass。worktree 测试硬编码主仓路径是 doc-wiki CI 假阴同根。

### 2.3 假覆盖：干扰项没真进被守护路径（list-flows-speedup / rm-113）

写「兜底/边界/负向」回归用例时，必须确认干扰项**真的进入了被守护的代码路径**，否则断言在空集上恒真 = 假覆盖。负向断言 `glob("foo-*")` 易误命中同前缀的**文档/文件**（非目录）→ 须 `is_dir() and name.startswith()` 过滤 + kill-mutant 验「非空满足」。

### 2.4 否定语境假阳（rm-122 / rm-123）

扫「禁止绝对化措辞」这类守护，裸 grep 会把合法引述（"禁用『清零』""绝对化 X 不可取"）误判为违规。须**逐行否定语境校验**：仅当行内含目标串**且无否定/限定词**才计违规。欠摄债、采纳率等描述本身禁用「清零/100%/彻底」绝对化措辞。

### 2.5 git-diff 快照式守护 = 假绿反模式（fix-rm111 · PR#65）

用 `git diff`/`numstat` 快照断言「零删改」，合并后 diff 为空 → 断言空过 = false-green。守护「某内容未被删」必须**正向内容断言**（锚点字符串逐字在场），不是查 diff。反向地，grep+白名单把散文提及误判为重定义 = false-red（如 ADR-id 唯一性应只查 frontmatter `^adr_id:` 定义，不扫散文）。kill-mutant 须双向验（真红 + 不误红）。

### 2.6 收紧约束须引真回归跑全量才暴露（rm-121）

PH-3 deny-list 词级收紧时，隔离单测刻意避开某文件形态 + 测试底座只遍历 `.md` → 端到端假绿；唯有 re-sync 后**全量 worktree 跑**才揪出 `.env.md`（点分量非末尾后缀）漏排。kill-mutant 须**合法语义变异**（如空 `frozenset()` 仍 compile OK），不能用语法错误（破坏整脚本 → traceback 25 failed = 无效 KM）；变异还原见 §3。

### 2.7 测试范围 / 环境（model-tier / checklist-writeback / stale-dogfood-guard / fix-sqlite）

- 阶段8 对 `.harness/` 的变更应跑**仓根 `tests/harness_tools/` 套件**，不是业务子项目裸 pytest（多根 collection error / 漏回归）。
- 结构守护断言宜只锚 **git 跟踪结构**、避开运行时产物以抗抖动；封板期 guard 被后续 sanctioned 阶段合法改变前提时须改判。
- 测试环境配置勿偏离线上默认（持久化/并发开关）；声称并发正确却只有进程内单连接测试 = 盲区（fix-sqlite 封板后 500 bug 同根）。dogfood 人工实测能抓自动化绿灯照不到的盲区（行为遵行验收见 `.harness/rules/验收执行规范.md`〔行为遵行验收〕）。

## 3. kill-mutant 双向验证与还原纪律

做 kill-mutant 验守护测试真红时，临时改坏目标文件后**还原务必用 `cp` 备份覆盖回，禁用 `git checkout <file>`**。

**Why**：阶段3 改动在 worktree 里**尚未 commit** 时，`git checkout <file>` 会把文件回退到 **HEAD**（= 改动前的基线），而非「mutation 前的工作树状态」——结果不是还原 mutation，而是连同未提交的阶段3 编码改动一起抹掉。2026-06-20 RM-2026-116 对 `generator.md` 做 kill-mutant（改坏 F4 explore 锚点验 G2 真红 · 确实 FAILED 证明真红），但 `git checkout .claude/agents/generator.md` 还原时把 F4 扩列整个抹回原始态（`grep "首选 explore"`=0 · 测试仍红）→ 须重新 Edit 恢复 F4 才修好。

**How to apply**：

1. kill-mutant 前先 `cp <file> <file>.bak`；mutation → 跑测看真红 → `cp <file>.bak <file>` 还原 → 跑测确认转绿 + `git diff` 确认无残留。
2. **委派 reviewer 做 kill-mutant 时在 prompt 显式警示**「还原用 cp 备份 · 禁 git checkout（会回退 HEAD 抹掉未提交阶段3 改动）」（RM-116 阶段6 已这样注入 · reviewer 正确用 cp）。
3. 只有改动**已 commit** 时 `git checkout` / `git stash` 才安全还原工作树。
4. **mutation 必须是合法语义变异、非语法错误**（rm-121）：KM 改坏目标须保持文件可解析（如把某集合改成空 `frozenset()` 仍 compile OK），这样只有被测断言失败 = 真红。用语法错误（破坏整脚本）会让 25 个测试全 traceback failed，无法区分"守护真红"还是"脚本炸了" = 无效 KM。kill-mutant 须双向验（真红 + 还原后不误红）。

> 还原后须确认无残留（参见 Edit/Write 后先核对返回纪律 · 详见 `.harness/rules/开发流程规范.md` DF-006 写法纪律：改完先核对再推进）。
