# 规则 · 验收回填与证据捕获规范

> **本规范是 `/backfill` 主持剧本与 `acceptance-backfill` 组件（`acceptance-run.sh` / `acceptance-capture.sh`）消费的权威格式约定**：案例锚点、chains manifest、进度文件、`.evidence/` 证据布局、降级路径、独立/链式两种映射。提取器实现与守护测试**同源引用本文**（单源，防实现与测试各表）。
> 各验收套件回填 README 只放一行指针指向本文，不复制内容（防双源漂移）。
> 核心纪律：**一切进入 result 文件的证据只能由确定性脚本产出**（pre/post 命令输出、transcript 摘录），主持会话零转写；**永不静默丢证据**；**判定权在用户**（落款"判定人=用户"）。

## 1. 案例锚点约定（三段式标题锚 · 归一化容差 · 占位语义）

### 1.1 三段式标题锚

案例文件"回填模板"节采用三段式结构（`chore-acceptance-result-template-3sections-20260621` 固化）：**【运行前】**（前置/造条件）、**【运行中】**（操作）、**【运行后】**（恢复/清理）三个小节标题。`acceptance-run.sh` 以此标题为锚提取【运行前】/【运行后】节内的 ```bash 代码块执行；`acceptance-capture.sh` 把 transcript 渲染写入 result 草稿【运行中】节。

### 1.2 归一化容差（权威定义 · 提取器与守护测试同源引用）

存量三段式标题**非 byte-uniform**（例：FS-05 `### 运行前 · 前置 / 造条件` vs FS-06 `### 运行前·前置/造条件`——空白与全/半角中点差异）。匹配规则：

- **先归一化**：去除标题文本中的所有空白字符，并消除全角中点（`·`）与半角中点等分隔符变体差异；
- **再按前缀识别**：归一化后按 `运行前` / `运行中` / `运行后` 前缀识别所属段落。

该容差规则以本节为**权威定义**；提取器实现与守护测试（块提取器全集可解析断言）均引用本节，不各自另定。

### 1.3 占位语义（可定位 ≠ 可执行）

提取器语义分两层：

| 层 | 判据 | 语义 |
|---|---|---|
| **可定位** | 三段式标题锚在场（归一化容差下命中） | 案例结构合规，可参与回填编排 |
| **可执行** | 该节内含 ```bash 围栏代码块 | 该段命令可被 `acceptance-run.sh` 实际执行 |

**可定位但不可执行**（节内为无标注占位符，无 ```bash 块）→ 该段视为"无 pre/post 块"：脚本 **exit 0**，并在 `.evidence/<CASE-ID>/` 写显式豁免留痕（`pre-skipped.note` / `post-skipped.note`，注明原因 = 占位符无可执行块），主持剧本据此提示用户按案例正文人工准备。**不臆造命令、不把占位文本当命令执行。**

## 2. chains manifest 约定与进度文件

### 2.1 chains.md 表结构

链式案例登记在**套件级旁挂 manifest** `.harness/acceptance/<套件>/chains.md`（不改存量 case 文件本体）。文件头一段用途说明 + 每链一个小节，链内表格固定四列：

```
| 链ID | 序号 | CASE-ID | 进入锚原文 |
```

- **链ID**：链的唯一标识（如 `fs-backend-card`）；
- **序号**：成员在链内的次序（1 起）；
- **进入锚原文**：本案例启动指令的**逐字原文**，从对应 case 文件只读摘取（链首 = 开卡 prompt 首行；链内 = "继续推进…"类指令原文）。守护测试对每条锚 grep 回对应 case 文件核一致性，防在途修订后锚静默漂移。

### 2.2 进度文件 `.backfill-progress`

落点 `results/.evidence/.backfill-progress`（套件级），行式 `KEY=VALUE` 格式，用于中断续跑与链会话钉点：

| KEY 形态 | 写入方 | 语义 |
|---|---|---|
| `batch=<CASE-ID>,<CASE-ID>,...` | 主持剧本 | 本批次成员清单 |
| `status.<CASE-ID>=pending\|captured\|captured-degraded\|finalized` | 捕获器写 `captured`/`captured-degraded`（exit 4 降级捕获）；`pending`/`finalized` 由主持剧本维护 | 案例状态（待跑 / 证据已捕获 / 降级捕获 / 已定稿判定） |
| `chain_session_uuid.<链ID>=<uuid>` | 捕获器（链首定位一次后写入，链内成员不再重匹配） | 链会话 UUID 钉点 |
| `chain_members.<链ID>=<CASE-ID>,...` | 捕获器 | 该链成员序缓存（自 chains.md 解析） |
| `chain_session_path.<链ID>=<路径>` | 捕获器 | 钉点会话 jsonl 的解析路径（诊断/续跑用） |

## 3. `.evidence/` 布局

```
results/
├── <CASE-ID>-result.md              # 回填结果（判定人=用户）
└── .evidence/
    ├── .backfill-progress           # 套件级进度文件（§2.2）
    └── <CASE-ID>/
        ├── pre-<时间戳>.log         # 【运行前】块执行留档（stdout/stderr/退出码）
        ├── post-<时间戳>.log        # 【运行后】块执行留档
        ├── pre-skipped.note         # 占位豁免留痕（§1.3，post-skipped.note 同理）
        ├── env                      # pre 块 fixture 变量状态文件（供 post 段 source）
        ├── <uuid>.jsonl             # 被测会话原始 transcript 归档
        └── result-backup-<时间戳>.md # 覆写既有非空 result 前的原文备份
```

**原始 jsonl 归档入库取舍**：默认**随卡入库**（证据链权威原件，脱敏为渲染层职责、归档保留原件）；体量敏感的套件可在自身 `results/README.md` 声明 gitignore 例外——取舍声明落套件 README，本规范只定默认值。

## 4. 降级路径（永不静默丢证据）

| 场景 | 行为 |
|---|---|
| **transcript 格式版本不识别** | `acceptance-capture.sh` **exit 4**：原始 jsonl 照常归档 `.evidence/<CASE-ID>/`，result 草稿【运行中】节**留空并提示人工摘录**——不臆造内容、不静默跳过 |
| **会话定位失败**（prompt 匹配不到 / 多候选歧义 / 链锚未命中） | **exit 3**：stderr 输出候选差异提示（列候选会话首条 user prompt 摘要），支持 `--session <jsonl路径>` 手工指定；不静默选错、不猜切 |
| **回填中断**（会话/批次中途停止） | 凭进度文件（§2.2）**续跑**：已 `captured`/`finalized` 的案例不重跑，链 UUID 钉点保留 |

通用铁律：**任何降级都不丢已获取的证据**——能归档先归档，节留空 + 显式提示替代臆造。

## 5. 独立 / 链式两种映射

### 5.1 选择判据

案例在套件 `chains.md`（§2.1）中登记 → **链式**；未登记 → **独立**。

### 5.2 操作差异

| 维度 | 独立案例 | 链式案例 |
|---|---|---|
| 会话定位 | 按启动 prompt 原文匹配，**逐案例定位**（完全匹配 + 最新 mtime 优先，歧义 exit 3） | **链首钉 UUID**：链首按开场 prompt 定位一次、UUID 写进度文件，链内成员不再重匹配 |
| 证据切片 | 整会话渲染进【运行中】节 | 按 **[进入锚, 出口锚)** user 轮次边界确定性切片（出口锚 = 下一成员进入锚或链尾） |
| pre/post | 每案例各跑一次 | **pre 链首一次、post 链尾一次**，链内成员跳过案例级 pre/post |
| 跑过头 | 不适用 | 会话已推进越过多个案例 → **链末一次回切**：按锚一次性回切各成员区间产出全部摘录 |

## 6. 旁路声明

本规范是 **checklist / 工具约定**：不进任何阶段门禁判定式、不夺裁决权、不焊 hook 强制触发（ADR-005 对齐）。`/backfill` 会话按验收簿记操作运行；**PASS/FAIL 判定权始终在用户**（result 落款"判定人=用户"），批次结束对总结报告只列待办不代写。
