# create-if-missing 模板改动不惠及存量项目

## 问题模式

以"存在即不动"（create-if-missing）方式分发的模板文件——典型如通过安装钩子铺设的项目根配置文件——有一个容易被忽视的传导缺口：**模板的改动只惠及新初始化的项目**。存量项目的对应文件在首次实例化后已归项目自身的版本控制管理，分发钩子出于"不敢覆盖用户自定义内容"的考虑对其只做 create-if-missing，因此**存量项目永远拿不到后续的模板更新**。

## 当时怎么解

一次版本核对中实证了这个缺口：某次发布包含若干处内容改动，其中经"落盘同步"直接覆盖的文件（agent 定义、规则文件等）全部生效，**唯独模板里新增的一段没有传导到存量项目**的根配置文件——因为那个文件是 create-if-missing 的，钩子看它已存在就不动了。

## 最终方案

- **发版内容尽量放进会被落盘同步覆盖的文件**（如 agent 定义、规则文件），模板里**只放指针**而非承载实质内容——这样更新走"覆盖同步"通道，天然惠及存量。
- 若**必须**改模板且要惠及存量，两条路：①在发版说明里写明**手工补段步骤**，让用户自行更新；②做**"带标记块的增量注入"机制**——钩子只维护自己的标记块内内容、不动用户其余部分，从而既能推送更新又不覆盖用户自定义。

## 适用性判断

- **何时适用**：以"模板 + create-if-missing"方式分发配置、且需要向**存量安装**持续推送更新的分发体系。
- **何时不适用 / 需谨慎**：纯新建场景（没有存量安装）无此问题；若模板文件本就是"交给用户完全自管"的文件，工具**不应**去覆盖它——此时缺口是设计使然而非缺陷。批判点：带标记块的增量注入机制本身有实现复杂度与"标记块与用户内容冲突"的风险，是否值得取决于更新频率与存量规模。
