# 从会话记录审计子 Agent 委派是否合规

## 问题模式

想判断一个 Agent 编排是否"正确地委派了子 Agent"，只靠关键词命中或翻看产物文档并不可靠——**关键词命中 ≠ 委派得当**。产物文档看起来齐全，不代表委派 prompt 真的写全了约束、白名单与必读清单；更隐蔽的是主会话**绕过委派、自己动手写了本该委派的执行产物**（自执行越权），这在产物里完全看不出来。

## 当时怎么解 / 最终方案

审计对象是**会话记录（transcript）本身**，不是流程产物文档。方法：

- transcript 以 JSONL 存于会话日志目录；子 Agent 的记录落在同名子目录下的独立 JSONL。
- 用脚本解析：遍历 assistant 消息的 `content[]`，取委派工具调用的 `tool_use` block，读其 `subagent_type` / `prompt` / `description`。
- **逐委派核查若干点**：①严格约束段是**真文本**而非仅关键词命中；②交付物白名单是**全限定路径**；③必读清单全路径、**无"见某技能/某文档"式二级寻路**；④需求与验收标准摘录在场；⑤探索边界声明在场；⑥（若为元流程委派）传了流程实例 / 阶段 / 目标技能。
- **另查主会话直接写/改的文件**：取主会话自身 Write/Edit 的目标路径——若主会话直写了本该委派出去的执行产物，即自执行越权。
- **查时序**：同一条 assistant 消息内出现多个委派 = 并发；据此判断是否违反串行/并发纪律。

实审经验：同一个编排者在不同任务上的严谨度可能不一致——元流程侧委派套齐了全部约束堪称范本，同期某张普通变更卡的阶段 1 却被主会话直接写了产物、只委派了评审。**易被绕过的正是"阶段 1/3/5 的执行委派"**，审计时应重点查这几处有没有真正派出执行角色。

## 适用性判断

- **何时适用**：需要审计 Agent 编排纪律、排查"主会话绕过委派自己干"的场景，或复盘一次编排为何超成本/超回合。
- **何时不适用 / 需谨慎**：轻量单 Agent 流程没有委派可审；此法依赖平台 transcript 的具体存储格式与委派工具名，换平台需调整解析脚本。更根本的批判：审计是**滞后**手段，更优做法是把这些检查点**前置**成委派前的机械校验（钩子在委派发生时即校验 prompt 是否含白名单/必读/约束段），把事后审计变成事前拦截。
