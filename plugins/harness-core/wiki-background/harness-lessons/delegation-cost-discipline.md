# 委派降本纪律——回合数才是成本主项

## 问题模式

多 Agent 编排里，委派子 Agent 的成本主要驱动是**"回合数 × 上下文深度"**，而不是委派的次数。子 Agent 的固定冷启动开销其实不大；真正贵在：

- 子 Agent 反复冷启动探索、反复重读大文件（尤其评审类任务重读整份规则文件）；
- 并行度低——本可一条消息并发的独立读/查/证被拆成多个串行回合。

一个反直觉的实测结论：**"合并小委派以省固定开销"是被否定的**——固定启动开销小，合并省不下什么，问题始终在回合与上下文深度。

## 当时怎么解

先试了"减少委派次数/合并小委派"，收效甚微。诊断转录后才定位到成本结构：评审类委派因证据注入太薄，子 Agent 只能反复回读规则/技能全文，缓存写占比高，单类任务就吃掉相当一块成本。

## 最终方案

一套委派纪律（按收益排序）：

1. **按阶段配模型档**：机械/廉价阶段用轻量模型，高风险阶段保持强模型；档位下调必须有明确依据。
2. **评审/编码委派 prompt 加厚**：把本任务所需的规则条目**摘录进 prompt**，明确**禁止子 Agent 重读整份规则/技能/agent 文档**，要求**批量读**，并写明**回合预算上限**。核心洞察——**证据注入太薄会让子 Agent 反复重读，反而更贵**。
3. **一条消息内并发**：互不依赖的读/查/证放进同一条消息并发发起，提高并行批量率。
4. **委派前低成本预查注入**：用廉价查询预取关键上下文，随委派 prompt 一起注入，省掉子 Agent 的冷启动探索回合。

## 适用性判断

- **何时适用**：重委派、成本敏感的多 Agent 流程，尤其含大量评审/编码子任务。
- **何时不适用 / 需谨慎**：一次性小任务或纯探索任务，过度配档与预算约束反而增加编排负担；模型档**下调**尤其危险——高风险阶段盲目降档会牺牲质量，降档须逐阶段核对而非套口诀。回合预算是"气味信号"而非硬铡刀，超预算应先反思是否证据注入不足，不必机械中止。
