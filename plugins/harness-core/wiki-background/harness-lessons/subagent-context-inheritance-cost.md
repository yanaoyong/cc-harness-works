# 子 Agent 强制继承主配置文件的上下文成本

## 问题模式

在 Claude Code 平台上，子 Agent（除少数内置的只读探索/规划型外）会**强制加载项目根 CLAUDE.md（含其 @import 链）**，且**没有开关可关闭**——不存在 agent 定义里的 frontmatter 字段或 per-agent 设置来改变哪些 agent 跳过它。官方明确：只有内置的探索型与规划型子 Agent 省略 CLAUDE.md 与 git 状态。自定义 agent 的 system prompt 只替换默认系统提示词，CLAUDE.md 照常加载。

后果：当根配置文件很大时，**每个被委派的子 Agent 启动即背负一份庞大的缓存写入**。在重委派的流程里，这部分继承成本累积起来可占单次任务成本相当可观的一块。

## 当时怎么解

早期一度以为可以通过 agent 定义让特定 agent 跳过 CLAUDE.md，或从 agent 文件里的注释推断"子 Agent 不继承根配置"。实测与官方文档双双否定：平台无此开关，注释式的推断也是过时错误事实。

## 最终方案

既然平台无开关，唯一可行解是**缩小"被继承体"本身**：

- 把根配置文件拆成**最小决策契约集**——只保留子 Agent 必然要背的硬约束（第一动作契约、关键停车点、委派要素、授权边界等 A 类决策时刻契约）。
- 把庞大的**参考层**（详细流程、编码规范全文等）改由**会话启动钩子（SessionStart）注入**，只面向主会话；子 Agent 不走 SessionStart（它走独立的子 Agent 启动事件），因此不再背参考层。
- 代价转移：子 Agent 不再自动背规则全文后，**委派时必须把本任务所需的硬约束摘录/必读路径显式注入 prompt**，否则子 Agent 无规范可依。

子 Agent 底座体量因此显著下降。

## 适用性判断

- **何时适用**：大量使用子 Agent 委派、且根配置文件较重的项目——收益直接体现在每次委派的继承成本下降与上下文窗口余量。
- **何时不适用 / 需谨慎**：若项目几乎不派子 Agent，或根配置本就精简，拆分收益有限、反而增加"主/子两套加载路径"的维护复杂度。拆分后务必验证子 Agent 仍能拿到编码硬约束——这条纪律必须靠委派注入补偿，否则质量会滑坡。
