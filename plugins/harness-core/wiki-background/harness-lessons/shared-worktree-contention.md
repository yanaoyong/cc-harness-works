# 多会话共享 git 工作树的分支争用

## 问题模式

两个独立会话共享**同一个 git 工作树**、各自推进不同分支时会互相踩踏：一方把工作树 `checkout` 到它的分支，另一方的 HEAD 就被切走、tracked 文件被换成对方分支的版本。

最危险的一点：很多阶段产物文件是**从模板复制来的 tracked 文件**，执行角色只是覆盖写其内容、尚未提交时它处于 tracked-modified 状态。一旦工作树被并发切分支，这些**未提交的修改会在 checkout 时丢失 / 被已提交的占位版覆盖**——表面看"文件还在"，实则内容已回退成模板占位版，极易造成"以为产物已写好、其实是空壳"的静默数据丢失。

## 当时怎么解 / 最终方案

- **产出后即备份**：委派子 Agent 产出关键文件后、任何可能的切分支动作前，先 `cp` 到独立临时目录备份；切回后用 `diff` / `cmp` 校验（`cmp` 报字节差异时先用 `diff` 看是否只是 BOM/编码层差异、内容实质一致）。
- **切分支前核对当前分支**：用 `git branch --show-current` 确认当前在哪个分支，**别假设还在自己分支**。
- **争用属用户裁决冲突**：发现工作树被切走，是"多会话争用"，切回会打断对方会话——这是**须由用户裁决**的冲突，**不擅自 checkout 抢占**，而应先问清对方会话是否已停。

## 适用性判断

- **何时适用**：多个并发 Agent / 会话可能共享同一份工作副本的环境（同一台机器、同一 clone、多终端）。
- **何时不适用 / 需谨慎**：单会话独占工作树时无此争用。批判点：备份+校验只是**缓解**，根治是让每个并发会话拥有**独立工作副本**（独立 clone 或独立 git worktree 隔离），从物理上消除争用；但独立 worktree 又可能引入它自己的问题（如钩子/状态目录随工作树分裂），需权衡后再选隔离粒度。
