# 整体重生成文件的同步脚本会回退人工改动

## 问题模式

一个发布/同步脚本用**脚本内嵌模板（heredoc）整体重生成**某个下游文件——典型如把主仓内容同步到公有分发仓时，用内嵌模板整体覆盖下游的说明文档。当该下游文件曾被**人手工维护**过、而脚本内嵌模板未同步跟进时，**每次同步都会用旧模板覆盖、悄悄回退掉那处手工改动**（净 diff 表现为下游文件被删掉若干手工加的行）。

## 当时怎么解 / 最终方案

- **同步后、push 前先看 diff**：跑完同步先 `git diff` 该文件；若发现只是模板回退了手工内容，就在下游工作副本用 `git checkout --` 丢弃该项变更、**只提交真正的子集改动**（真正要同步的内容），保留手工版。
- **根治两条路**：①把手工改动**回填进脚本内嵌模板真源**，让两边 diff 归零（改真源而非只改下游副本，否则下次再被覆盖）；②改脚本让它**读取既有下游文件并做增量 patch**，而非整体覆盖。
- **关键认知**：只要有人会手工改下游文件，本坑就**必然复现**——它不是"改一次就根治了"，而是"**每次同步都要查**"。这类坑最忌"上次修过了应该没事"的乐观假设。

## 适用性判断

- **何时适用**：用"整体重生成"（而非增量 patch）方式维护下游文件、且下游文件可能被人手工改动的分发/发布流程。
- **何时不适用 / 需谨慎**：下游文件**纯由脚本单向生成、无人手工介入**时，整体重生成是安全且简单的做法，无需增量化。批判点：整体重生成对"完全托管"的文件安全，对"半托管"（脚本 + 人工混合维护）的文件危险；理想设计是把**手工可变部分与脚本生成部分物理分离**（如生成部分包在标记块内、块外留给人工），从结构上消除覆盖冲突。
