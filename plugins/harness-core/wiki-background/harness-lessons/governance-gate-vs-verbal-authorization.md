# 机械治理门只认合规路径，口头授权不算数

## 问题模式

一个机械化的 git 治理钩子（fail-closed 拦截未授权向主分支 push / merge）**只认"分支 → PR → 授权 merge"这一条合规路径**，不认口头 / 自然语言授权 token。因此 merge 之后想直接改主分支上的文件（典型如回填 merge SHA 占位、订正随卡合入的文档）会被**硬拦**——即使当前 checkout 就在主分支、即使用户给了逐字口头授权也照拦不误。

相邻的一个坑：终态"翻牌"（把状态行从占位改成终态）用 Edit **替换**语义时，若 `old_string` 没框住**旧占位整行**，Edit 会变成"新增"而非"替换"，导致同一状态出现**重复行**并随 PR 合入主分支。

## 当时怎么解 / 最终方案

- **任何主分支变更都走分支 + PR**：包括 merge SHA 回填、post-merge 文档订正——别想直推，省得被拦后撤销重来。
- **占位可留、回填走轻量收尾 PR**：merge SHA 占位可接受先留在主分支（纯文档无害），要回填就预备一个轻量收尾分支 + PR，且这个收尾 PR 的 merge **仍需用户按 PR 逐字授权**（机械门按 PR 标识认授权，不认笼统的口头许可）。
- **翻牌用 Edit 务必框住旧整行**：`old_string` 要包含旧状态整行（含 PENDING / 占位标记），确保是替换而非新增；翻牌后用 `grep` 自检该状态行无重复。

## 适用性判断

- **何时适用**：用机械钩子强制"分支 → PR → 授权"路径的治理体系，尤其是有自动化 Agent 参与 git 操作、需要防止未授权改主分支的场景。
- **何时不适用 / 需谨慎**：没有此类机械门、或团队约定允许直推主分支的项目不受此约束。**关键认知（批判点）**：机械门拦截口头授权**正是它正常工作的表现**（这是安全边界，不是 bug）——不要把"被拦"当成故障去绕过它；**口头授权 ≠ 机械门的合规路径**，二者必须区分，绕过机械门（如临时关钩子直推）等于自毁安全边界。
