# Skill · 单元测试编写（unit-test-write）

## 1. 目的

为本次改动编写**有业务价值**的测试用例，保障阶段 8 门禁 `total_tests > 0` 且可回归。

## 2. 触发条件

进入 **阶段5**（阶段4 编码评审通过且 HITL-3 通过）时加载。

## 3. 输入

- 阶段4 通过的代码与改动接口/API 清单；
- 业务子项目路径（实值见 `HARNESS_CONFIG.yaml`；以 `spec.md` 为准，测试根/约定见 `../../rules/项目编码规范.md §3 PY-4`）；
- 可用时通过 MCP 查询被改动接口的线上真实出入参构造数据。

## 4. 步骤（SOP）

1. **改动驱动**：改了哪个表现/业务层方法就测哪个，而非只测无关模块。
2. 使用栈特定层约定的**测试客户端 / 测试夹具**（栈特定见 `../../rules/项目编码规范.md §3 PY-4`）。
3. 覆盖正常路径 + 关键边界/异常路径。
   - 如本卡涉及 API · 必须含契约测试用例（契测模板见 `.harness/templates/contract-testing/` · 依据 05-技术栈/01 §8）。
4. **迭代范围**：迭代期间（写→跑→改循环）只跑**改动相关子集测试**（绑定层 `testCommand()` 给出的等价测试命令限定到改动相关测试模块），全量套件留给**阶段 8 门禁**（`unit-test-ci`）一次执行。
5. 在业务子项目根执行并确认相关测试模块通过——运行绑定层 `testCommand()` 给出的等价测试命令（本项目实值见 `HARNESS_CONFIG.yaml` `test_command`；测试根/约定见 `../../rules/项目编码规范.md §3 PY-4`），先限定到**改动相关子集测试**，全量套件归**阶段 8**。

6. 产出 `unit_test/` 下报告，列出覆盖映射（接口 → 测试文件/用例名）。
7. **测试配置贴近线上默认**：测试夹具默认配置应贴近**线上默认**（尤其持久化/并发开关，如 SQLite WAL）；确有必要偏离须在夹具处**显式加注理由**，并纳入阶段 4 评审检查。
8. **并发正确性须真实并发用例**：凡声称强一致 / 计数正确性（如「M 次请求 == count M」），须有**多线程或真实 HTTP 客户端并发**用例支撑（断言成功数 == count 且**无 5xx**），不得仅以进程内单连接测试客户端顺序调用充当并发证据。

## 5. 产出物

- 测试模块（落于栈特定层约定的测试根，命名见 `../../rules/项目编码规范.md §3 PY-4`，或 spec 约定路径）；
- `../../changes/<变更目录>/unit_test/unit_test_report_vN.md`。

> 报告精简：本 SKILL 产出的报告类产物须满足 `../../rules/开发流程规范.md` **DF-012**（报告预算：内容硬约束 + 100 行软自检线 + 证据优先 + 不硬截断；`spec.md`/`tasks.md` 豁免）。

## 6. 完成判据

- 被改动接口均有对应测试（覆盖映射成立）；
- 本地确认 = 改动相关子集测试**全绿**（运行绑定层 `testCommand()` 给出的等价测试命令限定到改动相关测试模块；实值见 `HARNESS_CONFIG.yaml` `test_command`）；全量套件验证归**阶段 8 门禁**一次执行；
- 为阶段 8 提供可执行用例（`total_tests > 0` 且 `passed == total_tests`）。
- 若本卡声称并发/强一致正确性：须有真实并发用例（非进程内单连接）且断言无 5xx；测试夹具默认配置贴近线上默认或对偏离显式加注。
