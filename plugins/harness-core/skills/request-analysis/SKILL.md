# Skill · 需求分析（request-analysis）

## 1. 目的
把用户原始需求转化为结构化的需求规格（spec）与可执行的任务拆分（tasks），为后续评审与编码奠定基线。

## 2. 触发条件
进入 **阶段1**（收到需求）时由编排中枢加载。

## 3. 输入
- 用户需求描述；
- 必要时按需查阅 `wiki/`（领域上下文）与历史 `.harness/changes/`（同类需求）。

## 4. 步骤（SOP）

> **起手批量并发首读（proposal-012 §3② · 一条消息并发）**：进入阶段1 第一动作 = 把以下独立首读项在**一条消息内并发**读取、不逐个串行——`summary.md`（开卡范围）+ `../../rules/工程结构.md`（分层依据）+ 相关历史 `.harness/changes/` 同类变更（若有）。

1. 澄清需求意图与边界；存在歧义 → 在 **HITL-1** 升级人工确认，**不臆测**（歧义呈现方式见 §4.2「HITL-1 UQ 批量呈现」）。
2. 梳理影响面（涉及模块/链路/接口/数据），引用 `../../rules/工程结构.md` 判断分层归属。
3. 产出 `spec.md`：背景、范围（含不做的事）、验收标准、影响面、风险。
4. 拆分 `tasks.md`：每个子任务明确 目标 / 范围 / 输入输出 / 验收标准 / 依赖；并按 §4.1「阶段1 划组纪律」为各任务标注**并发组**。
5. 交卷前逐条走 §5「产前自检段」12 类高频缺陷清单，并在 `spec.md` 尾部附「产前自检声明」段。
6. 自检完整性后交付阶段2 评审。

### 4.1 阶段1 划组纪律（方向④ · 阶段3 并发的产物结构层固化）

> 目的：把并行从"Owner 临场判断"（实测几乎不触发）改为在 tasks.md **产物结构层**固化的「并发组」，让阶段3 得以**机械照组并发**（p012 三分律：改行为的纪律不自动执行，须固化进产物结构）。

**划组判据 = 文件落点不相交**：

1. **按落点划并发组**：拆完任务后，逐一登记每个任务的**文件落点白名单**（该任务允许写入的全限定路径），把**落点互不相交**的任务归入同一个可并发组 `G<n>`；每组在文件头「并发组总览」表登记该组的落点白名单（该组各任务落点的并集）。
2. **落点相交 → 同任务或串行**：两个任务若写入**同一文件**（落点相交），**禁止**分到"可并发"的不同任务里同组并发——要么合并为同一任务（由同一 generator 端到端独占该文件），要么显式标依赖序、跨组串行执行。这是「并发前落点白名单互不相交」前置检查的产物层来源。
3. **跨组依赖标依赖序**：有产出/输入依赖的任务（如"内容层定稿后才能同步镜像/bump 版本"）拆到不同并发组，后组以「依赖: 前组全部完成」显式标注，跨阶段/跨组保持串行。
4. **字段承载形态（轻量行式）**：并发组以**现行 tasks.md 五要素结构 + 每任务增一备注行**承载，不发明未评审的复杂字段格式——
   - 每任务备注行：`并发组 G<n> · 落点白名单 = <该任务的全限定路径列表>`（格式以 `_TEMPLATE/request_analysis/tasks.md` 模板备注行为唯一权威、此处同串引用不另立格式）；
   - 文件头附「并发组总览」表：列出各组、组内任务、组落点白名单（互不相交）、并发性说明。
5. **共享耦合契约先冻结**：跨任务共享的措辞/字段格式（各任务都要读、不能各自发明的），在 `spec.md` 冻结，各任务只读冻结契约、不在制品互读（K7 #6「委派前先冻结共享耦合接口」）。

> **落点仅是"写入"落点**：多个任务**只读**同一文件不算落点相交（读不产生写冲突）；相交判定只看**写入**白名单。
> **指针（不重定义）**：阶段3 Owner 如何"照组机械并发多 generator"的编排纪律，权威落点在 `../../agents/application-owner.md` 模块四「同阶段并行委派纪律」（含对跨卡并行禁令 failure-record-004 的边界引用）——本 SOP 只负责在阶段1 **产出**并发组，不重定义阶段3 的消费编排。

### 4.2 HITL-1 UQ 批量呈现（方向⑤ · 减往返、语义不减）

> 目的：把需用户拍板的未决项从"逐项往返多次询问"折叠为"**一次性批量呈现**"，压 HITL-1 的人工往返次数，语义与留痕一个不少。

1. **集中编号 UQ-n**：产出 `spec.md` 时，把所有**需用户拍板**的未决项（歧义选项、默认行为取值、承载形态、范围充分性等）集中收敛到 `spec.md` §5.2，逐项编号 **UQ-1、UQ-2 …**，每项写清：问题原文、可选项、Owner 建议项（标 Recommended）。
2. **一次性批量呈现**：在 **HITL-1** 用**一次 AskUserQuestion（一轮往返）**把所有 UQ-n 批量呈现给用户拍板，**不逐项发起多轮往返**——一轮清掉全部未决项。
3. **不臆测、不静默**：无 UQ 时才可直接进阶段2；有 UQ 未闭合**禁止**臆测推进过阶段1 门（对治高频缺陷 D3「未决项/占位残留」）。
4. **决议留痕回写**：HITL-1 关闭后，把用户逐项决议（含 AskUserQuestion 摘录 / 用户原话）回写 §5.2，原问题原文保留作留痕；决议若与 AC 原文有出入须同步回写 AC（守 D1「决议漂移」）。

> **语义守恒边界**：本节只减 HITL-1 内部**呈现的往返次数**，不改 HITL-1 的语义与真实性硬门——AskUserQuestion 须真实用户作答闭合、禁自答虚构（DF-015）。

## 5. 产前自检段（方向⑥ · generator 交卷前自查 · 前移压两轮率）

> 目的：把历史 `spec_review_v1` 中导致 REVISION/v2 的 **MUST FIX** 归类为下方 12 类高频缺陷清单，让 generator **交卷前逐条自查**，把阶段2 两轮率前移压低（对治评审两轮率约 50% · MUST FIX 高度模式化）。
>
> **不取代对抗评审（硬边界）**：产前自检是"产前减少低级/模式化缺陷"，**不取代、不放松**阶段2 的对抗性评审——`../expert-reviewer/` 评审 SOP 仍全量对抗，不因本段前移而放松任何评审步骤。
> **随 failure-record 滚动更新（非冻结常量）**：本清单是**滚动清单**，随新暴露的高频失败模式（新增 `failure-record-*` / 新的 v1 MUST FIX 模式）**增补**，非冻结常量；两仓通用、业务卡按需消费 D4/D5/D8 等业务专属项。

### 5.1 12 类高频缺陷自查清单（D1–D12）

> 来源：myharness `.harness/changes/*/request_analysis/review/spec_review_v1.md` 中导致 REVISION/v2 的 MUST FIX 归类。「典型历史卡」列供溯源、可随滚动更新增删。

| # | 缺陷类 | 定义（交卷前自查什么） | 典型历史卡 |
|---|---|---|---|
| D1 | **决议漂移** | HITL/Q 决议已定但无对应可机械验证 AC，阶段3 会静默丢失 | subagent-speedup、flow-speedup-20260611、wiki-query-adoption |
| D2 | **AC 验证口径不可达** | AC 写的 grep/命令/断言字串在改动前后实测不成立，或守护对象不覆盖新增物 | flow-speedup-20260611、sessionstart-autosync、varbrace、bash32 |
| D3 | **未决项/占位残留** | spec 含"阶段3再定"/`{{模板占位}}`/未闭合歧义就想过阶段1门 | price-discount、inventory、categories、template-scaffold |
| D4 | **数值·默认行为未定死** | 金额取整规则/缺失降级取值/边界默认值无唯一可验证定值 | categories、price-discount、inventory |
| D5 | **分层归属错位** | 违反工程结构 ES-001（adapter/service/repository 放错层） | inventory |
| D6 | **硬约束未前置** | 超时降级/禁密钥等仅列"风险"未进 FR/范围/AC，阶段4 必成 MUST FIX | price-discount |
| D7 | **影响面·级联·镜像清单不全** | 漏关联卡/漏运行时副本/漏守护测试/漏 parity 镜像/漏 lockstep | plugin-loader、closure-debt |
| D8 | **测试覆盖对象错位** | 验收把测试挂在非改动接口（只测 service 未测对外 API） | inventory、subagent-speedup |

### 5.2 交卷前自查 + spec 尾附「产前自检声明」（强制）

1. **交卷前逐条自查**：generator 在阶段1 交付前，须对 D1–D12 **逐条自查**本卡 spec/tasks 是否命中该缺陷模式。
2. **spec 尾附声明段**：须在 `spec.md` **尾部**附「产前自检声明」段，对 12 类**逐条**给出结论——"**适用/已规避**（附如何规避）"或"**N/A + 理由**"（如纯 SOP 卡对 D4 数值类声明 `N/A：本卡无数值计算`）。
3. **可审计**：声明须逐条落地（可 grep 审计的表格），**非仅"已自查"一句**——防自检沦为走过场（对治 R4「自检清单形式化」，如 p010「分档纸面化一个月」教训）。
